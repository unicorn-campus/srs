# 통신사 B2B SRS 작성 가이드 (개발요구사항정의서 양식)

> 문서 상태: **최종안(FINAL)** — LG유플러스 개발요구사항정의서 샘플(참조 양식)을 근거로 16장 골격으로 개정함.  
> 용도: 통신사 B2B 기획팀이 확정 RFP·RFP분석서를 근거로 SRS(솔루션요구사항정의서)를 **개발요구사항정의서**

> 각 장(G1~G16)은 3단 구조 **(a)작성 원칙 · (b)통신사 B2B 예시·문구 · (c)작성 체크리스트**로 구성되며,  
> 각 장에 측정 가능한 판정 기준 서술 예시를 포함함.  
> 참조 양식: LG유플러스 「SmartQA Voice Insight 사업 요구사항」 개발요구사항정의서(교육용 Dummy Sample) —  
> 16장 골격·표·필드 나열·⚠ 주의문·X-RAY 예외 케이스 구성을 재현 기준으로 삼음.  
> 근거 입력: 확정 RFP `work/rfp/RFP-MVNO장애보안사고_실시간통지_영향범위조회시스템.md`, RFP분석서  
> `work/rfp/rfp-analyze.md`. 통합·확정: 스펙 / 조율: 오케스트레이터(클로니).

## G0. 이 가이드 사용법

### 참조 양식 — 개발요구사항정의서란

- 본 가이드가 대상으로 하는 SRS는 수주 벤더 개발팀·아키텍트가 **그대로 구현 근거로 사용하는 개발요구사항  
  정의서**임. RFP의 "무엇을 왜 원하는가"를, 개발자가 바로 만들 수 있는 "시스템이 무엇을·어떤 데이터로·어떤  
  정책·성능으로 처리하는가"로 정형화함.  
- 참조 양식(LG유플러스 SmartQA)의 시그니처 구성 4종을 반드시 재현함.  
  - **시스템 역할 경계**: 수행 업무 / 미수행 업무(타 시스템 담당)를 명확히 분리(G3)  
  - **데이터·결과 필드 나열**: 입력·결과 데이터를 필드 단위로 상세 나열(G4·G12)  
  - **표 4종 이상**: 룰 적용단위(G9)·결과 상태값(G11)·운영자 화면(G14)·성능(G15)  
  - **X-RAY 예외 케이스**: "~하더라도 ~인 경우 ~함" 조건부 예외 규칙(G16)  
  - 추가: ⚠ 개인정보·보관 주의문(G4·G13)

### 3단 구조

- 각 장(G1~G16)은 동일한 3단 구조로 구성됨.  
  - (a) 작성 원칙: "무엇을 왜 쓰는가"를 표준·근거로 규정함  
  - (b) 통신사 B2B 예시·문구: 본 RFP(MVNO 장애·보안사고 실시간 통지 시스템) 맥락으로 각색한 예시를 제시함  
  - (c) 작성 체크리스트: 누락 방지용 자가점검 항목을 제시함

### 핵심 원칙

- **측정할 수 없으면 요구사항이 아님**: 성능·정책·상태 전이 등 모든 판정 기준을 [지표 + 목표값(임계값) +  
  측정조건·판정방법]으로 정량 서술함. 참조 양식은 추적성 RTM을 두지 않으므로, 정량 기준을 **성능 표·상태값  
  표·판정 기준 나열에 직접 내장**함.  
- **RFP 요구ID 근거**: 모든 기능·데이터·정책·성능 항목은 RFP분석서 요구ID(REQ-BIZ/TEC/OPS)에 근거하며,  
  근거 없는 임의 추가·삭제를 금지함. 요구ID는 문장 말미 대괄호로 병기함(예: `[REQ-BIZ-01]`).  
- **역할 경계 우선**: 본 시스템이 하지 않는 일(레거시·타 시스템 담당)을 먼저 못 박아 범위 확장(scope creep)과  
  정산 분쟁을 예방함.

### 16장 골격 표준

- 머리말(제목·안내·문서정보) → G1. 목적 → G2. 기본 원칙 → G3. 시스템 역할 경계 →  
  G4. 입력 데이터 요구사항 → G5. 데이터 가공·검증(전처리) 요구사항 → G6. 핵심 처리·판정 대상 요구사항 →  
  G7. 정책 요구사항 → G8. 특수·강화 정책 → G9. 룰/리스트 관리 요구사항 →  
  G10. 신규·예외 흐름 및 지연 처리 요구사항 → G11. 처리 결과 상태값 → G12. 결과 필수 포함 정보(필드) →  
  G13. 감사 로그 요구사항 → G14. 운영자 관리 화면 요구사항 → G15. 성능 요구사항 →  
  G16. 예외·보완 케이스 → 꼬리말(교육용 고지)

### SRS 조립 순서

1. G1에서 시스템이 무엇을 하는지 정의하고 수행 기능·타 시스템 경계를 확정함  
2. G2에서 RFP 실제 니즈·제약(RFP분석서 A4)을 설계·판정의 기본 원칙으로 재진술함  
3. G3에서 수행/미수행 업무를 분리하고, G4·G5에서 입력 데이터와 전처리 요건을 필드로 명세함  
4. G6~G10에서 판정 대상·정책·강화·룰/리스트·지연 흐름을 정형화함(핵심 처리 로직)  
5. G11·G12에서 결과 상태값(표)과 결과 필드(나열)로 처리 결과를 규정함  
6. G13~G15에서 감사 로그·운영자 화면(표)·성능(표) 등 비기능·운영 요건을 정량화함  
7. G16에서 상충·엣지 케이스를 X-RAY 예외 규칙으로 닫음

### RFP → 16장 매핑 원칙

- REQ-BIZ(업무) → G1 목적 · G6 판정/처리 · G7 정책 · G11 상태값 · G12 결과 필드  
- REQ-TEC(기술) → G4 데이터 · G5 전처리 · G13 감사 로그 · G15 성능  
- REQ-OPS(운용) → G3 역할 경계 · G10 지연 처리 · G14 운영자 화면 · G15 가용성  
- 실제 니즈·제약·상충(RFP분석서 A4) → G2 기본 원칙 · G16 예외 케이스  
- AI 요구(REQ-*-05 등) → G6 판정(AI 판정) · G11 상태값(추가검토→휴먼 검토 이관) · G12 필드(AI 신뢰도 점수)  
- 배치 원칙: 모든 요구ID를 최소 1개 장에 매핑하고 미매핑 0건을 확인함(최종 문서에 RTM은 넣지 않되, 작업 중  
  커버리지 대사는 work/ 산출물에서 수행).

### 표현 형식 규칙 (참조 양식 준수)

- **표로 표현**: 룰 적용단위(G9)·결과 상태값(G11)·운영자 화면(G14)·성능(G15)  
- **나열로 표현**: 수행/미수행 역할 경계(G3)·데이터 필드(G4)·결과 필드(G12)·예외 케이스(G16)  
- **⚠ 주의문**: 개인정보·보관 관련은 ⚠ 기호로 시작하는 강조 문장으로 표기(G4·G13)  
- 한국어, 명사체 종결(~함/~임), 한 줄 120자 이내, 정량 기준은 RFP/RFP분석서 근거 수치만 사용.

## G1. 목적

> 다루는 항목: 시스템이 무엇을 하는지 1문단 정의, 시스템이 수행하는 기능의 한정 나열, 본 시스템이 담당하지  
> 않고 레거시·타 시스템이 담당하는 업무 명시. RFP: 취지(WHY)·REQ-BIZ 전반.

### (a) 작성 원칙

- 목적 절은 개발자가 본론(데이터·처리·정책·성능)을 읽기 전에 **시스템의 존재 이유와 처리 범위**를 한눈에  
  파악하도록 하는 계약적 서문임. 시스템이 "무엇을 하는 시스템인가"를 1문단으로 정의함.  
- 정의 문단 뒤에 **시스템의 역할을 한정하는 기능 나열**을 배치함 — 참조 양식처럼 "시스템의 역할은 다음으로  
  한정함"으로 못 박아 기대 범위를 통제함.  
- 본 시스템이 하지 않고 **레거시·타 시스템이 담당하는 업무**를 목적 절 말미에 명시하여 경계를 선언함(상세는 G3).  
- 대상 시스템을 이름으로 식별하고 상위 RFP 명칭과 일치시킴("측정할 수 없으면 요구가 아님" — 범위 완전성도  
  요구ID 매핑률로 검증).

### (b) 통신사 B2B 예시·문구

- **정의 문단 예시**: "「MVNO향 통신 장애·보안사고 실시간 통지 및 영향범위 조회 시스템」(이하 본 시스템)은  
  MNO 망에서 발생하는 장애·보안사고 이벤트를 실시간 수신하여, 심각도별로 MVNO 담당자에게 자동 통지하고,  
  영향 가입자 규모·지역·서비스 범위를 실시간 조회하며, 복구정보(ETR)·사고 리포트·AI 영향예측을 제공하는  
  시스템임."  
- **역할 한정 나열 예시**: "본 시스템의 역할은 다음으로 한정함 — 이벤트 수신 · 심각도 자동 분류 · 다중 채널  
  통지 발송 · 영향범위 실시간 조회 · ETR 갱신 · 사고 리포트 생성 · AI 영향예측·응대지원 · 통지/감사 이력 관리."  
- **타 시스템 경계 문구 예시**: "MNO 망 물리적 복구, 가입자 요금·과금 처리, CRM 원본 갱신, 상담 응대 자체는  
  각 MNO 또는 레거시 사내 시스템이 담당함."

### (c) 작성 체크리스트

- [ ] 시스템이 무엇을 하는지 1문단으로 정의하고 대상 시스템명을 RFP와 일치시켰는가  
- [ ] 수행 기능을 "역할 한정" 나열로 명시했는가  
- [ ] 본 시스템이 하지 않고 타 시스템이 담당하는 업무를 목적 절에 선언했는가  
- [ ] 정의·역할이 RFP 취지(WHY)·REQ-BIZ와 정합하는가

## G2. 기본 원칙

> 다루는 항목: 솔루션 설계·판정의 핵심 원칙 4개 내외(2.1~2.4). 각 원칙에 설명 문단과 근거 나열.  
> RFP: 실제 니즈·제약·엔드유저 상충(RFP분석서 A4).

### (a) 작성 원칙

- 기본 원칙은 RFP의 **표면 요구가 아닌 실제 니즈**를 설계·판정의 상위 규범으로 재진술하는 장임. 개별 기능이  
  흔들릴 때 판단 기준이 되는 "이 시스템은 이렇게 판단·처리한다"는 철학을 3~5개 원칙으로 고정함.  
- 각 원칙은 소제목(2.1~2.n) + 설명 문단 + 근거 나열로 구성함. 원칙은 이후 정책(G7)·예외 케이스(G16)의  
  판단 근거로 재인용됨.  
- 상충하는 요구(신속성 vs 정확성, 자동화 vs 검수)는 원칙 단계에서 **우선순위**를 못 박아 하위 정책의 충돌을  
  예방함.

### (b) 통신사 B2B 예시·문구

- **2.1 이벤트만 보지 않고 사고 컨텍스트를 함께 판단함**: 통지 여부·심각도·채널은 이벤트 자체뿐 아니라 영향  
  규모·지역·시간대·대상 가입자 등급·과거 유사사고 이력을 함께 고려하여 결정함.  
- **2.2 통지 지연을 최소화하되 오탐 리스크는 통제함**: 정상 사고는 즉시 통지하되, 신규 사고유형·저신뢰 AI  
  예측·대량 클레임 구간은 즉시 확정하지 않고 일부 우선 통지 또는 지연 판정으로 전환할 수 있어야 함.  
- **2.3 미탐·오탐을 모두 관리함**: 통지 누락(미탐)은 가입자 이탈·규제 리스크로, 과도 통지(오탐)는 알림 피로로  
  이어지므로, 결과를 정상/부분/추가검토/지연/실패 등으로 세분함(G11 연계).  
- **2.4 설명 가능한 판정을 제공함**: 통지·지연·면책 판정 시 운영자가 사유(사유 코드·적용 정책·근거 데이터)를  
  확인할 수 있어야 함(G12 결과 필드 연계).  
- **우선순위 원칙 예시**: "상충 시 **가입자 보호(통지 신속성·정확성)를 최우선**으로 두며 운영 편의·비용은  
  후순위로 함."(RFP 2-1 근거)

### (c) 작성 체크리스트

- [ ] 표면 요구가 아닌 실제 니즈(RFP분석서 A4)를 설계 원칙으로 재진술했는가  
- [ ] 각 원칙이 소제목 + 설명 문단 + 근거 나열로 구성되었는가  
- [ ] 상충 요구의 우선순위(가입자 보호 최우선 등)를 원칙에서 못 박았는가  
- [ ] 원칙이 이후 정책(G7)·예외 케이스(G16)의 판단 근거로 연결되는가

## G3. 시스템 역할 경계

> 다루는 항목: 3.1 수행하는 업무(나열) / 3.2 수행하지 않는 업무(나열, 타 시스템 담당). RFP: 범위/제외범위.

### (a) 작성 원칙

- 참조 양식의 시그니처 장임. 본 시스템이 **수행하는 업무**와 **수행하지 않는 업무**를 두 개의 나열로 명확히  
  분리함. "하지 않을 것"의 명문화가 정산·범위 분쟁 예방의 핵심임.  
- 미수행 업무는 단순 나열이 아니라 **담당 주체(어느 레거시·타 시스템)**를 함께 적어 책임 경계를 못 박음.  
- RFP·SRS에 없는 기능은 자동으로 미수행이며 변경(CR) 대상임을 전제함.

### (b) 통신사 B2B 예시·문구

- **3.1 수행하는 업무(나열)**: MNO 이벤트 수신 · STT/페이로드 검증 · 심각도 자동 분류 · 영향범위 실시간  
  산출 · 통지 대상·채널 결정 · 다중 채널 통지 발송 · ETR 15분 주기 갱신 · 사고 리포트 자동생성 ·  
  AI 영향예측·응대 스크립트 초안 · 통지/감사 이력 저장 · 운영자 관리 화면 제공.  
- **3.2 수행하지 않는 업무(나열, 담당 주체 병기)**  
  - MNO 망 물리적 복구 — MNO 담당(관찰 지표만 수집)  
  - 가입자 개인 식별정보 원문 제공 — 미수행(집계·범위 단위만)  
  - AI 결과의 무검수 자동 확정·발송 — 미수행(담당자 HITL 검수 필수)  
  - 요금·과금·CRM 원본 갱신 — 레거시 사내 시스템 담당  
  - 상담 응대 자체·고객 직접 통보 — 고객센터·레거시 채널 담당

### (c) 작성 체크리스트

- [ ] 수행 업무와 미수행 업무를 각각 나열로 분리했는가  
- [ ] 각 미수행 업무에 담당 주체(MNO·레거시·타 시스템)를 병기했는가  
- [ ] 역할 경계가 RFP 범위/제외범위·벤더 책임 경계와 일치하는가  
- [ ] 미수행 항목이 G16 예외 케이스·정산 경계와 모순되지 않는가

## G4. 입력 데이터 요구사항

> 다루는 항목: ⚠ 개인정보·보안 주의문, 4.1~4.n 도메인별 입력 데이터 항목의 필드 단위 상세 나열.  
> RFP: REQ-TEC(데이터)·REQ-BIZ(조회 대상).

### (a) 작성 원칙

- 시스템이 처리 판단에 사용하는 입력 데이터를 **도메인별(4.1, 4.2…)로 묶고 필드 단위로 상세 나열**함. 필드가  
  누락되면 개발 단계에서 인터페이스 재협의가 발생하므로 망라성을 우선함.  
- 개인정보·민감정보를 포함할 수 있는 도메인은 장 서두 또는 해당 절에 **⚠ 마스킹·최소수집 주의문**을 배치함.  
- 각 필드는 판정·처리에 쓰이는 이유가 드러나도록 도메인을 구성함(예: 컨텍스트 판정용 보조 정보 도메인 분리).

### (b) 통신사 B2B 예시·문구

- **⚠ 주의문 예시**: "⚠ 가입자 정보·통지 대상 연락처는 개인정보를 포함할 수 있으므로 저장·조회 시 필드 단위  
  마스킹 정책을 적용함(별첨 마스킹 규칙 준거)."  
- **4.1 사고 이벤트 정보(MNO 수신)**: 이벤트ID · 수신시각 · 사고유형(장애/보안) · 심각도 원천값 · 영향 국사/셀 ·  
  영향 서비스 범위 · 원천 서명·재전송 여부.  
- **4.2 영향범위 산출 입력**: 자사 가입자 분포(지역·서비스·요금제 집계 차원) · 조인 키(지역·망 식별자·서비스  
  구분) · 사전집계 뷰 갱신시각.  
- **4.3 통지 대상 정보**: 담당자 사번 · 소속 팀ID · 역할 등급 · 통지 채널 우선순위 · 수신 가능 시간대 ·  
  연락처(마스킹).  
- **4.4 가입자 컨텍스트(집계)**: 영향 가입자 규모 · 등급 분포(VIP/일반) · 최근 클레임 이력(집계) · 해지의향  
  지표 — ⚠ 개인 식별자 미포함.  
- **4.5 정책 판단 보조 정보**: 과거 유사사고 이력 · MNO 원천 데이터 적시성 · 팀 통지 준수 이력 · 시간대(야간) ·  
  AI 예측 신뢰도.

### (c) 작성 체크리스트

- [ ] 입력 데이터를 도메인별(4.1~4.n)로 묶고 필드 단위로 상세 나열했는가  
- [ ] 개인정보·민감 도메인에 ⚠ 마스킹·최소수집 주의문을 배치했는가  
- [ ] 컨텍스트 판정용 보조 정보 도메인을 별도로 구분했는가  
- [ ] 각 데이터가 근거 요구ID(REQ-TEC/REQ-BIZ)와 연결되는가

## G5. 데이터 가공·검증(전처리) 요구사항

> 다루는 항목: 입력 데이터의 정규화·검증·보정 대상(5.1, 5.2…). RFP: 데이터 요건.

### (a) 작성 원칙

- 입력 데이터가 판정 로직에 들어가기 전 거치는 **검증·정규화·보정** 규칙을 명세함. 원천 데이터 품질 편차를  
  흡수하여 후속 판정의 일관성을 확보함.  
- 인식·보정 대상(오류 유형)과 정규화 대상(표준화 항목)을 나눠 나열함.  
- 시각 정합성(NTP 동기 등)처럼 SLA 자동 측정의 전제가 되는 전처리 요건을 포함함.

### (b) 통신사 B2B 예시·문구

- **5.1 원천 이벤트 검증·보정**: 페이로드 스키마 검증 · 서명 검증 · 중복 이벤트 제거 · 재전송 병합 · 결측  
  필드 기본값 보정 · 심각도 원천값 이상치 탐지.  
- **5.2 정규화 대상**: 사고유형 코드 표준화 · 지역/망 식별자 매핑 · 서비스 코드 표준화 · 시각 표준화(수신  
  시각 기준 정렬·NTP 동기) · 영향범위 집계 차원 정규화.  
- **정합성 문구 예시**: "'수신' 시각 기준을 MNO 이벤트(Webhook) 수신 시각으로 정렬하고, 통지 SLA(수신~통지  
  1분) 자동 측정을 위해 시각 동기(NTP)를 전처리 요건에 포함함."

### (c) 작성 체크리스트

- [ ] 원천 데이터 검증·보정 대상(스키마·서명·중복·결측)을 나열했는가  
- [ ] 정규화 대상(코드·식별자·시각·집계 차원)을 나열했는가  
- [ ] 시각 정합성 등 SLA 측정 전제 전처리를 포함했는가

## G6. 핵심 처리·판정 대상 요구사항

> 다루는 항목: 6.1~6.n 판정/처리 기준(규칙·키워드·임계값·감정·AI 등). RFP: REQ-BIZ 핵심 기능·REQ-BIZ-05(AI).

### (a) 작성 원칙

- 시스템이 무엇을 근거로 처리·판정하는지 **판정 기준을 유형별(6.1~6.n)로 나눠** 명세함. 각 기준은 합격/불합격  
  또는 등급을 가르는 임계값을 동반함("측정할 수 없으면 요구가 아님").  
- 규칙 기반 판정과 AI 기반 판정을 구분하고, AI는 **확률형 판정**(임계값·표본·룰 폴백·HITL 검수)으로 서술하여  
  결정론적 합격을 과약속하지 않음.  
- 각 판정 기준은 정책(G7)·상태값(G11)과 연결되도록 결과 유형을 지정함.

### (b) 통신사 B2B 예시·문구

- **6.1 심각도 판정**: 사고유형·영향 규모·영향 서비스 조합으로 긴급/높음/보통 분류 — 분류 규칙 대사 100%  
  정확. [REQ-BIZ-01]  
- **6.2 통지 대상·채널 판정**: 심각도·담당자 등급·시간대로 대상·채널·발송 순서 결정 — 수신~담당자 도달 5분  
  이내(종단). [REQ-BIZ-01]  
- **6.3 영향범위 판정**: 자사 가입자 분포 × MNO 영향정보 조인으로 영향 규모·지역 산출 — 피크 300 TPS에서  
  평균 2초 & P99 3초, 표본 100건 대사 100% 일치. [REQ-BIZ-02]  
- **6.4 AI 영향예측 판정(확률형)**: 과거 유사사고로 영향 규모·확산 추이 예측 — 백테스트 오차율 ±15% 이내 건을  
  합격 집계, 표본 부족 시 룰 폴백, 결과는 참고용(HITL). [REQ-BIZ-05a, REQ-TEC-05]  
- **6.5 응대 지원 판정(확률형)**: 예상 문의 FAQ·응대 스크립트 초안 생성 — 통지 후 10분 이내 초안, 상담사  
  검수 후 사용. [REQ-BIZ-05b]

### (c) 작성 체크리스트

- [ ] 판정 기준을 유형별(6.1~6.n)로 나눠 명세했는가  
- [ ] 각 판정에 측정 가능한 임계값·판정방법이 있는가  
- [ ] AI 판정을 확률형(오차율·표본·룰 폴백·HITL)으로 서술했는가  
- [ ] 각 판정 결과가 정책(G7)·상태값(G11)과 연결되는가

## G7. 정책 요구사항

> 다루는 항목: 대상별·조건별 정책(등급·우선순위·조건부 예외)(7.1~7.n). RFP: 우선순위·정책.

### (a) 작성 원칙

- 동일 이벤트라도 **대상·조건에 따라 다른 처리 정책**을 적용할 수 있어야 함을 명세함(대상 등급·팀·시간대·  
  사고유형 기준). 정책은 G2 기본 원칙을 하위 규칙으로 구체화함.  
- 정책 간 충돌 시 적용 우선순위(G9 룰 우선순위와 연계)를 명시함.  
- 조건부 예외("예외가 있어도 추가 판정이 필요한 경우")를 별도 절로 두어 무조건 통과를 방지함.

### (b) 통신사 B2B 예시·문구

- **7.1 담당자 등급 기준 정책**: 신규 담당자(입사 90일 이내)는 통지 확인·에스컬레이션 강화, 숙련 담당자는  
  표준 정책 적용.  
- **7.2 심각도·채널 정책**: 긴급=전 채널 동시 발송+슈퍼바이저 알림, 높음=주 채널+백업, 보통=주 채널.  
- **7.3 시간대·구간 정책**: 야간·대량 클레임 구간은 통지 우선순위 상향 및 지연 판정 억제.  
- **7.4 조건부 추가 판정 정책**: 통지 억제 대상이라도 신규 사고유형·감정 급변·과거 클레임 고객 다수 포함 시  
  추가 판정을 수행함.  
- **우선순위 문구 예시**: "정책 충돌 시 상위 위반·규제 정책이 하위 허용 정책에 우선함(G9 우선순위 단계 준거)."

### (c) 작성 체크리스트

- [ ] 대상·조건별 정책을 유형별(7.1~7.n)로 명세했는가  
- [ ] 정책 충돌 시 적용 우선순위를 명시했는가  
- [ ] 조건부 추가 판정(무조건 통과 방지) 절을 두었는가  
- [ ] 정책이 G2 기본 원칙·G9 우선순위와 정합하는가

## G8. 특수·강화 정책

> 다루는 항목: 특정 조건(신규 대상·고위험 구간)에서 판정을 강화하는 규칙(나열). RFP: 리스크 대응.

### (a) 작성 원칙

- 리스크가 큰 특정 조건에서 표준 정책보다 **강화된 처리**를 적용하는 규칙을 나열함. 강화 트리거(대상·기간·  
  임계값)와 강화 내용을 함께 적음.  
- 강화 정책은 G2(오탐·미탐 관리)·G10(지연 처리)과 연동됨.

### (b) 통신사 B2B 예시·문구

- 신규 사고유형(전체·팀·담당자 최초 발생)은 실시간 판정 필수 + 슈퍼바이저 우선 알림.  
- 대량 통지 구간은 일부 대상 우선 통지 후 나머지 지연 리포팅.  
- 위험 등급 임계 초과 시 즉시 에스컬레이션 및 휴먼(HITL) 검토 우선 배정.  
- MNO 원천 데이터 지연 구간은 통지 성실성 기록을 강화하여 사후 면책 근거를 확보함.

### (c) 작성 체크리스트

- [ ] 강화 트리거(조건·기간·임계값)를 명시했는가  
- [ ] 강화 내용(실시간 필수·우선 통지·에스컬레이션 등)을 나열했는가  
- [ ] 강화 정책이 G10 지연 처리·G16 예외 케이스와 정합하는가

## G9. 룰/리스트 관리 요구사항

> 다루는 항목: 9.1 적용 단위(표), 9.2 우선순위(다단계 나열), 9.3 등록·승인, 9.4 관리·이력. RFP: 정책 관리·감사.

### (a) 작성 원칙

- 허용/억제 룰(WhiteList/BlackList 등)의 **적용 단위·우선순위·등록·이력**을 명세함. 적용 단위는 **표**로,  
  우선순위는 상위→하위 **다단계 나열**로 표현함(참조 양식 준수).  
- 상위 위반·규제 룰이 하위 허용 룰에 우선함을 명문화하여 허용 룰의 오남용을 방지함.  
- 등록·수정·해제 이력은 전량 감사 로그(G13)로 남김.

### (b) 통신사 B2B 예시·문구

- **9.1 룰 적용 단위(표)**

| 적용 단위 | 설명 |
|-----------|------|
| 전체 공통 | 모든 팀·채널에 공통 적용 |
| 부문/팀 | 부문·팀ID 기준 적용 |
| 담당자 | 담당자 사번 기준 적용 |
| 사고유형 | 장애/보안 유형 기준 적용 |
| 채널 | 통지 채널(SMS/이메일/API) 기준 적용 |
| 조합 | 팀 + 사고유형 + 채널 조합 기준 적용 |

- **9.2 우선순위(다단계 나열, 상위→하위)**: 규제·법적 즉시 대응 → 확정 억제(BlackList) → 긴급 운영 차단 →  
  대상 제재 상태 → 조합 허용룰 → 사고유형 허용룰 → 팀 허용룰 → 담당자 허용룰 → 전체 공통 허용룰 →  
  신규 사고 정책 → 일반 룰 판정 → 모니터링 정책. 상위 위반 정책에 해당하면 허용룰이 있어도 억제·보류함.  
- **9.3 등록·승인**: 등록 단위·대상·적용 기간·사유·요청자·승인자·변경 이력을 필수 기록하며, 즉시/예약/임시/  
  만료 지정 적용을 지원함.  
- **9.4 관리·이력**: 등록·수정·해제는 모두 감사 로그로 남김.

### (c) 작성 체크리스트

- [ ] 룰 적용 단위를 표로 표현했는가  
- [ ] 우선순위를 상위→하위 다단계 나열로 명시하고 상위 우선 원칙을 못 박았는가  
- [ ] 등록·승인 필수 항목(대상·기간·사유·승인자·이력)을 명세했는가  
- [ ] 룰 변경 이력이 감사 로그(G13)로 연결되는가

  ⚠ 주의문으로 명시함.  
- 통지 이력 등 SLA 근거 로그는 무결성 검증(해시 대사) 수단을 함께 요구함.

- **⚠ 주의문 예시**: "⚠ 개인정보 항목 조회는 조회자·시각·목적·조회 대상을 반드시 감사 로그에 기록하며, 최소  
  1년 이상 보관함. 통지 이력은 위변조 방지 로그(해시체인·전자서명·WORM)로 3년 이상 보관하고 무결성 검증  
  (해시 대사)에 100% 통과함."

### (c) 작성 체크리스트

## G16. 예외·보완 케이스

> 다루는 항목: "~하더라도 ~인 경우 ~함" X-RAY 스타일 조건부 예외 규칙 나열. RFP: 상충·엣지 케이스(A4).

### (a) 작성 원칙

- 표준 규칙만으로 판단이 어긋날 수 있는 **엣지 케이스**를, 참조 양식의 **X-RAY 스타일**("~하더라도 ~인 경우  
  ~함")로 나열함. 각 케이스는 [일반 규칙 성립] + [예외 조건] + [예외 처리]의 3요소를 갖춤.  
- 예외 케이스는 G2 기본 원칙·G7 정책·G11 상태값과 정합해야 하며, 상충 요구의 우선순위를 최종 확정함.  
- 벤더 책임 경계(면책 조건)를 예외 케이스로 명문화하여 정산 분쟁을 예방함.

### (b) 통신사 B2B 예시·문구

- "통지 억제 룰에 등록된 대상이더라도, 신규 사고유형이면 추가 판정을 수행함."  
- "통지 SLA를 충족하더라도, 원천 데이터가 MNO 미제공으로 지연된 경우 벤더 면책 판정으로 분류함."  
- "AI 영향예측 신뢰도가 임계치를 초과하더라도, 표본이 부족한 신규 사고유형이면 룰 기반 폴백을 병행함."  
- "영향범위 조회가 가능하더라도, 개인 식별이 가능한 소규모 집계 구간은 범위를 상향 집계하여 개인 특정을 차단함."  
- "지연 판정으로 전환되었더라도, 24시간 이내 완료되지 않으면 자동으로 슈퍼바이저에게 알림을 발송함."  
- "심각도가 보통으로 분류되었더라도, 동일 사고가 임계 시간 내 반복 수신되면 심각도를 상향 재판정함."

### (c) 작성 체크리스트

- [ ] 예외 케이스를 X-RAY 스타일("~하더라도 ~인 경우 ~함")로 나열했는가  
- [ ] 각 케이스가 [일반 규칙 + 예외 조건 + 예외 처리] 3요소를 갖췄는가  
- [ ] 벤더 면책 조건 등 정산 경계를 예외 케이스로 명문화했는가  
- [ ] 예외 케이스가 G2·G7·G11과 정합하는가
