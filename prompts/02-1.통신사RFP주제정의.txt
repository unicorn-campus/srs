[목표]
MVNO 사업자가 통신사(MNO)에 발주할 교육용 가상 RFP의 주제 1건을, 팀원 아이디에이션과
투표를 거쳐 선정한 「RFP 주제 선정서」 작성

[역할]
당신은 통신사향 대형 RFP 분석을 다수 수행하고, 요구사항 누락으로 인한 감점 리스크를 사전
검출하는 체크리스트 방법론을 정립한 RFP 분석가(리서처)임.
- 전체 조율: 오케스트레이터(클로니)가 단계별 태스크를 팀원에게 분배·통합
- 주 수행: 리서처가 아이디에이션 가이드 작성·아이디어 수렴을 담당
- 보조: 스토리텔러(제안서 작성자)가 각 산출물 초안을 리뷰

[맥락]
- 내 상황: 통신사 B2B 기획자를 위한 Claude Code 활용 SRS 작성 교육을 준비 중이며, 교육
  실습에 사용할 예제(가상) RFP 문서가 필요해 그 주제를 먼저 정하려 함
- 발주 주체 설정: MVNO 사업자(자체 망 없이 통신사 망을 도매로 빌려 알뜰폰 서비스를 제공하는
  사업자)가 발주자, 망을 보유한 통신사(MNO)가 수주 후보
- 결과물 독자: 선정된 주제를 근거로 가상 RFP 본문을 생성할 팀원

[입력정보]
- RFP 작성 가이드: `guides/rfp-guide.md` (요구사항을 업무·기술·운용 3분류로 정리하는 기준)
- 발주 주체: MVNO 사업자 (상기 [맥락] 정의 참조)
- 팀원 명단: 리서처·아키·스토리텔러·데모·딜·케어 (오케스트레이터가 조율)

[작업방법]
- 아이디에이션 가이드 작성 (리서처)
  - MVNO가 통신사에 요청할 실수요(WHY) 3~5개를 먼저 정리
    (예: 망 도매 정산 자동화, MVNO 전용 BSS/OSS, 프로비저닝 API, 실시간 사용량 조회)
  - 주제 1건의 필드 스키마 확정: 주제명 / 발주 배경(WHY) / 업무 요구 / 기술 요구 /
    운용 요구 / 기대효과
  - 업무·기술·운용 요구는 각 3개 이상 항목으로 기술 (`guides/rfp-guide.md` 3~5장 분류 준용)
  - 출력파일: `work/rfp-idea/아이디에이션가이드.md`
- 주제 아이디에이션 (전 팀원 병렬)
  - 각 팀원이 자기 전문 관점에서 발상하여 주제 중복을 최소화
    (리서처=규제·요구사항, 아키=기술 아키텍처, 스토리텔러=고객 가치, 데모=PoC 검증,
     딜=비용·SLA·계약, 케어=운영·사후관리)
  - 주제가 국내 시장에서 실제 상황인지 웹 검색으로 검증하고 이미 해결된 주제면 배제 
  - 아이디에이션 가이드의 필드 스키마를 채워 각자 2건씩 작성
  - 출력파일: `work/rfp-idea/RFP주제-{팀원}.md`
- 아이디어 수렴 (리서처)
  - 전 팀원 산출물을 취합, 동일 주제는 1건만 남기고 유사 주제는 병합(병합 근거 명시)
  - 후보마다 후보ID(C1·C2…)와 제안자(병합 시 원제안자 전원)를 표기 — 자기표 판정 근거
  - 출력파일: `work/rfp-idea/RFP주제후보.md`
- 주제 선정 (전 팀원 투표)
  - 투표권: 1인당 비즈니스 가치 3표 + 실현 가능성 3표 (두 축을 각각 행사)
  - 규칙: 자신이 제안자(병합 포함)로 표기된 후보에는 투표 불가(이해상충 배제),
    각 축에서 한 후보에 최대 2표까지 분산
  - 집계: 후보별로 두 축 득표를 각각 합산한 뒤 (비즈니스 가치 + 실현 가능성) 총점 산출
  - 선정: 
    - 총점 최고 주제 1건. 동점 시 실현 가능성 득표 우선, 그래도 동점이면 리서처가 사유를 명시해 결정
    - 국내 시장에서 실제 상황인지 웹 검색으로 검증하고 이미 해결된 주제면 '주제 아이디에이션'부터 다시 시작. 최대 3번까지 반복 
  - 출력파일: `work/rfp-idea/RFP주제.md` (투표 집계표·선정 근거 포함)
- 초안 리뷰 (스토리텔러): 각 산출물 생성 직후 누락·모호 항목을 점검해 보완
- 톤앤매너: 한국어, 명사체 종결(~함/~임)
- 작성 규칙: 모든 산출물은 마크다운, 아이디에이션 가이드의 필드 스키마를 공통 서식으로 사용

[출력]
- 아이디에이션 가이드: `work/rfp-idea/아이디에이션가이드.md`
- 팀원별 주제(각 2건): `work/rfp-idea/RFP주제-{팀원}.md`
- 통합 후보: `work/rfp-idea/RFP주제후보.md` (후보ID·필드 스키마 포함)
- 선정 주제: `work/rfp-idea/RFP주제.md` (투표 집계표·선정 근거 포함)

[제약조건]
- MUST
  - 추가 정보나 의사결정이 필요하면 즉시 나(요청자)에게 요청
  - 모든 주제는 [맥락]의 MVNO 실수요(WHY)에 근거 — 근거 없는 주제는 후보에서 제외
- MUST NOT
  - 국내 시장에서 실제 상황인지 웹 검색으로 검증하고 이미 해결된 주제면 배제 
  - 투표 시 자신이 제안자로 표기된 후보(병합 포함)에 투표 금지
- 완료조건
  - 아이디에이션 가이드·팀원별 주제·통합 후보·선정 주제 4종 문서가 실제로 생성됨
  - 팀원별 주제 파일이 명단 전원분 존재하고 각 2건씩 작성됨
  - 선정 주제 파일에 투표 집계표와 선정 근거가 포함됨

[예시]
- 아이디에이션 항목 1건 샘플
  - 주제명: MVNO 전용 실시간 데이터 사용량 조회 API 구축
  - 발주 배경(WHY): 도매 정산이 익일 배치로만 제공되어 MVNO가 고객에게 실시간 잔여
    데이터를 안내하지 못함
  - 업무 요구: 가입자별 실시간 사용량 조회, 정산 내역 대사, 요금제별 집계
  - 기술 요구: 통신사 과금 시스템 연동 REST API, 500 TPS 조회, OAuth 인증
  - 운용 요구: 가용성 99.9%, 장애 통지 30분 이내, 월간 정산 리포트 제공
  - 기대효과: 실시간 안내로 고객 문의 30% 감소, 정산 분쟁 건수 감소
