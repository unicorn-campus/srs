[목표]
RFP 요구사항을 근거로 솔루션요구사항정의서(SRS)를 작성할 때 팀원이 공통으로 참고할
SRS 작성 가이드 문서 작성

[역할]
당신은 통신사 B2B 솔루션 SRS 작성을 다수 수행하고, ISO/IEC/IEEE 29148 기반 요구사항 명세
표준화와 인수 기준 명문화로 범위·정산 분쟁을 사전 차단해온 요구사항 정의 전문가(스펙)임.
- 전체 조율: 오케스트레이터(클로니)가 단계별 태스크를 팀원에게 분배·통합
- 주 수행: 스펙이 아젠다 확정·정보 통합·가이드 확정을 담당
- 보조: 리서처·아키·스토리텔러·데모·딜·케어 전 팀원이 역할별 정보수집·초안 작성·리뷰에 참여

[맥락]
- 내 상황: 통신사 B2B 기획자를 위한 Claude Code 활용 SRS 작성 교육을 준비 중이며, 확정된
  RFP(`work/rfp/RFP-{주제}.md`)를 근거로 팀원이 공통 기준으로 SRS를 작성할 수 있도록
  가이드 문서가 먼저 필요함
- 결과물 독자: 이후 실제 SRS 본문과 RFP 분석서(04단계)를 작성할 팀원 전체
- 선행 사례: `guides/rfp-guide.md`(02-1 산출물)가 동일 목적의 선례 — 섹션마다
  (a)작성원칙 (b)통신사 B2B 예시·문구 (c)작성 체크리스트 3단 구조로 구성됨

[입력]
- RFP: `work/rfp/RFP-{주제}.md`
- RFP 작성 가이드(구조 선례): `guides/rfp-guide.md`
- SRS 주요 항목(초안 스키마):
  - 고객사 문제 정의
  - 제안 솔루션의 기능·범위
  - 기술 요건 · 데이터 요건
  - 인수 기준(Acceptance Criteria)
- 입력 게이트: `work/rfp/RFP-{주제}.md`가 없으면 즉시 나(요청자)에게 보고하고 중단 —
  임의 RFP로 대체하지 않음

[처리]
- 입력 파악 및 이해 (스펙)
  - RFP 존재 확인 — 부재 시 [입력] 입력 게이트 규칙에 따라 중단·보고
  - `guides/rfp-guide.md`의 3단 구조(작성원칙·예시·체크리스트) 숙지 — SRS 가이드도
    동일 구조로 작성
- SRS 가이드 아젠다 작성
  - 아젠다 초안: 스펙이 SRS 주요 항목을 근거로 Level1 섹션 구조로 작성
  - 아젠다 리뷰(병렬): 아키(기술·데이터 요건 실현 가능성) · 리서처(RFP 요구 반영 완결성) ·
    딜(인수 기준·정산 리스크) 3인이 각자 관점에서 누락·중복 점검
  - 출력파일: `work/srs/srs-agenda.md`
  - 아젠다 확정 후 나(요청자)에게 승인 요청 — 승인 전 다음 단계 진행 금지
- SRS 가이드 정보수집 (전 팀원 병렬: 리서처·아키·스토리텔러·데모·딜·케어)
  - 본인 기존 지식과 신뢰성 있는 외부 자료(출처 URL 필수)를 확정 아젠다 섹션별로 배치
  - 출력파일: `work/srs/srs정보수집-{팀원}.md`
- SRS 가이드 초안 작성 (전 팀원 병렬)
  - 확정 아젠다 섹션을 팀원별로 분담해 작성, 정보수집 결과를 근거로 반영
  - 출력파일: `work/srs/srs-guide-{팀원}.md`
- SRS 가이드 1차안 통합 (스펙)
  - 팀원별 초안을 아젠다 순서로 취합, 중복·모순 항목 조정
  - 출력파일: `work/srs/srs-guide-draft.md`
- SRS 가이드 1차안 리뷰 (전 팀원 병렬)
  - 각 팀원이 본인 전문 관점에서 누락·모호 항목을 지적 목록으로 정리
- SRS 가이드 확정 (스펙)
  - 리뷰 지적사항 반영, 확정 아젠다의 전 섹션 충족 여부 최종 점검
  - 출력파일: `guides/srs-guide.md`
- 톤앤매너: 통신사 B2B 기획 실무자가 실제 SRS 작성에 바로 활용 가능한 실무 지침 수준
- 작성 규칙:
  - `guides/rfp-guide.md`와 동일하게 섹션마다 (a)작성원칙 (b)통신사 B2B 예시·문구
    (c)작성 체크리스트 3단 구조 준수
  - 한국어, 명사체 종결(~함/~임), 한 줄 120자 이내
  - 각 섹션에 인수 기준(측정 가능한 완료 기준) 서술 방식의 예시를 최소 1개 포함

[출력]
- SRS 가이드 아젠다: `work/srs/srs-agenda.md`
- SRS 가이드 정보수집서: `work/srs/srs정보수집-{팀원}.md` (팀원별 각 1개)
- SRS 가이드 초안: `work/srs/srs-guide-{팀원}.md` (팀원별 각 1개)
- SRS 가이드 1차안: `work/srs/srs-guide-draft.md`
- SRS 가이드(최종): `guides/srs-guide.md`

[제약조건]
- MUST
  - 추가 정보나 의사결정이 필요하면 즉시 나(요청자)에게 요청
  - 아젠다는 나의 승인 후에만 정보수집·초안 작성 단계로 진행
  - 모든 항목은 RFP(`work/rfp/RFP-{주제}.md`)와 [입력]의 SRS 주요 항목에 근거 —
    근거 없는 임의 추가 금지
  - 각 섹션은 `guides/rfp-guide.md`와 동일한 3단 구조(작성원칙·예시·체크리스트)로 작성
- MUST NOT
  - RFP 부재 시 임의 RFP로 대체 금지
  - 아젠다 미승인 상태에서 정보수집·초안 작성 등 후속 단계 진행 금지
- 완료조건 (검증 가능한 증거 기준)
  - [출력]의 산출물 5종이 지정 경로에 실제로 생성됨을 파일 존재로 확인
  - 최종 가이드(`srs-guide.md`)가 확정 아젠다의 전 섹션을 포함하고, 섹션마다
    (a)작성원칙 (b)예시 (c)체크리스트가 모두 존재
  - 아젠다 확정 시점에 나의 승인 여부가 산출물에 기록됨

[예시]
(아젠다 섹션 예시) "## 3. 기술 요건" 섹션의 3단 구조
- (a) 작성원칙: RFP 2-2 기술 요구를 정량 지표로 재정의
- (b) 예시: "API 응답시간 3초 이내 (RFP REQ-TECH-02 근거)"
- (c) 체크리스트: "☐ 모든 기술 요건에 측정 가능한 지표가 존재하는가"
