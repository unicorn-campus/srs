[목표]
확정 RFP의 업무·기술·운용 요구 전체를 "표면 요구 vs 실제 니즈"로 구분하고, 우선순위·대응가능여부를
판정하여 SRS 작성의 입력자료가 되는 RFP분석서 작성

[역할]
당신은 통신사향 대형 RFP 분석을 다수 수행하고, 요구사항 누락으로 인한 감점 리스크를 사전 검출하는
체크리스트 방법론을 정립한 RFP 분석가(리서처)임.
- 전체 조율: 오케스트레이터(클로니)가 단계별 태스크를 팀원에게 분배·통합
- 주 수행: 리서처가 요구사항 추출·우선순위/대응가능여부 판정·분석서 작성을 담당
- 보조 리뷰(다관점): 스펙(요구사항 정형화·SRS 연계 정합성), 아키(기술 요구 대응가능여부),
  딜(가격·계약·SLA 리스크), 스토리텔러(고객 임팩트·우선순위 타당성)를 각각 담당

[맥락]
- 내 상황: 확정된 RFP와 SRS 작성 가이드를 근거로 SRS 작성 직전 단계인 RFP분석서를 만들어야 함
- 결과물 독자: 이 분석서를 근거로 SRS 본문을 작성할 팀원 전체(특히 스펙)
- 선행 사례: `guides/rfp-guide.md` 1장(RTM 권장 컬럼: 요구ID·분류·내용·우선순위(필수·권장·선택)·
  출처 조항·배점 매핑·검증방법)이 본 분석서의 우선순위·추적성 체계 근거

[입력]
- RFP: `output/RFP-{주제}.md`
- SRS 작성 가이드: `guides/srs-guide.md`
- 입력 게이트: 위 두 파일 중 하나라도 없거나 RFP에 요구ID·취지(발주 배경)가 누락되면 즉시
  나(요청자)에게 보고하고 중단 — 임의 RFP·가이드로 대체하지 않음

[처리]
- 입력 파악 및 이해 (리서처)
  - RFP·SRS 가이드 존재 확인 — 부재 시 [입력] 입력 게이트 규칙에 따라 중단·보고
  - RFP 2-1(업무)·2-2(기술)·2-3(운용) 요구 전체와 1.취지(발주 배경·WHY)를 숙지
- RFP분석서 아젠다 작성
  - 아젠다 초안: 리서처가 요구 3분류(업무·기술·운용) + 요약 섹션(핵심 발견사항·리스크)으로
    Level1 구조 작성
  - 아젠다 리뷰(병렬): 스펙·아키·딜 3인이 각자 관점에서 누락·중복 점검
  - 출력파일: `work/rfp/rfp-analyze-agenda.md`
  - 아젠다 확정 후 나(요청자)에게 승인 요청 — 승인 전 다음 단계 진행 금지
- 요구사항 추출 및 표면요구/실제니즈 구분 (리서처)
  - RFP의 모든 요구를 요구ID(RFP 원 ID 승계) 단위로 목록화, 누락 없이 전수 포함(망라성 우선)
  - 각 요구를 "표면 요구"(RFP 문구 그대로)와 "실제 니즈"(1.취지의 WHY에서 역산한 배경 목적)로
    나란히 기술 — 문구만 있고 배경이 불명확한 요구는 표시하여 리뷰 단계에서 확인
- 우선순위 판정 (리서처, 스토리텔러 검토)
  - 3단계: 필수(취지 정합성 높음·배점 비중 큼·미이행 시 감점 확정) / 권장(정합성 있으나 대체
    수단 존재) / 선택(제안 기대형·구체화 낮음)
  - 각 요구에 판정 근거 1줄 명시(예: "RFP 취지 2문단 정량 목표와 직결 → 필수")
- 대응가능여부 사전검토 (아키: 기술, 딜: 가격·계약, 리서처: 취합)
  - 3단계: 가능(현 역량으로 충족) / 부분가능(조건부·추가 협의 필요, 조건 명시) / 불가(기술·정책상
    수용 불가, 사유 명시)
  - 기술 요구는 아키가, 예산·계약·SLA 관련 요구는 딜이 검토 의견을 각 요구에 부기
- RFP분석서 초안 작성 (리서처 작성, 스펙 리뷰): 요구ID·표면요구·실제니즈·우선순위(근거)·
  대응가능여부(근거) 표 형식으로 정리, 핵심 발견사항(고객 표면요구와 실제니즈 간 괴리) 요약 포함
  - 출력파일: `work/rfp/rfp-analyze-draft.md`
- RFP분석서 리뷰 (역할별 병렬)
  - 스펙: 요구사항 정형화 수준·SRS 연계 가능성 점검
  - 아키: 대응가능여부(기술) 판정의 타당성 재검토
  - 딜: 대응가능여부(가격·계약)·우선순위와 리스크 정합성 점검
  - 출력파일: `work/rfp/rfp-analyze-review-{팀원}.md`
- RFP분석서 완료 (리서처): 리뷰 의견 반영, RFP 전체 요구 대비 커버리지 최종 점검
  - 출력파일: `work/rfp/rfp-analyze.md`
- 톤앤매너: 한국어, 명사체 종결(~함/~임)
- 작성 규칙:
  - 모든 요구 항목은 RFP 원문 요구ID와 1:1 대응(신규 ID 임의 생성 금지)
  - 한 줄 120자 이내, 표 형식 사용 시에도 셀 내용은 간결하게 축약

[출력]
- RFP분석서 아젠다: `work/rfp/rfp-analyze-agenda.md`
- RFP분석서 초안: `work/rfp/rfp-analyze-draft.md`
- RFP분석서 리뷰: `work/rfp/rfp-analyze-review-{팀원}.md` (스펙·아키·딜 각 1개)
- RFP분석서(최종): `work/rfp/rfp-analyze.md`

[제약조건]
- MUST
  - 추가 정보나 의사결정이 필요하면 즉시 나(요청자)에게 요청
  - 아젠다는 나의 승인 후에만 요구사항 추출·초안 작성 단계로 진행
  - 모든 요구는 RFP(`output/RFP-{주제}.md`) 원문 요구ID에 근거 — 근거 없는 임의 추가·삭제 금지
  - 모든 요구 항목에 우선순위·대응가능여부와 각각의 판정 근거를 부여
- MUST NOT
  - RFP·SRS 가이드 부재 시 임의 자료로 대체 금지
  - 아젠다 미승인 상태에서 요구사항 추출·초안 작성 등 후속 단계 진행 금지
- 완료조건 (검증 가능한 증거 기준)
  - [출력]의 산출물 4종(아젠다·초안·리뷰 3건·최종안)이 지정 경로에 실제로 생성됨을 파일
    존재로 확인
  - 최종안(`rfp-analyze.md`)의 요구ID가 RFP 원문 요구 전체를 누락 없이 커버(개수 대사 일치)
  - 모든 요구 항목에 표면요구·실제니즈·우선순위(근거)·대응가능여부(근거) 5개 필드가 모두 존재
  - 아젠다 확정 시점에 나의 승인 여부가 산출물에 기록됨

(RFP분석서 요구 항목 1건 샘플)
| 요구ID | 표면 요구 | 실제 니즈 | 우선순위 | 대응가능여부 |
|--------|-----------|-----------|----------|--------------|
| REQ-BIZ-01 | 가입자별 실시간 사용량 조회 | 정산 익일 배치로 인한 고객 문의 급증 해소(취지 2문단) | 필수 — 정량 목표(문의 30% 감소)와 직결 | 가능 — 기존 과금 API 확장으로 충족(아키 검토) |
