[목표]
RFP분석서와 SRS 작성 가이드를 근거로, 참조 양식(개발요구사항정의서)과 동일한 16장 구성의 통신사 B2B
솔루션요구사항정의서(이하 SRS)를 마크다운 최종안 + Word 문서로 작성

[역할]
당신은 통신사 B2B 솔루션 SRS 작성을 다수 수행하고, ISO/IEC/IEEE 29148 기반 요구사항 명세와 인수 기준
명문화로 범위·정산 분쟁을 사전 차단해온 요구사항 정의 전문가(스펙)임.
- 전체 조율: 오케스트레이터(클로니)가 단계별 태스크를 팀원에게 분배·통합
- 주 수행: 스펙이 아젠다 확정·섹션 통합·최종 SRS 확정·Word 변환 총괄을 담당
- 보조(다관점): 아키(시스템 역할 경계·데이터·성능·아키텍처 실현가능성), 케어(운영자 화면·감사 로그·운영),
  딜(성능 SLA·정책·계약 리스크), 데모(예외 케이스·판정 상태값·검증 시나리오), 스토리텔러(목적·기본원칙
  서술 명료성)를 각각 담당

[맥락]
- 내 상황: 통신사 B2B 기획자 대상 Claude Code SRS 작성 교육용으로, 현업 개발·구축팀이 그대로 구현 근거로
  쓰는 "개발요구사항정의서" 양식(시스템 역할 경계·데이터 필드 명세·판정 정책·결과 상태값·성능·예외 케이스
  중심)에 맞춘 SRS가 필요함. 기존 v1은 ISO 29148·추적성 RTM 중심이라 개발·구축팀이 바로 쓰기엔 성격이 다름
- 목표 산출물 성격: 수주 벤더가 그대로 구현 근거로 사용하는 개발요구사항정의서 — 참조 양식과 동일한 16장
  골격·표·나열·예외 케이스 구성을 재현함
- 결과물 독자: 수주 벤더 개발팀·아키텍트, 사내 개발·영업·법무, 이 SRS로 06단계 제안서를 작성할 기획자
- 선행 사례: `work/rfp/rfp-analyze.md`가 요구ID·우선순위·대응가능여부를, `guides/srs-guide.md`가 작성원칙·
  체크리스트를 제공 — 두 산출물의 요구를 아래 "16장 골격 표준"에 재배치하여 작성

[입력]
- RFP분석서: `work/rfp/rfp-analyze.md`
- SRS 작성 가이드: `guides/srs-guide.md`
- 문서 골격 표준: 아래 [처리]의 "16장 골격 표준"을 목표 구조로 사용(참조 양식과 동일 구성)
- 입력 게이트: 위 두 파일 중 하나라도 없으면 즉시 나(요청자)에게 보고하고 중단 — 임의 자료로 대체하지 않음

  - RFP분석서의 요구ID별 표면요구·실제니즈·우선순위·대응가능여부를 숙지하고, 아래 "16장 골격 표준"과

  - 아젠다 리뷰(병렬): 아키(데이터·성능·경계 실현가능성)·케어(운영·감사·화면)·딜(성능 SLA·정책 리스크)·

  - 아젠다 확정 후 나(요청자)에게 승인 요청 — 승인 전 다음 단계 진행 금지
- SRS 섹션별 초안 작성 (역할 분담, 병렬) — 16장을 4개 클러스터로 분담
  - 클러스터A(1.목적·2.기본원칙·3.시스템 역할 경계): 스펙 작성(수행/미수행 경계 명확 분리), 스토리텔러 리뷰
  - 클러스터B(4.데이터 요구사항·5.전처리·12.결과 필수 포함 정보): 아키 작성(필드 단위 상세 나열), 스펙 리뷰
  - 클러스터C(6.판정 대상·7.정책·8.강화 정책·9.룰/리스트·10.지연 흐름·11.결과 상태값·16.예외 케이스):
    스펙 작성, 데모 리뷰(상태값·예외 케이스 검증가능성)
  - 클러스터D(13.감사 로그·14.운영자 관리 화면·15.성능 요구사항): 케어 작성, 딜 리뷰(성능 SLA·계약 연계)
  - 출력파일: srs-draft-{클러스터}.md (클러스터별 각 1개)
- SRS 1차안 통합 (스펙)
  - 클러스터 초안을 16장 순서로 취합, RFP 요구ID 기준 커버리지 교차 검증(누락·중복 확인)
  - 참조 구성 재현 점검: 표 4종 이상(9장 룰 적용단위·11장 결과 상태값·14장 운영자 화면·15장 성능),
    ⚠ 주의문(4·13장), X-RAY 예외 케이스(16장), 수행/미수행 역할 경계(3장)가 모두 존재하는지 확인
  - 출력파일: srs-draft.md
- SRS 리뷰 (역할별 병렬)
  - 전 팀원이 각자 전문 관점에서 누락·모호·측정 불가능한 기준을 지적 목록으로 정리
  - 출력파일: srs-review-{팀원}.md
- SRS 최종안 확정 (스펙)
  - 리뷰 지적사항 반영, RFP분석서 요구ID 대비 커버리지 최종 대사(누락 없음 확인)
  - 최종 문서는 참조 양식과 동일하게 머리말(제목·안내·문서정보) → 1~16장 → 꼬리말 순으로만 구성함.
    승인상태·리뷰반영내역·통합검증메모·추적성 매트릭스(RTM) 등 방법론 부록은 최종 문서에 넣지 않음
    (작업 과정 산출물은 work/ 하위에만 보관)
  - 출력파일: output/솔루션요구사항정의서.md
- SRS Word 문서 변환 (스펙, docx 스킬 사용)
  - 확정된 output/솔루션요구사항정의서.md를 아래 "Word 문서 구성 규격"에 따라 참조 양식과 동일 구성의
    Word로 변환하고, 변환 후 PDF 렌더링으로 제목·나열·표 4종 구성이 참조 양식과 일치하는지 육안 확인
  - 출력파일: output/솔루션요구사항정의서.docx

- 16장 골격 표준 (참조 양식과 동일 구성 — 각 장에 [내용 성격]·[RFP 매핑] 병기)
  머리말
  - 문서 제목: "문서. {솔루션명} 사업 요구사항"(볼드)
  - 안내 문구: 교육용 가상 문서 고지 1줄
  - 문서 정보: "{솔루션명} 사업 요구사항 V1.0 / {수행 조직} {작성자}"(볼드)
  1. 목적 [문단+나열] — 시스템이 무엇을 하는지 1문단 정의, 시스템 역할 한정(수행 기능 나열), 본 시스템이
     담당하지 않고 레거시/타 시스템이 담당하는 업무 명시. RFP: WHY·REQ-BIZ 전반
  2. 기본 원칙 [소제목+문단+나열] — 솔루션 설계·판정의 핵심 원칙 4개 내외(2.1~2.4), 각 원칙에 설명 문단과
     근거 나열. RFP: 실제니즈·제약·상충(A4)
  3. 시스템 역할 경계 [나열] — 3.1 수행하는 업무(나열), 3.2 수행하지 않는 업무(나열). RFP: 범위/제외범위
  4. 입력 데이터 요구사항 [주의문+소제목+나열] — ⚠ 개인정보·보안 주의문, 4.1~4.n 도메인별 데이터 항목을
     필드 단위로 상세 나열. RFP: REQ-TEC(데이터)·REQ-BIZ(조회 대상)
  5. 데이터 가공·검증(전처리) 요구사항 [소제목+문단+나열] — 입력 정규화·검증·보정 대상. RFP: 데이터 요건
  6. 핵심 처리·판정 대상 요구사항 [소제목+나열] — 6.1~6.n 판정/처리 기준(규칙·키워드·임계값·AI 등).
     RFP: REQ-BIZ 핵심 기능·REQ-BIZ-05(AI)
  7. 정책 요구사항 [소제목+문단+나열] — 대상별·조건별 정책(등급·우선순위·조건부 예외). RFP: 우선순위·정책
  8. 특수·강화 정책 [나열] — 특정 조건(신규 대상·고위험 구간) 강화 규칙. RFP: 리스크 대응
  9. 룰/리스트 관리 요구사항 [표+나열] — 9.1 적용 단위(표), 9.2 우선순위(다단계 나열), 9.3 등록·승인,
     9.4 관리·이력. RFP: 정책 관리·감사
  10. 신규·예외 흐름 및 지연 처리 요구사항 [소제목+나열] — 10.1 신규 판단 기준, 10.2 실시간/지연 전환,
      10.3 지연 상태값. RFP: REQ-TEC(성능·이중화)·운영
  11. 처리 결과 상태값 [표] — 상태 / 정의 / 조건 / 후속 조치(다단계). RFP: 판정 결과·후속 대응
  12. 결과 필수 포함 정보(필드) [나열] — 결과·응답에 포함할 필드 전량 나열(추적·설명가능성). RFP: 설명가능성
  13. 감사 로그 요구사항 [나열+주의문] — 로그 대상 이벤트 나열, ⚠ 보관기간·개인정보 조회 기록 주의문.
      RFP: REQ-TEC(무결성·보관)
  14. 운영자 관리 화면 요구사항 [표] — No / 화면 / 주요 기능 / 권한. RFP: 운영·REQ-OPS
  15. 성능 요구사항 [표] — 구분 / 요구사항 / 기준(응답시간·처리량·동시처리·가용성·RTO·RPO·로그보관 등
      정량 지표). RFP: REQ-TEC(NFR)·REQ-OPS(가용성)
  16. 예외·보완 케이스 [나열] — "~하더라도 ~인 경우 ~함" X-RAY 스타일 조건부 예외 규칙 나열.
      RFP: 상충·엣지 케이스(A4)
