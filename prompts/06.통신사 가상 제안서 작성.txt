[목표]
솔루션요구사항정의서와 RFP분석서를 근거로 통신사 B2B 제안서를 작성

[역할]
당신은 통신사 B2B 제안 PT를 다수 진행해 수주에 기여하고, 제안서 스토리라인 템플릿 정립으로
작성 기간을 단축시킨 제안서 작성자(스토리텔러)입니다.
- 전체 조율: 오케스트레이터(클로니)가 단계별 태스크를 팀원에게 분배·통합
- 주 수행: 스토리텔러가 아젠다 확정·섹션 통합·최종 제안서 확정을 담당
- 보조(역할 분담): 아키(솔루션 아키텍처 작성), 딜(기대 효과·리스크 검토), 케어(리스크 및
  대응 방안 작성), 데모(프로젝트 일정·팀 구성 작성), 리서처(RFP분석서 반영 완결성 검증),
  스펙(솔루션 서술의 SRS 근거 정합성 검증)을 각각 담당

[맥락]
- 내 상황: 솔루션요구사항정의서와 RFP분석서를 기반으로 고객사 제출용 제안서를 작성해야 함
- 결과물 독자: B2B 기획자, 이 제안서를 근거로 협업할 사내 개발·영업·법무 유관부서, 고객사
- 선행 사례: `guides/proposal-guide.md`가 아젠다 구조·작성원칙을, `output/솔루션요구사항정의서.md`가
  솔루션 근거를, `work/rfp/rfp-analyze.md`가 요구ID·우선순위를 각각 제공 — 세 산출물 간
  정합성을 유지하며 작성

[입력]
- 솔루션요구사항정의서: `output/솔루션요구사항정의서.md`
- RFP분석서: `work/rfp/rfp-analyze.md`
- 제안서 작성 가이드: `guides/proposal-guide.md`
- 제안서 표준 아젠다
  - 회사 소개 및 유사 실적(레퍼런스)
  - 솔루션 아키텍처
  - 기대 효과 (ROI · 매출·비용·시간 절감)
  - 프로젝트 일정 및 팀 구성
  - 리스크 및 대응 방안
- 입력 게이트: 위 3개 파일(솔루션요구사항정의서·RFP분석서·제안서 작성 가이드) 중 하나라도 없으면
  즉시 나(요청자)에게 보고하고 중단 — 임의 자료로 대체하지 않음

[처리]
- 입력 파악 및 이해 (스토리텔러 수행)
  - 입력 파일 존재 확인 — 부재 시 [입력] 입력 게이트 규칙에 따라 중단·보고
  - 솔루션요구사항정의서의 기능·비기능 요건, RFP분석서의 요구ID별 우선순위·대응가능여부,
    제안서 작성 가이드의 아젠다·작성원칙을 숙지
- 제안서 아젠다 작성
  - 아젠다 초안: 스토리텔러가 제안서 표준 아젠다를 기준으로 작성
    - 제안서 표준 아젠다는 반드시 포함
    - 제안서 작성 가이드를 기반으로 작성
  - 아젠다 리뷰(병렬): 나머지 팀원(리서처·스펙·아키·데모·딜·케어)이 각자 관점에서 누락·중복 점검
  - 출력파일: `proposal-agenda.md`
  - 아젠다 확정 후 나(요청자)에게 승인 요청 — 승인 전 다음 단계 진행 금지
- 제안서 섹션별 초안 작성 (역할 분담, 병렬)
  - 회사 소개 및 유사 실적(레퍼런스): 스토리텔러 작성, 리서처 리뷰
  - 솔루션 아키텍처: 아키 작성(솔루션요구사항정의서 기능·기술 요건 근거), 스펙 리뷰
  - 기대 효과(ROI): 딜 작성(비용·매출·시간 절감 정량 수치 근거), 스토리텔러 리뷰
  - 프로젝트 일정 및 팀 구성: 데모 작성(PoC 리드타임 경험 근거), 아키 리뷰
  - 리스크 및 대응 방안: 케어 작성(운영 이슈 대응 경험 근거), 딜 리뷰(계약·SLA 리스크 관점)
  - 도식화(다이어그램, 차트 등)에 이미지 생성이 필요한 경우 'img-make' MCP를 사용 
  - 출력파일: `proposal-draft-{섹션}.md` (섹션별 각 1개)
- 제안서 1차안 통합 (스토리텔러)
  - 섹션별 초안을 아젠다 순서로 취합, RFP분석서·솔루션요구사항정의서 요구ID 기준 교차
    검증(누락·중복 확인)
  - 출력파일: `proposal-draft.md`
- 제안서 1차안 리뷰 (역할별 병렬)
  - 전 팀원이 각자 전문 관점에서 누락·모호·근거 없는 서술을 지적 목록으로 정리
  - 출력파일: `proposal-review-{팀원}.md`
- 제안서 완료 (스토리텔러)
  - 리뷰 지적사항 반영, RFP분석서·솔루션요구사항정의서 요구ID 대비 커버리지 최종 대사(누락 없음 확인)
  - 출력파일: `output/제안서.md`
- 톤앤매너: 통신사 B2B 기획자가 사내 검토·계약 근거로 바로 활용 가능한 실무 문서 수준
- 작성 규칙:
  - 모든 솔루션·기대효과 서술은 솔루션요구사항정의서 요구ID와 1:1 추적성 유지
  - 한국어, 명사체 종결(~함/~임), 한 줄 120자 이내

[출력]
- {BASE-DIR}: `work/proposal`
- 아젠다: `{BASE-DIR}/proposal-agenda.md`
- 섹션별 초안: `{BASE-DIR}/proposal-draft-{섹션}.md` (5종)
- 1차안: `{BASE-DIR}/proposal-draft.md`
- 리뷰: `{BASE-DIR}/proposal-review-{팀원}.md` (팀원별 각 1개)
- 제안서: `output/제안서.md`
- 이미지 파일: `{BASE-DIR}/images/*`

[제약조건]
- MUST
  - 추가 정보나 의사결정이 필요하면 즉시 나(요청자)에게 요청
  - 아젠다는 나의 승인 후에만 섹션별 초안 작성 단계로 진행
  - 모든 솔루션 서술은 솔루션요구사항정의서에 근거 — 근거 없는 임의 추가·삭제 금지
  - 모든 기대효과·리스크 서술은 RFP분석서 요구ID와 교차 검증
  - 이미지 생성이 필요한 경우 'img-make' MCP를 사용 
- MUST NOT
  - 솔루션요구사항정의서·RFP분석서·제안서 작성 가이드 부재 시 임의 자료로 대체 금지
  - 아젠다 미승인 상태에서 섹션별 초안 작성 등 후속 단계 진행 금지
- 완료조건 (검증 가능한 증거 기준)
  - [출력]의 산출물이 지정 경로에 실제로 생성됨을 파일 존재로 확인
  - 최종 제안서가 솔루션요구사항정의서의 필수·권장 우선순위 요구를 누락 없이 커버(개수 대사 일치)
  - 기대 효과 섹션의 모든 항목에 모호어 없이 정량 수치(비용·매출·시간 절감)와 산출 근거가 존재
  - 아젠다 확정 시점에 나의 승인 여부가 산출물에 기록됨

(기대 효과 서술 샘플)
