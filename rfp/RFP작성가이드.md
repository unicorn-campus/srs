## 이 가이드 사용법

- 각 섹션(## 1.~## 9.)은 동일한 3단 구조로 구성됨.  
  - (a) 작성 원칙: 이론 원문 근거로 "무엇을 왜 쓰는가"를 규정함  
  - (b) 통신사 B2B 예시·문구: 예제 RFP에 그대로 넣을 수 있는 각색 예시·문장을 제시함  
  - (c) 작성 체크리스트: 누락 방지용 자가점검 항목을 제시함  
- 교육용 예제 RFP를 만들 때는 아래 "예제 RFP 조립 순서"를 따라 섹션을 순차로 채움.  
- 핵심 원칙은 "구체성보다 망라성 우선"임 — 항목을 빠뜨리지 않는 것이 상세히 쓰는 것보다 중요함 (p.13, p.18).  
- SLA·가용성·NFR 등 여러 섹션이 겹치는 주제는 한 곳(주로 4·5 섹션)에 정리하고 다른 섹션에서 상호 참조함.

### 예제 RFP 조립 순서

1. 워크시트 빈칸(섹션 9-2)을 준비함 → 각 칸에 요구ID를 부여하여 RTM(요구사항 추적표)로 연결함  
2. 섹션 순서대로 채움: 취지(2) → 업무 요구(3) → 기술 요구(4) → 운용 요구(5) → 예산(6) →  
   프로젝트 기간(7) → 특기사항(8)  
3. 3대 요구(무엇을·얼마에·언제까지)가 모두 명기되었는지 확인함 (p.7~8)  
4. 문서 말미의 "통신사 B2B RFP 작성 종합 체크리스트"로 최종 누락 점검 후 확정함  
5. 이론 완성 샘플(상세판, p.108~112) 구조와 대조하여 정합성을 확인함

### 통신사 B2B RFP 확장 골격 (이론 5개 섹션 + 확장)

- 0. 인사문 → 1. 사업 개요·추진 배경(취지) → 2. 요구사항(2-1 업무 / 2-2 기술 / 2-3 운용)  
- 3. 사업비·대가 산정 → 4. 사업 기간·일정 → 5. 제안 안내(평가기준·배점·제출요령) →  
  6. 계약 조건(SLA·NDA·저작권)  
- 통신사 특성상 "운용 요구"에 SLA 정량 지표를 명시하는 것이 일반 IT RFP 대비 핵심 차별점임  
  (공공SW 요구사항 유형화 및 ISO/IEC/IEEE 29148 3계층 매핑 근거)[^fesi][^iso29148]

## 1. RFP의 3대 핵심과 문서 골격

### (a) 작성 원칙

- RFP는 HW·SW·서비스 구입 요건을 정리하여 발주사(구매자)가 벤더(판매자)에게 제안을 의뢰하는 문서임.  
  작성 측과 읽는 측의 이해가 반드시 일치하지 않으므로, 양자가 WIN-WIN으로 나아가도록 정보를 정확히  
  전달하는 수단임 (p.6).  
- IT 업계에 표준 서식은 없으나 "만들고 싶은 시스템의 개요를 벤더에게 정확히 전달"한다는 목적에서  
  기본 작성법은 정해짐 (p.7).  
- RFP에 필요한 정보는 3가지로 집약됨 (p.7, 그림1-2).  
  - 무엇을 하고 싶은가 → 업무 요구·기술 요구·운용 요구  
  - 얼마에 하고 싶은가 → 예산  
  - 언제까지 하고 싶은가 → 납기  
- 거대 시스템이 수백 페이지여도 이 3대 요구로 집약된다는 점은 불변임. RFP를 받는 벤더는 3개 요구가  
  반드시 포함되어 있는지 체크함 (p.7~8).  
- RFP 본체 표준 골격 (p.11~12): 0. 시작(인사문) / 1. 요지(취지) / 2. 시스템에 요구하는 기능(2-1 업무·  
  2-2 기술·2-3 운용) / 3. 예산 / 4. 프로젝트 기간 / 5. 특기사항. 3요소 중 가장 중요한 것은 "무엇을  
  하고 싶다"이며 업무·기술·운용 요구로 나뉨 (p.13).  
- RFP 작성은 조달 4단계(①사전조사 ②RFP 작성 ③제안서 평가·결정 ④계약) 중 ② 작업임. RFP 공표는  
  조달 과정을 좌우하는 정보를 발신하는 마일스톤임 (p.8~9).  
- 작성 메리트: 요구 명확·구체화, 우선순위 명확화, 공평한 정보 제공, 복수 벤더 동시 의뢰로 시간 단축,  
  평가축 확보, 투자 대 효과 검토 계기. 단기(3개월~반년) 안건에서도 필수임 (p.10).  
- 미작성 리스크: 요구 애매로 과잉·일반론 제안, "말했다/안 했다" 전달 미스, 가격만으로 판단 (p.10).

### (b) 통신사 B2B 예시·문구

- 3대 요구의 통신사 B2B 구체 형태  
  - 무엇을(WHAT): 회선/전용선/MPLS/SD-WAN 등 서비스 종류, 커버리지(전국 국사·해외 PoP), 대역·이중화  
    구성, 보안(망분리·전용회선 암호화), 관리 포털·API 연동 요구  
  - 얼마에(HOW MUCH): 초기 구축비(회선 개통·장비)와 월 정액 이용료(러닝)를 분리 제시, 3~5년 TCO 기준  
  - 언제까지(WHEN): 회선 개통 리드타임(국사 실사→포설→개통)이 길어 컷오버 역산이 일반 IT보다 보수적  
- 통신사 RFP는 흔히 400개 안팎 질의 항목, 네트워크 사양용 중첩 Excel 매트릭스, 여러 PDF에 흩어진 SLA  
  요구, 복수 규제 프레임워크 참조 컴플라이언스 섹션으로 구성됨[^anchor]  
- 발주 주체가 공공기관·대기업인 경우가 많아 RFP가 곧 평가·계약의 법적 기준 문서로 작동하며, 평가위원  
  배점표와 1:1 매핑되므로 요구사항 누락은 곧 감점으로 직결됨

### (c) 작성 체크리스트

- [ ] 3대 요구(무엇을·얼마에·언제까지)가 모두 명기되었는가 (p.7~8)  
- [ ] 표준 골격 0~5 섹션이 모두 존재하는가 (p.11~12)  
- [ ] 각 요구사항에 요구ID를 부여하고 RTM으로 평가배점·제안서·계약까지 추적 연결했는가  
  (권장 컬럼: 요구ID / 분류 / 내용 / 우선순위(필수·권장·선택) / 출처 조항 / 배점 매핑 / 제안서 대응  
  페이지 / 검증방법 / 계약·SLA 반영)[^rtm-pm][^rtm-jama]  
- [ ] 사전조사(①) 단계에서 고객사 기존 회선·장비 현황 실사, 망 구성도, 국사/PoP 커버리지를 확보했는가

## 2. 취지(발행 배경) 작성

### (a) 작성 원칙

- 취지(1. 요지)는 "왜 시스템을 구축하는가"라는 목적·배경을 밝혀 경영의 의사를 전달하는 역할임 (p.14).  
  취지에 쓸 내용은 상류의 정보화 기획 단계에서 검토하며, 선택지를 열거한 뒤 자사 강점을 살릴 수 있는  
  것을 선택함 (p.14).  
- 취지 서술 3요소 스토리 (p.14). 이는 서사의 3막 구조(설정→갈등→해결)와 대응됨[^moxie]  
  1. 경영전략·경영자 생각을 압축한 키 메시지를 서두 배치 (설정: 지향하는 목적지)  
  2. 자사 경영환경·문제점에서 왜 이번 시스템이 필요한지 기술 (갈등: As-Is와 To-Be의 간극)  
  3. 비즈니스상 어떤 효과를 기대하는지 기술 (해결: 시스템이 간극을 메우는 수단)  
- 취지는 「경영/사업전략」과 「회사 개요(사업 내용·업계 동향)」로 구분됨 (p.15, 그림1-6). 배경이 제대로  
  전해질수록 벤더 제안 품질이 향상됨 (p.15).  
- 인사문(0. 시작) 명의는 중소기업이면 사장, 대기업·중견기업이면 CIO·부장 등 프로젝트 오너가 적절함 (p.14).  
- 상세판 샘플은 취지에 정량 목표를 포함함 — 예: 수주율 25% 이상(현재 20% 미만), 리피트율 75% 이상  
  (현재 50% 미만) (p.108). 기대효과는 "현재값 → 목표값" 대비 형식일 때 설득력이 가장 큼.  
- 키 메시지 압축 3단계: 수집(경영전략·CEO 메시지·중기계획 키워드) → 선별(WHY만 남기고 HOW는 이관)  
  → 응축("우리는 [무엇]을 통해 [어떤 상태]가 되고자 함" 단문). WHY First 원칙에 근거함[^sinek]  
- 문제점은 경영목표에서 역산함 — 경영목표 → 필요 비즈니스 역량 → 그 역량을 막는 현재 문제점 순으로  
  서술해야 논리가 이어짐 (IT-비즈니스 정렬 프레임워크 접근)[^opengroup]

### (b) 통신사 B2B 예시·문구

- 발주 배경 3유형과 키 메시지 톤  
  - DX(디지털 전환)형: "일하는 방식의 근본 전환" — 변화의 불가피성·시급성 강조  
  - 비용절감·효율형: "지속가능한 원가구조" — 정량 절감 목표를 전면 배치  
  - 신사업·매출확대형: "새로운 성장엔진" — 시장 기회·경쟁 우위 선점 논리 강조  
- 정량 목표(통신 KPI 각색) 예: 지점 간 통신 장애 연 X건 이하, 대역 부족 민원 Y% 감소, 개통 리드타임  
  Z일 단축  
- 문제점 장면화 예: "정보가 분산되어 있음"(추상) → "고객사 담당자가 회선 장애를 문의해도 계약·회선·  
  청구 이력을 한 화면에서 찾지 못해 응대가 지연됨"(장면)  
- 근거 동향: 기업은 2030년까지 매출의 10%를 디지털 전환에 지출 전망(DX형 근거)[^gsma]. 파이버·자동화·  
  클라우드 도입 선도 통신사는 IT 비용 최대 30% 절감·ARPU 10~15% 상승 가능(비용절감·신사업형 근거)[^mck].  
  지정학·데이터 주권·사이버보안 부상으로 AI·보안이 B2B 고객 우선순위 상위[^ey]

### (c) 작성 체크리스트

- [ ] 키 메시지 한 문장으로 프로젝트 존재 이유가 설명되는가 (WHY First)  
- [ ] 현재 문제점이 추상어가 아니라 구체적 장면·간극으로 서술되는가  
- [ ] 문제점이 경영목표에서 역산되어 논리적으로 연결되는가  
- [ ] 기대효과가 "현재값 → 목표값" 정량 대비로 제시되는가  
- [ ] 키 메시지와 기대효과가 수미상관을 이루는가  
- [ ] 발주 배경 유형(DX/비용절감/신사업)에 맞는 톤인가  
- [ ] 회사 개요·업계 동향이 벤더의 배경 이해를 돕도록 부가되었는가  
- [ ] 인사문 명의가 프로젝트 오너(CIO·부장 등)로 지정되었는가

## 3. 업무 요구 작성

### (a) 작성 원칙

- 업무 요구는 취지를 따라가며 추출함. 목표에 대해 "필요한 것, 현재 안 되어 있는 것, 있지만 불완전한 것"을  
  알아냄. 처음엔 랜덤이어도 자꾸 밝혀낸 뒤 중복·정리하여 종합함 (p.15).  
- 업무 요구 6계층 (p.15~16, 그림1-7)  
  - ① 업무 기본 요구: 근본적 업무 니즈(데이터 일원화 등). 기본 요구만으로도 충분한 효용이 있음  
  - ② 업무 기능 요구: 기본 요구를 실현하는 개별 기능(정보 입력·화면/장표 취출 등)  
  - ③ 업무 플로우 요구: 업무 처리 순서·흐름  
  - ④ 데이터 요구: 데이터 항목수·형식·테이블 구조·문자수 제약  
  - ⑤ 화면 요구: 입력·조회 화면의 항목 배치·표시 형식  
  - ⑥ 장표 요구: 출력 장표 항목·용지 규격(A4 가로 등)  
- 상세히 적을수록 개발 착수 후 요건정의 작업을 줄일 수 있으나, "이 레벨까지가 RFP"라는 절대 기준은  
  없고 기업·안건 사정에 따라 깊이가 다름 (p.16~17).  
- 구체화 vs 벤더 위임 트레이드오프: 시스템 요원이 부족하면 추상적 작성을 남기고 불명확한 부분은 벤더  
  제안에 맡겨도 됨. 단, 최소한 요구는 항목 레벨로 제시함. "구체성보다 망라성이 중요" (p.13, p.18).  
- 엔드유저 협력 표준 순서: ①앙케이트 예비정보 → ②설문 기반 인터뷰 → ③핵심 인력 RFP 검토 미팅·업무  
  견학(매니저 경유가 유리) → ④벤더 제시 전 여러 번 리뷰. 업무 견학은 참가 의식도 높임 (p.18~19).  
- 요구 표현: 비즈니스 요구(왜·무엇을)·기능 요구(어떻게)·비기능 요구(얼마나 잘)를 구분함. 6계층 중  
  ①은 비즈니스 요구, ②~⑥은 기능 요구에 가깝고, 성능·용량·보안 등 비기능 요구는 섹션 4로 이관함[^req-func][^altex]  
- 검증가능성(testability) 확보: 각 요구에 측정 가능한 확인 기준을 붙임. "쉽다·빠르다·많다" 정성 표현은  
  수량화해 재작성함(Given-When-Then). INVEST 중 Small·Testable이 검증가능성과 직결됨[^prolifics][^eitt][^invest]

### (b) 통신사 B2B 예시·문구

- 도메인 골격: 통신사 시스템은 OSS(망 성능·인벤토리·장애·프로비저닝)와 BSS(상품·주문·청구·고객)로  
  구분됨[^srm][^ossbss]  
- 업무 영역별 6계층 + PoC 확인 기준 예시  
  - 고객관리(CRM): 기본=분산 고객·계약 정보 단일 뷰 일원화 / 확인=법인 고객 1건 검색 시 소속 회선·  
    계약·청구가 1화면 표시, 조회 3초 이내  
  - 회선관리(Inventory): 기본=회선 자원 현황·상태 실시간 파악 / 확인=회선 10만 건 목록 조회·필터 후  
    엑셀 다운로드 오류 없이 완료  
  - 청구(Billing): 기본=사용량 기반 정확한 요금 산정 / 확인=표본 계약 100건 청구 산정 결과가 검증표와  
    100% 일치, 정정 건은 사유·이력 기록  
  - 개통·프로비저닝: 기본=주문 접수 후 개통까지 자동 처리 / 확인=신규 주문 1건이 자동 플로우로 개통  
    완료, 중간 실패 시 오류 단계·사유 표시 후 재처리 성공  
  - 주문관리: 기본=다채널(포털·콜센터) 주문 단일 접수·상태 관리 / 확인=상태 변경 시 담당자·고객 알림,  
    처리 지연 시 에스컬레이션  
- 확인 기준 변환 규칙(나쁨→좋음)  
  - "화면 조회가 빨라야 함" → "고객 검색 결과가 조회 요청 후 3초 이내 표시됨"  
  - "데이터가 정확히 이관되어야 함" → "기존 회선 100만 건 이관 시 정합률 99.99% 이상, 불일치 건은  
    오류 리포트로 출력됨"  
- 엔드유저 이원화: 사내 운영자(콜센터·개통·청구)는 "처리 효율·오류 감소", 고객사 관리자는 "가시성·  
  셀프서비스"를 원해 상충할 수 있으므로 상충 지점을 RFP 단계에서 식별해 우선순위를 명시함

### (c) 작성 체크리스트

- [ ] 업무 영역을 빠짐없이 나열했는가(망라성 우선, 구체성은 그다음) (p.13, p.18)  
- [ ] 각 요구에 측정 가능한 확인 기준(PoC 판정 기준)이 붙어 있는가  
- [ ] 각 기능 요구 앞에 원 비즈니스 요구(왜)가 명시되어 목적 상실을 방지하는가  
- [ ] 위임 요구라도 "기대 결과(확인 기준)"와 PoC 시나리오는 제시했는가  
- [ ] 엔드유저(사내 운영자·고객사 관리자) 양측을 분리 인터뷰하고 상충 우선순위를 정했는가  
- [ ] 6계층별 PoC 검증 난이도(상/중/하)를 표시해 벤더의 현실적 데모 계획을 유도했는가

## 4. 기술 요구 작성

### (a) 작성 원칙

- 기술 요구와 운용 요구를 합쳐 "시스템 요구"라 부름. 업무 요구는 정보시스템·현업이 협력하지만 시스템  
  요구 정리는 정보시스템 부서가 주체임. OS·언어·DB·미들웨어에 대해 "어떤 기술/제품을 선택하고 어떤  
  기능을 이용할지"를 정리함 (p.20).  
- 기술 요구 4대 분류 (p.21, 그림1-10 / 상세판 p.111)  
  - IT 인프라: OS·미들웨어, 하드웨어, 네트워크  
  - 시스템 능력: 퍼포먼스(성능), Capacity(용량), 확장성  
  - 장해 대책: HW·SW 장해, 네트워크 장해, 장해 시 대체안  
  - 보안(시큐리티): 정보 누출 방지, 액세스권 설정(예: 관리자·매니저·일반사원 3단계)  
- 성능·장해 대책은 각 요소까지 브레이크다운함. 추상적이라도 좋으니 케이스 바이 케이스로 망라함 (p.20~21).  
  작성 순서 권장: IT 인프라 → 시스템 능력(정량) → 장해 대책(가용성 목표) → 보안. 뒤 항목이 앞 스펙을  
  결정하므로 능력·장해 목표를 먼저 확정하고 인프라를 역산함.  
- 상세 지정형 vs 벤더 제안 위임형: 문언 차이가 커도 결과적으로 동일 제품이 제안되는 경우가 많아 위임형이  
  반드시 열등한 것은 아님. 금융 등 사회적 영향 큰 기업은 엄격 지정, 시스템 요원 부족 기업은 위임이 유리  
  할 수 있음 (p.21). 제품 지정 시에도 "능력에 맞는 요구"로 범용·시장수요 높은 제품을 택해 유지보수  
  리스크를 회피함. 마이너 언어·서포트 미확립 제품은 3년 후 리스크를 수반함 (p.22~23).  
- 기존 인프라 활용 요구가 실무에 자주 나타남(초기 투자 억제·설치 스페이스 부재·리스 잔존·이전 플랜 등).  
  Capacity(전체·동시 사용자수)는 HW 스펙 검토에 필수라 누락 시 벤더가 반드시 질문함 (p.24).  
- NFR은 [지표 + 목표값 + 측정조건] 3요소로 기술해야 제안·검수·SLA 연계가 가능함. ISO/IEC 25010  
  9개 품질특성에 매핑하면 누락 없이 망라됨[^iso25010][^arc42]. TPS는 Little's Law(TPS=동시사용자/  
  Request Interval) 기반으로 산정함[^littles]

### (b) 통신사 B2B 예시·문구

- IT 인프라: 회선/전용선/장비·PoP, 회선 대역폭·이중화 회선, 사업자망 연동 규격(전용회선·VPN·인터넷),  
  DMZ 구성. 배포 형태(온프레미스/프라이빗/퍼블릭/하이브리드) 선택 기준 명시. 공공·금융 대상 클라우드  
  제안 시 CSAP 인증 보유를 필수 조건으로 명시[^csap][^skshieldus]  
- NFR 정량 명시 양식(RFP 삽입용)[^jidum][^elancer-perf]

| 구분 | 지표 | 목표값 | 측정조건 |
|------|------|--------|----------|
| 성능 | 화면 평균 응답시간 | 2초 이내 | 동시접속 100명, 정상부하 시 |
| 성능 | 온라인 TPS | 500 TPS 이상 | 피크 시간대 |
| 용량 | 동시 사용자 | 1,000명 | 개통 1년차 기준 |
| 용량 | 데이터 증가량 | 500GB/년 | 3년 보관 전제 |
| 확장성 | 무중단 Scale-out | 지원 | 서비스 중단 없이 노드 증설 |

- 장해 대책 3단 연계: 가용성 목표(정량) → 이중화 구성(수단) → SLA 배상(계약). 가용성 목표만 못박고  
  이중화 방식은 위임하면 더 나은 아키텍처를 유도함  
  - 가용성 목표: 서비스별 목표 가용률(99.9% / 99.95% / 99.99%). 99.9%는 연 약 8시간 46분, 99.99%는  
    약 52분 다운타임 허용. 서버 이중화만으로 99.99% 달성 불가 — 네트워크·스토리지·DB·DNS까지 이중화  
    필요[^speedy]. 데이터센터 요구 Tier 등급(예: Tier 3 이상) 명시로 가용성 수준 정량 지정 가능[^tier]  
  - 재해복구(DR): RTO·RPO 정량 명시, DR 센터 거리·동기/비동기 복제  
  - 장해 대체안: Active-Active / Active-Standby, 자동 페일오버 시간(예: 30초 이내)  
- 보안: 회선 암호화·접근권한 3단계(관리자·운영자·조회자), 망분리(대상 업무·물리/논리 방식 지정)[^fsc][^mango],  
  개인정보보호법 안전성 확보조치 기준(접근권한 최소·지체없는 말소, 인터넷망 송수신 암호화 등) 준수 명시[^pipc-law][^cisp]  
- SLA 수치는 운용·계약 섹션(5·8)의 SLA 배상 조항과 반드시 일치시킴 → 상세는 5-2·8-8 참조

### (c) 작성 체크리스트

- [ ] 4대 분류(IT 인프라·시스템 능력·장해 대책·보안)를 모두 망라했는가  
- [ ] NFR을 [지표+목표값+측정조건] 3요소로 정량 명시했는가 (ISO/IEC 25010 특성 매핑)  
- [ ] Capacity 수치(전체·동시 사용자, 대역, 트래픽 피크)를 제시했는가 (누락 시 벤더 반드시 질문, p.24)  
- [ ] 가용성 목표 → 이중화 → SLA 배상 3단이 연계되고 5·8 섹션과 수치가 일치하는가  
- [ ] 규제 항목(망분리·CSAP·개인정보 암호화)은 상세 지정, 아키텍처·튜닝 수단은 위임으로 배분했는가  
- [ ] 기존 인프라 활용 자산·리스 잔여기간·연동 규격을 명시해 과잉 견적·연동 누락을 방지했는가

## 5. 운용 요구 작성

### (a) 작성 원칙

- 운용 요구는 실제 업무 이용을 상정한 운용 작업·보수 서비스·교육/연수 등에 관한 요구임. 개발에 관심이  
  쏠려 경시되기 쉬우나, 시스템은 적절히 운용해야 투자 가치를 제공하므로 추상적이라도 빠짐없이 조사함 (p.25).  
- 운용 요구(광의) 5분류 (p.25, 그림1-13 / 상세판 p.111)  
  - (협의) 운용 요구: 일상 운용·백업 등 운용 계획  
  - 보수 요구: HW·SW·네트워크 보수 서비스  
  - 교육·연수 요구: 엔드유저·시스템 요원 교육  
  - 이행(전환) 요구: 본가동 이행, 특히 리플레이스 시  
  - 아웃소스 요구: 운용 전부/일부 아웃소싱  
- 가장 보편적 요구는 "시스템 운용 부하 경감"임. "영업시간 = 시스템 가동 시간 = 보수 대상 시간"이 기본  
  사고이며, 필요 레벨을 검토해 오버스펙(불필요한 24시간 365일 온사이트 등)을 피함 (p.26~27).  
- 제안 기대형 허용: 운용 요구는 구체화가 어려운 것이 사실이므로 "무엇무엇을 제안하고 싶다"처럼 순수  
  제안을 바라는 요구를 스트레이트로 제출해도 무방함. 이를 통해 뜻밖의 좋은 제안을 받을 수 있음 (p.27).  
- 엔드유저 트레이닝은 도입 성공 필수 업무이므로 반드시 항목에 넣음. 누락 시 프로젝트 중·종반 예산 추가·  
  스케줄 무리를 부름 (p.28).  
- 운용 요구 회답으로 벤더 파트너십 성향(팔고 손 떼는 벤더 vs 컷오버 후부터 진짜 교제인 파트너형)을  
  판별 가능함. RFP는 유저와 벤더 간 정보의 캐치볼임 (p.25~27).

### (b) 통신사 B2B 예시·문구

- SLA 정량 지표(운용 요구 핵심). ITIL 4는 SLA를 "요구·기대 서비스 수준을 식별하는 문서화된 합의"로  
  정의함[^itil-alloy]. 대표 지표: 가용성/가동률, 응답·해결시간, 서비스 시간, 장애 통지 시간, 통신 특유  
  지표인 패킷 손실률·전송 지연[^etri]. SLA 조건: 측정 방식 명확·자동 반복 측정·장애 시 즉시 적용·담당자  
  교체와 무관하게 동일 해석[^elancer-sla]. 베스트프랙티스: 비즈니스 목표 정렬·적용/미적용 범위 정의·  
  평이한 언어·정기 리뷰·KPI 리포트[^bmc][^spoc]  
- 가용성 산식 Availability = MTBF/(MTBF+MTTR). 나인 하나가 늘면 허용 다운타임이 1/10로 감소. 4나인  
  이상은 수동 대응 불가로 이중화·자동 failover 전제[^hyperping][^azure]. 장애 등급별(긴급/높음/보통) MTTR  
  목표를 표로 제시하고, 에러 버짓(100%−목표)으로 월간 허용 장애 시간을 합의함  
- 운영지원 체계: 1선(헬프데스크)→2선(운영 엔지니어)→3선(개발/벤더) 에스컬레이션, SPOC 접수 창구,  
  정기점검(PM) 주기·사전 통지 리드타임, 백업·복구(RTO/RPO)·DR 훈련, 월간 SLA 리포트·KPI 대시보드,  
  변경관리(RFC·CAB·배포 창)  
- 보수 요구: 대상 범위(HW/SW/NW·서드파티 포함 여부), 응답 방식(온사이트/원격, 8×5/24×7, 4시간 이내),  
  예비품(핫스페어), 패치·버전업, 보수 등급 선택지(松竹梅=Gold/Silver/Bronze) (p.111). 유지관리비는  
  요율제로 산정하며 국내 표준은 KOSA "SW사업 대가산정 가이드"임(2024 개정: FP당 553,114→605,784원  
  9.52% 인상, 상용SW 요율 최대 20%)[^kosa][^koit] → 예산 연계는 6-2 참조  
- 이행(전환) 요구: 운영 런북(구축 중 동시 작성이 최고 가치), 3단계 지식 이전(Shadow→역할 분담→독립  
  운영, 빅뱅 대비 지식 격차 60% 감소), 문서화 필수, 사인오프 후 30/60/90일 리뷰, 전환 리스크 레지스터[^hitachi][^adapt][^opsio].  
  데이터 이행(정합성·리허설·롤백), 병행 운영·컷오버 창, 안정화(hypercare) 기간 SLA 적용, 기존 벤더 협조  
  책임 소재  
- 교육·연수: 대상별(엔드유저·운영 요원·관리자), 형태(집합/온라인·회차·인원), 교재·교육환경, 운영 이관  
  교육  
- 아웃소스: 범위(MSP·NOC/SOC·헬프데스크)와 책임 경계(RACI), 위탁에도 SLA·KPI 보고 의무, 인력  
  보안·교체 시 지식 이전

### (c) 작성 체크리스트

- [ ] 5분류(운용·보수·교육·이행·아웃소스)를 모두 검토했는가  
- [ ] SLA를 정량 지표(가용률·응답/복구시간·패킷손실·지연)로 명시하고 위약 기준을 연계했는가  
- [ ] 가용성 목표가 4·8 섹션 수치와 일치하는가 (상호 참조)  
- [ ] 서비스 시간을 "영업시간 기준"으로 산정해 오버스펙(불필요한 24×365)을 피했는가 (p.26~27)  
- [ ] 엔드유저·운영 요원·관리자 교육을 항목에 넣고 계상처(예산 내/별도)를 지정했는가 (p.28)  
- [ ] 이행 요구에 런북·지식 이전·안정화 기간·롤백 계획이 포함되었는가  
- [ ] 구체화 어려운 항목은 "제안 기대형"으로 남기되 기대 방향을 제시했는가

## 6. 예산 작성

### (a) 작성 원칙

- IT 투자 비용은 초기비용(이니셜)과 계속비용(러닝)으로 구분됨 (p.29~30, 표1-2).  
  - 이니셜: HW(서버·주변기기·랙·네트워크기기·PC·케이블), SW(OS·DB·미들웨어·개발툴·라이센스), 개발비  
    (요건정의·설계·개발·테스트·시방서·이행), 기타(컨설팅·연수·설치·하우징)  
  - 러닝: HW 보수, SW 보수(버전업 포함), 개발 보수(유상 개선/추가), 기타(회선요금·인터넷·헬프데스크)  
- 러닝 코스트는 3~5년까지 포함해 검토함. 상정 외 비용(SW 라이센스·데이터 이행·유저 연수)으로 예산  
  재검토·규모 축소를 겪는 사례가 잦음 (p.30).  
- 예산 공개: 원문 저자는 RFP에 예산 포함을 권장함. 제시 가능(권장) 조건 — IT 투자 경험 풍부, 투자 대  
  효과 검토 충분, 예산 거의 확정 (p.30). 메리트 — 현실적 견적, 벤더 간 동일 금액 제안으로 비교 용이,  
  선정 스피드업 (p.30~31). 단, 경험이 적어 전망이 어려우면 무리하지 말고 벤더 견적에 의지함(빗나간  
  제시는 교섭에 악영향). 제3자 어드바이스 활용 유효 (p.31).  
- 흥정(의도적 저가 제시)은 이론·실무 모두 비권장 — 좋은 조달로 연결되는 경우가 드물고 벤더 제안 품질  
  저하를 초래함. 단 "예기치 못한 비용 대비 보험"으로 사내용 예산을 남겨두는 것은 매니지먼트상 지혜임 (p.32~33).  
- 보수비·연수비 등 별도 예산 항목은 반드시 계상처(예산 내/별도)를 명시함. 연수는 무료가 아님(예: 2억원  
  예산의 3%인 600만원). 공적 보조금·조성금 검토도 기술 가능함 (p.33~34). 상세판은 "예산 초과 시 가능/  
  필수 범위 명시", "보수비 월간·연간 별도 견적", "실비정산"까지 요구함 (p.112).

### (b) 통신사 B2B 예시·문구

- TCO(총소유비용) 프레임 필수 — HW·SW 획득비는 TCO의 15~30%에 불과하고 70~85%가 에너지·유지보수·  
  다운타임·IT 전달비용임. 저가 초기제안이 고비용 운영으로 이어지는 함정을 차단하려면 "생애주기 총비용"으로  
  구성함[^ibm-tco][^pure-tco]  
- 통신사 과금 구조 분리 기재: 초기 개통비(NRC, Non-Recurring Charge)와 월정 이용료(MRC, Monthly  
  Recurring Charge)를 분리 요구하면 벤더 간 비교가 명확해짐. 운영비용에 회선 이용료·유지보수·헬프데스크와  
  트래픽 증가 시 대역 증설(over-usage) 비용 조건을 포함함  
- 자주 누락되는 별도 예산 항목(통신사 B2B): 데이터 이행·병행운영 비용, 회선 이중화·백업회선·DR 사이트  
  비용, 하자보수 종료 후 유상 유지관리비, SW 라이센스 갱신·버전업, 커스터마이징 맨먼스 단가(CR 대응  
  근거). "실비정산(교통비·출장비)"과 "정액"을 명확히 구분 요구함  
- 유지관리 요율: 통상 개발비의 10~15% 권고, 상용SW는 등급별 12%(5등급)~20%(1등급), 공공 평균 약  
  11.1%[^itdaily-rate][^hyperhire]. RFP에 요율 상한 또는 산정 근거(대가산정 가이드 준거) 제출을 요구함[^kosa2][^iitp]  
- 공개 전략(협상 우위): 총액 비공개하되 규모대(band) 제시, 또는 상한만 제시·하한 비공개하는 절충안을  
  통신사 B2B에서 자주 사용함

### (c) 작성 체크리스트

- [ ] 초기(이니셜)·계속(러닝) 비용을 구분하고 러닝을 3~5년 TCO로 검토했는가  
- [ ] NRC/MRC를 분리 기재하도록 요구했는가  
- [ ] 별도 예산 항목(데이터 이행·이중화·DR·유지관리·라이센스·CR 단가)의 계상처를 명시했는가  
- [ ] 연수비·보수비를 예산 내/별도 중 어디로 계상하는지 지정했는가 (p.33~34)  
- [ ] 유지관리 요율 상한 또는 산정 근거 제출을 요구했는가  
- [ ] 실비정산과 정액을 구분 요구했는가  
- [ ] 예산 공개/비공개(또는 band·상한) 전략을 안건 성격에 맞게 결정했는가

## 7. 프로젝트 일정 작성

### (a) 작성 원칙

- 완료일(컷오버)은 RFP 작성 시점에도 반강제적으로 결정되는 경우가 많으나, 개시일(킥오프)은 애매한  
  경우가 많아 주의가 필요함 (p.34~35).  
- RFP 발행에서 킥오프까지 통상 최소 2개월(경험 있어도 1개월) 필요함 (p.35, 그림1-19).  
  - RFP 발행 → (2주) 제안 수령 → (1~2주) 벤더 결정 → (1~2주) 조달·계약 → (1주) 킥오프. 총 7~11주+α  
- 개시일 미검토 시 "견적 받으면 즉시 시작"으로 오판하여 계약·요원 확정·서버 납입(3주) 등으로 착수가  
  지연되고 컷오버가 어려워지는 비극이 발생함 (p.35). 샘플은 RFP 제시(7/1) 후 2개월로 9/1 킥오프, 연말  
  연초 휴가 활용 1/5 본가동 설정 (p.36).  
- 시험운용: 기업 활동은 1개월 1사이클이므로 새 시스템은 1개월 시험운용이 상식임(미션 크리티컬은 더 장기).  
  본가동 전 1개월 확보 명시는 벤더가 그 시점까지 프로그램을 거의 완성해야 함을 뜻함 (p.36).  
- 업무 성수기(피크) 회피가 가장 중요한 스케줄 요구임. 피크 전에 시험운용·본가동을 마쳐 새 흐름에 익숙한  
  상태로 피크를 맞이하도록 설계함 (p.36).  
- 컷오버 확정 요인 (p.34, 그림1-18): ①외적 요인(법규·세율 개정) ②경영판단·영업상 이유 ③컷오버상  
  시간제약(연말연초·연휴·여름휴가 활용) ④개발 볼륨 예상(규모별 3·6·12개월) ⑤유저 지불 사정(연도말  
  검수 완료). 상세판은 "월말 마지막 주(25일~말일) 협의 회피"까지 명시함 (p.112).

### (b) 통신사 B2B 예시·문구

- 회선 개통 리드타임 반영: 국사 실사→포설→개통이 길어 컷오버 역산을 일반 IT보다 보수적으로 잡음  
  (섹션 1-b 상호 참조)  
- 일정은 곧 비용 리스크: 지연 시 요원 투입 기간 연장·병행운영 연장·위약금으로 직결됨. 컷오버일을 외적  
  요인으로 고정한 경우 역산 일정에 버퍼(예비기간)를 두지 않으면 지체상금 리스크가 커짐(지체상금은 8-6 참조)  
- 비즈니스 피크 캘린더 확보: 고객사 결산기·프로모션 성수기·연말정산 등 트래픽 피크 중 컷오버는 장애 시  
  SLA 위반·서비스 크레딧 리스크를 극대화하므로 일정 설계 단계에서 회피함  
- 검수-대금 연동: 검수 완료 마일스톤을 대금지급 조건과 연동해 명시함(착수금-중도금-잔금 분할, 각 단계  
  검수 합격 시 지급). 발주사 귀책(요구 확정·검수 지연)에 의한 지연은 지체상금 면책으로 균형 설계함

### (c) 작성 체크리스트

- [ ] 컷오버일과 킥오프일을 모두 명시하고 RFP→킥오프 7~11주+α를 반영했는가  
- [ ] 본가동 전 시험운용 기간(통상 1개월)을 확보했는가  
- [ ] 회선 개통 리드타임을 컷오버 역산에 반영했는가  
- [ ] 업무 성수기·비즈니스 피크 캘린더를 회피하도록 설계했는가  
- [ ] 검수 완료 마일스톤을 대금지급 조건과 연동했는가  
- [ ] 협의 스케줄 제약(예: 월말 마지막 주 회피)을 명시했는가

## 8. 특기사항·제출요령 작성

### (a) 작성 원칙

- 제안서 양식(용지·가로/세로) 지정 시 비교 검토 잡무가 편해짐. 구성 내용 지시 시 항목별 비교로 우열  
  판단이 가능하고 벤더도 요점 이탈을 피함 (p.43~44). 샘플 구성 지시: 시스템 개념도·HW 구성·SW 구성·  
  전체 스케줄·개발 계획·도입 계획·운용 계획·요원 체제 계획·연수 계획 및 견적 (p.12, p.43).  
- 견적은 항목별 내역(HW·SW·개발비·컨설팅·기타·경비)을 내도록 요구함. 지시 없으면 "시스템 一式"(명세  
  없는 총액)으로 제출되어 비교가 곤란함. 우량 벤더일수록 명랑회계(투명 견적)라 명랑성도 판단 재료임.  
  "당사 요구에 없더라도 부가가치 높은 것은 적극 제안 바람" 문언을 넣으면 벤더 변명을 차단함 (p.44).  
- 제출조건: 마감일시·제출처·제출형태·질문 창구를 기술함. RFP 제시~제출 기간은 통상 2~4주(대규모는  
  4주 이상). 너무 짧으면 검토 부족, 길면 긴장감 상실 (p.43). 사무국 정보(담당자·연락처)를 빠짐없이  
  기재하고, 프레젠테이션 일시는 후일 조정이 편함 (p.44).  
- 사무국·공평성: 제안서 수령은 동일 동시각 지정이 바람직함. Q&A는 공평하게(해석 문제·누락은 전 벤더  
  공유, 공평성 훼손 질문은 이유 전달 후 미회답, 기간 한정 메일링리스트 활용). RFP 제시는 반드시 설명회를  
  실시함 (p.41~42).  
- 계약 관련 조항: 저작권은 기본적으로 발주사 귀속, NDA(비밀유지) 체결 요청 (p.112).

### (b) 통신사 B2B 예시·문구

- 견적 명세를 협상·정산 기준선으로: 항목별 단가(맨먼스·라이센스·회선 단가)를 분리 요구해야 CR 시 단가  
  근거로 활용 가능함. NRC/MRC 분리·3~5년 TCO 기준 제출 요구로 저가 초기제안 함정을 차단함(6장 참조)  
- 저작권 분리 귀속(통신사 B2B 관행): "신규 개발 산출물은 발주사 귀속, 벤더 기존 보유자산(선행 저작물)은  
  벤더 귀속 + 발주사에 사용권 부여". 공공 SW사업은 원칙적으로 산출물 저작권을 국가·개발사 공동 소유하되  
  특약 조정 가능(소프트웨어진흥법 체계)[^swlaw][^swguide]  
- 하자보수(무상) vs 유지관리(유상) 경계 명확화: 무상 하자보수는 사업 완료일로부터 1년, "오류·결함 수정"에  
  한정됨. 추가개발·환경개선·추가 방문/인력·추가 교육·무상 업그레이드·개발자 상주(시범운영 지원)·UI 개선·  
  HW/SW 설정변경은 유상 유지관리로 구분함[^swcontract][^etnews][^softwarekr]  
- 지체상금(지연 페널티): 요율·산정기준·면책사유(발주사 귀책·불가항력)를 명시함. 국가계약 기준 요율 —  
  물품 제조·구매 1000분의 0.8, 용역 1000분의 1.3, SW사업 물품+용역 일괄 1000분의 1.5. 산식 = 계약금액  
  × 요율 × 지체일수[^easylaw][^gnca]  
- NDA는 RFP 배포 시점과 계약 체결 시점 두 번 필요함. RFP 자체가 사업전략·시스템 구성 정보를 담으므로  
  제안 참여 벤더 전원에게 RFP 수령 조건으로 NDA를 요구함. 표준 조항 — 비밀정보 범위·표시, 목적 외  
  사용금지, 제3자 공개 금지, 반환·파기, 계약기간(통상 2~3년), 위반 시 손해배상·위약벌[^nda-hanwha][^nda-tipa][^nda-rp].  
  통신사 B2B는 개인정보보호·망분리 준수 의무를 NDA와 함께 명시하는 것이 관행임  
- SLA 계약 설계: RFP 단계에서 SLA 지표와 미달 시 서비스 크레딧(페널티) 산정식을 제시하거나 벤더 제안을  
  요구함. 크레딧은 징벌이 아닌 인센티브 설계로 — 월 사용료의 일정 %를 '위험부담'으로 설정하고 기대목표치/  
  최소목표치 이원화로 균형 설계함[^elancer-sla][^cio-sla][^itdaily-sla] (SLA 지표 정의는 5-2 참조)  
- 제출조건은 "공평성=향후 계약 정당성"의 근거 — 공공·준공공 조달은 절차적 공정성이 계약 유효성에 직결됨

### (c) 작성 체크리스트

- [ ] 제안서 양식·구성(9개 항목)과 견적 항목별 명세(단가 분리)를 지정했는가  
- [ ] "부가가치 제안 환영" 문언을 넣어 벤더 변명을 차단했는가  
- [ ] 제출 마감일시(2~4주)·제출처·형태·질문 창구·사무국 연락처를 기재했는가  
- [ ] 제안서 동일 동시각 수령·공평한 Q&A·설명회 실시를 규정했는가  
- [ ] 저작권 귀속 원칙(분리 귀속)을 명시했는가  
- [ ] 하자보수(무상 1년) vs 유지관리(유상) 경계를 정의했는가  
- [ ] 지체상금 요율·면책사유를 명시했는가  
- [ ] NDA(제안·계약 2회)와 개인정보·망분리 준수 의무를 명시했는가  
- [ ] SLA 지표·서비스 크레딧 산정식(또는 제안 요구)을 제시했는가

## 9. RFP 샘플·워크시트 활용

### (a) 작성 원칙

- 완성 샘플(상세판, p.108~112) 구조 = 본문 2페이지 압축 샘플(p.11~12)의 확장 완성형  
  - 0. 시작(인사문) / 1. 제안 요청의 취지(정량 목표 포함) / 2. 시스템에 요구하는 기능(2-1 업무: 기본·  
    기능·플로우·계획·장표 / 2-2 기술: IT 인프라·시스템 능력·장해 대책·시큐리티 / 2-3 운용: 운용·보수·  
    교육/연수·이행·아웃소스) / 3. 예산(초과 시 범위·보수 월간/연간 별도·실비정산) / 4. 프로젝트 기간  
    (시험운용·협의 제약) / 5. 특기사항(제안서 구성·견적 명세·저작권·NDA·마감·사무국)  
  - 상세판은 업무 기능·플로우·장표 요구, 보수 松竹梅 선택지, 이행·아웃소스, 액세스권 3단계가 구체화됨  
    (p.109~111)  
- 워크시트(p.113~116)는 항목만 제시한 빈 템플릿으로 각 섹션 필수 기입 항목을 나열함(0 인사문 / 1 취지 /  
  2-1 업무 / 2-2 기술 / 2-3 운용 / 3 예산 / 4 기간 / 5 특기사항).  
- 조립 방법: 워크시트 빈칸을 섹션 순서(취지→업무→기술→운용→예산→기간→특기)대로 채우면 상세판과  
  동일 구조가 조립됨. 채우는 원칙 — 3대 요구 망라, 구체화 어려운 항목은 위임하되 최소 항목 레벨 제시,  
  별도 예산·계약 조항(저작권·NDA)·제출요령 명시 (p.113~116, p.7, p.18, p.33, p.44, p.112).

### (b) 통신사 B2B 예시·문구 (상세판 각색)

- 1. 취지: 정량 목표를 통신 KPI로(지점 간 통신 장애 연 X건 이하, 대역 부족 민원 Y% 감소)  
- 2-2 기술: IT 인프라→회선/전용선/장비·PoP, 시스템 능력→대역·동시세션·확장성, 장해 대책→이중화·  
  우회경로·자동 절체, 시큐리티→망분리·회선 암호화·접근권한 3단계(관리자·운영자·조회자)  
- 2-3 운용: 운용→NOC 24/365 모니터링, 보수→온사이트 SLA 등급(松竹梅=Gold/Silver/Bronze),  
  이행→기존 회선 무중단 전환, 아웃소스→관제 위탁 범위  
- 5. 특기사항: 개인정보보호법·망분리 규제 준수 서약, 저작권·NDA, 정보보안 서약서  
- 통신 B2B 필수 추가 워크시트 칸: SLA 지표표, Capacity 수치표, 커버리지(국사/PoP) 목록, 규제 준수  
  체크란. 각 빈칸에 RTM 요구ID를 부여함  
- 국내 공공기관은 사업유형별 제안요청서 작성 가이드·예시(워크시트형 표준 양식)를 배포하므로 통신·정보화  
  RFP 작성 시 준용하면 요구사항 상세화 누락을 줄일 수 있음[^swit-guide][^swasc]  
- 통신 RFP 응답은 엔지니어링·상품·법무·컴플라이언스·재무·영업 다부서 기여가 필요하므로 섹션별 오너십을  
  지정함(리서처는 요구사항 오너로 배점 매핑 담당). SLA는 서비스 설명·측정 지표·양자 책임·위약·지표 추가/  
  삭제 프로토콜을 포함함[^anchor][^cio-sla]

### (c) 작성 체크리스트 (제출 전 셀프 리뷰 품질 게이트)

- [ ] 워크시트 빈칸을 섹션 순서대로 빠짐없이 채웠는가(망라성 우선)  
- [ ] 각 빈칸에 RTM 요구ID를 부여해 제안서·평가·계약까지 추적 가능한가  
- [ ] 통신 B2B 필수 추가 칸(SLA 지표·Capacity·커버리지·규제 준수)을 반영했는가  
- [ ] 상세판 샘플(p.108~112) 구조와 대조해 누락 섹션이 없는가  
- [ ] 섹션별 오너십(담당)을 지정하고 배점 매핑을 완료했는가
