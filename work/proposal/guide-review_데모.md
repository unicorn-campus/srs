# 제안서 작성 가이드 리뷰 — 데모(PoC 엔지니어) 관점

> 리뷰 대상: `work/proposal/proposal-guide-draft.md`  
> 근거 대조: `work/proposal/제안서작성이론.md`(작성 원칙 (p.XX) 근거)  
> 리뷰 범위: §4 PoC 검증 · §7 소구력 「利」 실증 · §9 데모 시연(담당 3개 지점) + 3개 지점 관통 정합성  
> 관점: 담당 정확성 · 논리 흐름/교육생 이해도 · 정직성(실측 vs 예시)

## 1. 종합 판정

**조건부 합격 — 담당 3개 지점의 골격·실무성은 우수하나, "실측" 표기의 정직성 가드와 섹션 간 수치 정합성  
보완이 필요함.**

강점  
- §4 PoC를 "완벽한 데모 약속"이 아닌 "검증 가능한 계획"(6요소: 검증항목·가설·성공기준·베이스라인·측정방법·  
  기간/환경)으로 규정한 것은 실무적으로 정확하며, 성공기준·베이스라인 부재 시 합·불 판정 불가라는 명시도  
  타당함.  
- §9 데모의 **3단 폴백(라이브 → 녹화 → 스크린샷) · 독립 환경 · 기기 이중화 · 오류 시 짧은 사과 후 피벗**은  
  Fast fail 원칙에 정확히 부합하는 강한 서술임.  
- PoC(§4) → 「利」 실증(§7) → 라이브 데모(§9)로 이어지는 "계획 → 근거 → 재현" 사슬과 상호참조(§6 별지·§7·§9  
  링크) 구조는 논리 흐름이 자연스럽고 교육생이 따라가기 쉬움.  

보완 필요(핵심)  
- **정직성**: 예시 수치는 대부분 "예시"로 표기됐으나, "실측"이라는 단어를 쓸 수 있는 조건(PoC 실제 수행 여부·  
  타이밍)이 불명확하여, 실측이 없는데 "실측"으로 오기할 여지가 있음(지적 3·4).  
- **정합성**: §9 내부에서 데모 스텝 순서·수치가 어긋나고(지적 2), 외부 인용 수치 1건은 출처 검증이 필요함  
  (지적 1).  

## 2. 지적 목록

### [높음] §4·§7·§9 관통 — PoC 실측치의 "타이밍 전제" 미명시로 인한 정직성 위험

- 위치: §7 (line 442 ~ 449, line 465 체크리스트) · §9 예시 (line 544 "제안서 §7의 실측치를 방금 실시간으로  
  재현") · §4 (line 274 ~ 277)  
- 문제: §7은 "利의 최강 증거는 PoC/파일럿 실측치", "실측 결과로 쓰라"고 하고 §9 예시는 "§7의 실측치를 재현"  
  이라 하여 **제안서 제출 시점에 이미 실측치가 존재함**을 전제함. 그러나 통신 B2B 다수는 PoC가 숏리스트·심사  
  단계 또는 수주 후에 수행되어, 제출 시점엔 실측치가 없는 경우가 흔함. 타이밍 전제가 불명확하면 "실측 없는데  
  실측으로 표기"하는 부정직을 유발함(정직성 원칙 위반).  
- 권고: §4 또는 §7 서두에 **「利」 근거의 3분기**를 1줄로 명시함 — ①사전 PoC 수행 → "실측치"로 표기,  
  ②타사·자사 유사 구축 레퍼런스 → "레퍼런스 실적"으로 표기, ③미수행 → "예상치(가정 명시)"로 표기. §9 라이브  
  데모는 "PT 현장 실측"임을 별도 구분함.  

### [중간] §9 — 데모 스텝 순서·수치가 요약(line 534)과 예시(line 540 ~ 544)에서 불일치

- 위치: §9 (b) line 534 vs line 540 ~ 544  
- 문제: 3스텝 요약은 "장애 주입 → 30초 내 영향범위 자동조회 → 통지 발송"(조회 → 통지 순)이나, 5슬롯 예시는  
  "주입 → 통지(step3) → 영향범위 조회(step4)"(통지 → 조회 순)로 **순서가 뒤바뀜**. 또한 요약은 "30초 내  
  영향범위 자동조회", 예시·§4 표·§7은 "영향범위 조회 P95 2초"로 **동일 동작을 다른 수치로 서술**함(30초 vs  
  2초). 동일 섹션 내 모순은 실측 신뢰성을 깎고 교육생 이해를 저해함.  
- 권고: 스텝 순서를 한쪽으로 통일하고, "영향범위 산출/조회" 수치를 전 섹션(§1·§2·§4·§7·§9)에서 하나로  
  고정함. "영향범위 자동산출 30초"와 "조회 응답 P95 2초"가 서로 다른 지표라면 명칭을 구분해 병기함.  

### [중간] §9 line 536 — "영업의 81%가 나쁜 데모로 딜을 잃은 경험" 수치 출처 검증 필요

- 위치: §9 (b) line 536, 각주 [^demo-navattic]  
- 문제: 구체 수치 "81%"를 단정적으로 인용함. 해당 출처(Navattic "Why Most Demos Fail")에 이 수치가 실재하지  
  않으면 "근거 없는 수치"로 정직성 원칙에 위배됨. 리뷰 시점에 원문 대조가 되지 않아 확정 불가함(검증 필요  
  항목).  
- 권고: [^demo-navattic] 원문에서 "81%" 문구를 직접 확인함. 확인되면 "출처 기준"임을 명기하고, 확인되지  
  않으면 수치를 삭제하거나 "다수 영업이 ~ 경험"으로 정성 서술로 완화함.  

### [중간] §7 「利」 — "실측/예상" 표기 규율과 금액 환산 가정 명시가 부재

- 위치: §7 (a) line 442 ~ 443, (b) line 447 ~ 449, (c) line 465  
- 문제: "실측 결과로 쓰라"는 권고는 옳으나, **실측이 없을 때 예상치를 "실측"으로 표기하지 말라는 정직성 가드가  
  없음**. 또한 "利 = Before → After + 금액 환산"에서 금액 환산은 가정(단가·손실률·트래픽 등) 위에 성립하나  
  가정·산출 근거 명시 요구가 없어, 근거 없는 ROI 수치가 나올 수 있음.  
- 권고: (c) 체크리스트에 "실측/레퍼런스/예상을 구분 표기했는가", "금액 환산의 가정·산출식을 별지에 명시했는가"  
  2항을 추가함. line 465의 "PoC 실측으로 제시"는 "실측 or 레퍼런스 or 예상(가정 명시)"로 완화함.  

### [낮음] §4 PoC 표(line 279 ~ 283) — 베이스라인 표기·PoC/파일럿 행 혼재

- 위치: §4 (b) line 279 ~ 283  
- 문제: 성공기준 열은 "(정량, 예시)"로 표기됐으나 **베이스라인 열(15분·수십 분·단일채널 유실)은 예시/측정대상  
  표기가 없어** 확정 사실처럼 읽힘. 베이스라인은 가정이 아니라 PoC 중 측정·사전 합의 대상임. 또한 한 표에  
  PoC 행(2주)과 파일럿 행(4주)이 섞여 있어 §4 (a)의 "PoC와 파일럿 구분" 원칙과 표가 어긋나 보임.  
- 권고: 베이스라인 열 머리에 "(현행·측정/합의 대상)" 표기를 추가함. 표에 "구분(PoC/파일럿)" 열을 넣거나 표를  
  둘로 분리해 (a) 서술과 정합화함.  

### [낮음] §9 line 543 — 라이브 데모 단건 조회로 "P95 2초"를 시연한다는 표현의 기술적 부정확

- 위치: §9 (b) line 543 "대시보드에서 영향 고객·회선 범위를 즉시 조회(P95 2초)"  
- 문제: P95(95백분위수)는 다수 표본으로 산출하는 지표이므로 **라이브 데모의 단건 조회로는 "P95 2초"를 증명할  
  수 없음**. 데모에서 관측되는 것은 단건 응답 지연임. 엄밀성 관점의 소소한 과장 소지가 있음.  
- 권고: 데모 슬롯은 "이 조회 응답 약 2초(단건 관측)"로, "P95 2초"는 §4 PoC 측정 결과로 구분 표기함. 데모=단건  
  관측, PoC=P95 측정임을 명확히 함.
