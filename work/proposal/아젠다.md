## 1. 문서 목적·범위

### 1-1. 근거와 범위

- 유일한 1차 근거: RFP+제안서매뉴얼 **제2장 「RFP에 근거한 제안서 작성방법」(p58~107)** 전문과  
  **제안서 샘플·워크시트(p117~124)**임. 모든 진술에 근거 페이지 (p.XX~XX)를 표기함.  
- 자료1 RFP 상세판(p108~112) · 자료2 RFP 워크시트(p113~116)는 **발주자(RFP) 산출물**이므로 본 가이드  
  범위 밖이며, 필요 시 rfp-guide 상호참조로만 다룸.  
- 본 가이드가 다루는 범위는 **"벤더가 RFP를 받은 뒤 제안서를 작성·제출·발표하는 단계"에 한정**함  
  (RFP 수령 → 취독·분석 → 제안서 작성 → 리뷰·제출 → 프레젠테이션, p90~107). RFP 요구 정의 자체는  
  다루지 않고 rfp-guide로 넘김.

### 1-2. rfp-guide 대비 관점 반전(핵심 전제)

- rfp-guide(제1장) = **발주자(구매자)가 RFP에 요구를 정의**하는 관점.  
- 본 가이드(제2장) = **벤더(수주자)가 그 요구에 회답하는 제안서를 작성**하는 관점 — **관점이 정반대**임.  
- 겹치는 주제(견적·스케줄·운용·특기사항)는 **"제안서에 어떻게 담는가"로만** 다루고, 요구 정의·산정  
  근거·규제 상세는 rfp-guide 해당 섹션 참조로 넘겨 **내용 중복을 회피**함(상호참조 지점은 4장 참조).

### 1-3. 3단 구조 계승

- rfp-guide와 **동일한 3단 구조**를 모든 Level1 섹션에 그대로 계승함.  
  - (a) 작성 원칙: 제2장 원문 근거로 "무엇을 왜 쓰는가"를 규정함  
  - (b) 통신사 B2B 예시·문구: 통신사 제안서에 그대로 넣을 수 있는 각색 예시·문장을 제시함  
  - (c) 작성 체크리스트: 누락 방지용 자가점검 항목을 제시함

### 1-4. 섹션별 팀원 오너십(위임 기준)

- 총괄·시나리오·발표: 스토리텔러 / 시스템·네트워크: 아키 / 운용·SLA: 케어·딜 / PoC·데모: 데모 /  
  견적·계약: 딜 / RFP 회답·추적성: 스펙 / RFP 분석: 리서처.  
- 섹션 표(2장)의 "오너" 열에 1차 담당을 명시함.

### 2-2. 섹션 상세

**## 1. 좋은 제안서의 조건과 구성요소** (근거 p58~66, 오너: 스토리텔러)  
- 제안활동 2단계(RFP 이전 1차 제안=범용 / RFP 이후 2차 제안=고객 커스터마이즈, "진짜 승부") 구분과,  
  좋은 제안서 4요소(RFP 요구 충족·부가가치·매력 가격·알기쉬움) 및 4조건(빠짐없이·중복없이·적게·  
  알기쉽게=MECE)을 규정함 (p59~65).  
- **Yes/No 명확 회답 원칙**(MECE 4조건과 병기): 충족 못하는 요구도 "No"로 명확히 회답하고 전제조건·  
  대안을 명시함. 애매·회피는 평가 배점표 감점에 직결되므로 예스·노우를 확실히 함 (p63~64·p71).  
- **2독자(관통) 원칙**: 제안서 본체 = 경영자·의사결정자용(적게·알기쉽게) / 별지 = 정보시스템부서용  
  (상세·기술 심화). 제2장 전체를 관통하는 사상이며, §3·§5·§6의 본체/별지 분리가 이 원칙의 적용임  
  (p64·p73·p80).  
- 제안서 필요충분 구성요소 = 일정수식어 + 8개 테마 + 특기사항 + 별지 골격을 개관함 (p65~66).

**## 2. 일정수식어와 1부 — 표지·인사문·목차·제안취지·제안대상영역** (근거 p67~72, 오너: 스토리텔러·스펙)  
- 성격이 다른 세 요소를 하위로 분리해 다룸(일정수식어=매너 / 제안취지=최중요 / 대상영역=범위 확정).  
  - (2-a) 일정수식어 = 표지 5요소·인사문·목차 규칙. 내용 가치는 낮으나 비즈니스 매너로 필수인 요소임 (p67~69).  
  - (2-b) 테마① 제안의 취지 = **제안서에서 가장 중요한 페이지**(엘리베이터 테스트). "제안의 특징 + RFP에  
    대한 회답" 2요소로 구성하며, 본 가이드에서 **별도 가중을 두어 강조**함 (p69~71).  
  - (2-c) 테마② 제안대상영역 = 대상 범위 정의·확인(수비). 문장 개조식 + 표로 인식 차이 발생을 억제함 (p71~72).

**## 3. 시스템 구성 — 개념도·네트워크·하드웨어·소프트웨어** (근거 p72~74, 오너: 아키)  
- 테마③. 시스템개념도(경영자도 이해 가능, 가장 중요)·네트워크구성도·하드웨어구성도·소프트웨어구성도  
  4종 도해의 작성 원칙과, HW/SW 상세는 별지로 돌리는 기준을 다룸(§1 2독자 원칙의 적용) (p72~74).  
- **통신사 B2B 네트워크 구성도(이중화·우회경로·PoP)를 담는 핵심 섹션**임. 이중화·PoP 등 기술 요구  
  정의 자체는 rfp-guide 4 참조, 본 가이드는 도해 표현만 다룸(상세는 4-2 참조).

**## 4. 프로젝트의 내용과 운용체제 — 수행방안·운용지원·SLA·연수** (근거 p74~76, 오너: 케어·딜·데모)  
- 테마④ 프로젝트의 내용(작업 플로우·유저/벤더 역할·성과물)과 테마⑤ 운용체제(운용지원 서비스 옵션·  
  엔드유저/관리자 연수)를 다룸 (p74~76).  
- **통신사 B2B SLA 제안(서비스 등급·NOC 24/365·하우징·원격감시)과 PoC/파일럿 검증 단계를 담는  
  핵심 섹션**임.

**## 5. 프로젝트 체제·스케줄·견적** (근거 p76~79, 오너: 딜·오케스트레이터)  
- 3부(테마⑥⑦⑧)를 하위 앵커로 분리함 — rfp-guide 예산(6)·일정(7) 단독 섹션과 병렬성을 확보함.  
  - (5-a) 테마⑥ 프로젝트 체제: 유저+벤더 대응 조직도·오너/PM/PL/사무국·보고라인 (p76~77)  
  - (5-b) 테마⑦ 스케줄: 마스터스케줄(개발+도입+진척회의), 일람성 우선·정밀 간트는 별지 (p77~78)  
  - (5-c) 테마⑧ 견적: "시스템 一式" 금지·항목별 상세·이니셜/러닝 구분·의사결정자용 요약 vs  
    정보시스템부서용 별지(§1 2독자 원칙의 적용) (p78~79)

**## 6. 특기사항과 별지** (근거 p79~81, 오너: 딜·스펙)  
- 목적이 다른 두 파트를 하위로 분리해 다룸(특기사항=계약 리스크 예방 / 별지=기술 심화 자료).  
  - (6-a) 특기사항 = 전제조건("거절")과 요망("부탁")으로 리스크를 상호 공유하고 사양변경·계약 분쟁을  
    예방하는 파트임 (p79~80).  
  - (6-b) 별지 = RFP와의 대조표·상세견적·복수안 비교·HW/SW 상세스펙·플로우차트·화면/장표 샘플·상세  
    스케줄·PoC/파일럿 검증 결과 자료·회사개요·실적 등 정보시스템부서용 기술 심화 자료임(§1 2독자  
    원칙의 적용) (p80~81).

**## 7. 제안 시나리오와 소구력 — 3부구성·결론선행·트리구조** (근거 p82~89, 오너: 스토리텔러)  
- 좋은 시나리오 3요소(알기쉬움·연속성·RFP 충족)와 이를 실현하는 3방법(3부구성 배치·결론선행형 서술·  
  트리구조로 본체/별지 구분)을 다룸 (p82~88).  
- 소구력(호소력) = 知(논리성)·情(열의)·利(투자 대 효과). RFP에 정확히 회답하는 것이 유저의 '정'에  
  호소하는 최선의 수단임 (p88~89).  
- **통신사 B2B: PoC/파일럿 실증 결과를 「利」(투자 대 효과)의 최강 증거로 배치**해 소구력을 강화함  
  (검증 결과 원자료는 §6 별지, 데모 시연은 §9와 연계).

**## 8. 제안서 작성 프로세스와 리뷰** (근거 p90~99, 오너: 리서처·스토리텔러)  
- RFP 수령 전 전초전(영업·SE 정보수집·사내 조정·우수 멤버 확보), 설명회 참가, 제안서 작성팀 편성  
  (PM 후보를 작성 멤버에 포함), 작성 4단계(RFP 취독·분석 → 제안취지 확립 → 시나리오별 검토 →  
  하나로 완성), 용어집 활용, 제3자 리뷰(최소 1회)·제출을 다룸 (p90~99).

**## 9. 프레젠테이션** (근거 p100~107, 오너: 스토리텔러·데모)  
- 참가 멤버 4~5명·역할분담(고급매니저·PM후보·영업·SE·회의록), 발표자료(제안서≠발표자료: 요점·개조식·  
  모식도), 리허설 6점 체크, 실전 3유의점(읽지 말고 말하기·결론선행·질의응답 마지막)을 다룸 (p100~107).  
- **통신사 B2B PoC 데모 시연 연계**를 발표 시나리오에 포함함.

**## 10. 제안서 샘플·워크시트 활용** (근거 p117~124, 오너: 스펙)  
- 표지 → 인사 → 목차 → ①취지 → ②대상영역 → ③시스템구성(4도해) → ④프로젝트내용 → ⑤운용체제 →  
  ⑥체제 → ⑦스케줄 → ⑧견적 → ⑨특기사항 → 별지 순 조립법을 다룸 (p117~124).  
- 통신 B2B 추가 칸(네트워크 구성도·SLA 등급표·커버리지)과 RTM 회답 추적(RFP↔제안서↔인수기준)을  
  반영하며, 문서 말미에 "제안서 작성 종합 체크리스트"를 배치함.

## 3. 각 섹션 3단 구조 적용 방침

- 모든 섹션(§1~§10)은 (a)작성 원칙 / (b)통신사 B2B 예시·문구 / (c)작성 체크리스트 3단으로 전개함  
  (rfp-guide와 동일 포맷·동일 순서 유지).  
- (a) 작성 원칙: 제2장 원문 근거로 "무엇을 왜 쓰는가"를 규정하고 모든 진술에 (p.XX~XX)를 표기함.  
- (b) 통신사 B2B 예시·문구: 병렬 수집물(외부자료)은 **페이지 근거가 아닌 출처 URL을 필수 표기**하며,  
  이론에 없는 URL은 지어내지 않음. 네트워크 구성도·SLA 제안·PoC 연계 등 통신 특성을 각 섹션에 배분함.  
- (c) 작성 체크리스트: 누락 방지용 자가점검 항목을 제시하고, 문서 말미에 §10의 종합 체크리스트로 집약함.  
- 섹션별 (b) 통신 특성 배분(확보 지점)  
  - 네트워크 구성도 → §3 / SLA 제안 → §4 /  
    PoC 연계 → §4(검증 단계)·§6 별지(검증 결과 자료)·§7 소구력 「利」 근거·§9(데모 시연) /  
    NRC·MRC·이니셜/러닝 견적 표현 → §5 / 규제 준수·전제조건 → §6 / RFP 회답 대조표 추적 → §2·§6·§10.

## 4. rfp-guide 대비 차별점·상호참조 지점

### 4-1. 관점 차별점 요약

| 주제 | rfp-guide(발주자·제1장) | 본 가이드(벤더·제2장) |
|---|---|---|
| 취지 | 발주자가 WHY(경영전략·문제·기대효과) 서술 | 벤더가 RFP 취지에 "제안 특징 + RFP 회답"으로 응답 |
| 시스템/기술 | 기술 요구(4대 분류) 정의 | 시스템구성 4도해로 실현안 제시 |
| 운용·SLA | 운용 요구·SLA 지표·위약 정의 | 운용지원 서비스·SLA 등급을 제안·견적 |
| 예산·견적 | 예산 공개·TCO·NRC/MRC 요구 | 항목별 견적 제시(一式 금지·이니셜/러닝) |
| 일정 | 컷오버 역산·리드타임 요구 | 마스터스케줄 제시(일람성·간트 별지) |
| 특기사항 | 발주자 제출요령·계약 조항 | 벤더 전제조건·요망·리스크 공유 |
