# RFP 주제 아이디에이션 가이드

> 용도: MVNO 사업자(발주자)가 통신사 MNO(수주 후보)에 발주할 **교육용 가상 RFP**의 주제 1건을  
> 선정하기 위한 팀원 공통 발상 가이드임. 모든 팀원은 본 가이드의 **필드 스키마**를 공통 서식으로  
> 사용하여 각자 2건씩 주제를 작성함.  
> 작성 주체: 리서처(RFP 분석가) / 조율: 오케스트레이터(클로니) / 초안 리뷰: 스토리텔러

---

## 1. 목적

- 통신사 B2B 기획자를 위한 **Claude Code 활용 SRS 작성 교육**의 실습 예제로 사용할 가상 RFP의  
  기반 주제 1건을 선정함  
- 선정된 주제는 후속 작업(`prompts/02-2`)에서 가상 RFP 본문으로 확장됨

---

## 2. 발주 구조 (⚠️ 구조 역전 주의)

- **발주자 = MVNO 사업자**: 자체 망이 없이 MNO의 망을 **도매로 빌려** 알뜰폰 서비스를 제공하는 사업자  
- **수주 후보 = MNO(통신사)**: 망을 보유하고 MVNO에 도매 제공하는 이동통신사  
- 일반 SI RFP(발주=일반기업, 수주=SI벤더)와 **주체가 다름**. 따라서 모든 주제는 MNO가 MVNO에게  
  제공해야 하는 **도매·망임대 맥락의 고유 WHY**에 근거해야 함 — 일반 사내 시스템 구축 주제는 부적합함

---

## 3. MVNO 실수요(WHY) — 주제의 근거 풀

> 모든 주제는 아래 실수요 중 하나 이상에 반드시 근거함. 근거 없는 주제는 후보에서 제외함.

| WHY-ID | 실수요(Pain Point) | 배경 |
|--------|-------------------|------|
| W1 | **망 도매 정산 자동화** | 도매대가 정산이 수작업·익일 배치라 대사 오류·정산 분쟁이 잦음 |
| W2 | **MVNO 전용 BSS/OSS 통제권** | 가입·개통·청구를 MNO 시스템에 의존해 자체 상품 출시·운영 자율성이 부족함 |
| W3 | **실시간 프로비저닝 API** | 유심 개통·번호이동·부가서비스 신청이 수동·배치라 개통 리드타임이 길고 실패 대응이 지연됨 |
| W4 | **실시간 사용량 조회** | 데이터 사용량이 익일 반영되어 고객에게 실시간 잔여량·요금을 안내하지 못함 |
| W5 | **도매 용량·트래픽 가시성** | 도매 대역·QoS·트래픽 현황을 실시간으로 볼 수 없어 용량 협의·품질 대응이 사후적임 |

- 팀원은 위 WHY를 조합·확장해도 되나, **MVNO 관점의 실수요**임을 유지해야 함

---

## 4. 필드 스키마 (공통 서식) ★필수

각 주제는 아래 6개 필드를 모두 채움. 업무·기술·운용 요구는 **각 3개 이상** 항목으로 기술함.

```
### [후보명 또는 주제 일련번호] 주제명
- **발주 배경(WHY)**: 어떤 MVNO 실수요(W1~W5)에 근거하는지 + 현재 문제 장면(As-Is)
- **업무 요구** (3개 이상): 무엇을 하고 싶은가 — 업무 기능·플로우 관점
- **기술 요구** (3개 이상): 어떻게 실현하는가 — 인프라·성능·연동·보안 관점 (가능하면 정량)
- **운용 요구** (3개 이상): 어떻게 운영·유지하는가 — SLA·보수·이행 관점 (가능하면 정량)
- **기대효과**: "현재값 → 목표값" 정량 대비 형식 권장
```

### 4-1. 업무 요구 작성 기준 (RFP작성가이드 3장 준용)

- 통신 도메인 골격: **OSS**(망 성능·인벤토리·장애·프로비저닝) / **BSS**(상품·주문·청구·고객)  
- 각 요구는 측정 가능한 **확인 기준(PoC 판정 기준)**을 붙임  
  - 나쁨: "조회가 빨라야 함" → 좋음: "가입자 실시간 사용량 조회가 요청 후 3초 이내 응답됨"

### 4-2. 기술 요구 작성 기준 (RFP작성가이드 4장 준용)

- 4대 분류 망라: **IT 인프라 / 시스템 능력(성능·용량·확장성) / 장해 대책 / 보안**  
- NFR은 **[지표 + 목표값 + 측정조건]** 3요소로 정량 기술  
  - 예: 온라인 조회 500 TPS 이상 / 피크 시간대 / 응답 2초 이내

### 4-3. 운용 요구 작성 기준 (RFP작성가이드 5장 준용)

- 5분류 검토: **운용 / 보수 / 교육·연수 / 이행(전환) / 아웃소스**  
- SLA 정량 지표: 가용률(99.9%/99.95%), 응답·복구시간(MTTR), 장애 통지 시간, 통신 특유(패킷손실·지연)  
- 도매 맥락 특화 지표(정산 정합률·API 성공률 등) 권장

| 팀원 | 관점 | 발상 초점 |
|------|------|----------|

## 6. 작성 규칙

- 언어: 한국어, **명사체 종결**(~함/~임)  
- 형식: 마크다운, 본 가이드의 필드 스키마를 **공통 서식**으로 사용  
- 각 팀원은 **2건**씩 작성, 출력파일: `work/rfp-idea/RFP주제-{팀원}.md`  
- 범위: 교육 실습에 적합하도록 **단일 도메인**으로 한정(과대 범위 금지), 요구 복잡도는 SRS 실습에 충분하게  
- 한 줄 120자 이내, 빈 줄 없이 줄바꿈 시 줄 끝 스페이스 2개

---

## 7. 필드 스키마 샘플 (참고용 1건)

### [샘플] MVNO 전용 실시간 데이터 사용량 조회 API 구축

- **발주 배경(WHY)**: W4(실시간 사용량 조회). 도매 정산이 익일 배치로만 제공되어 MVNO가 고객에게  
  실시간 잔여 데이터를 안내하지 못하고, 요금 초과 문의가 콜센터에 집중됨(As-Is)
- **업무 요구**
  - 가입자별 실시간 데이터·음성·SMS 사용량 조회 (확인: 요청 후 3초 이내 응답)
  - 요금제별·기간별 사용량 집계 및 대사 (확인: 표본 100건 집계가 검증표와 100% 일치)
  - 정산 내역과 사용량 원장 대사 리포트 제공 (확인: 일 단위 대사 불일치 건 자동 리포트)
- **기술 요구**
  - 통신사 과금 시스템 연동 REST API (지표: 조회 성공률 99.9% / 정상부하 / 월 측정)
  - 조회 성능 500 TPS 이상 (측정조건: 피크 시간대, 응답 2초 이내)
  - OAuth 2.0 인증·전송구간 암호화(TLS 1.2 이상), 접근권한 3단계(관리자·운영자·조회자)
- **운용 요구**
  - 가용성 99.9% 이상 (연 다운타임 약 8시간 46분 이내)
  - 장애 통지 30분 이내, MTTR 4시간 이내
  - 월간 정산·사용량 리포트 및 API 호출 통계 대시보드 제공
- **기대효과**: 실시간 안내로 요금 초과 문의 30% 감소(현재 월 1,000건 → 700건), 정산 분쟁 건수  
  월 20건 → 5건으로 축소
