## 0. 인사문

- 귀사의 무궁한 발전을 기원함. 당사 OO모바일(이하 "발주사")은 통신사(MNO) 망을 도매로 임차하여 알뜰폰
  서비스를 제공하는 MVNO 사업자로서, 망 장애·보안사고 발생 시 가입자를 신속·정확하게 보호하기 위한
  「통신 장애·보안사고 실시간 통지 및 영향범위 조회 시스템」 구축을 추진함.  
- 본 제안요청서(RFP)는 위 시스템 구축을 위한 요구사항·예산·일정·계약 조건을 정리하여 제안을 요청하는 문서임.
  귀사의 적극적인 제안을 요청함.  
- 명의: 발주사 최고정보책임자(CIO) / 사업총괄 부문장  
- 발행일: (가상) 2026-07-15

## 1. 사업 개요·추진 배경(취지)

### 1-1. 키 메시지 (WHY First)

- **"당사는 실시간 장애·사고 가시성을 확보하여, MNO 망 사고가 발생해도 가입자를 가장 먼저 안심시키는 알뜰폰
  사업자가 되고자 함."**

### 1-2. 발주사 개요·업계 동향

- 발주사는 MNO 망을 도매 임차하여 이동통신 재판매(알뜰폰) 서비스를 제공하는 MVNO임. 망 자산을 직접
  보유하지 않으므로 망 장애·보안사고에 대한 1차 정보는 MNO에 종속됨.  
- 업계 동향: 대규모 망 장애·개인정보 보안사고는 알뜰폰 사업자의 고객 신뢰·해지율에 직접적 영향을 미치며,
  통지 지연은 곧 고객센터 마비와 가입자 이탈로 연결됨.

### 1-3. 현재 문제점 (As-Is) — 장면화

- MNO의 대규모 망 장애·보안사고 발생 시, 발주사(MVNO)는 공식 통지를 실시간으로 받지 못하고 뉴스·SNS·
  자체 모니터링을 통해 뒤늦게 인지하는 구조임.  
- 2025년 4월 MNO 유심(USIM) 정보 유출 사고 당시, 해당 MNO 망을 사용하는 알뜰폰 사업자 고객센터가 문의
  폭주로 마비됨.  
- 2018년 MNO 국사(지사) 화재 당시, 알뜰폰 가입자 대상 보상 안내가 사고 발생 약 3주 후에야 개별 공지됨.  
- 결과적으로 "장애·사고 인지 지연 → 고객센터 응대 마비 → 가입자 이탈"의 악순환이 반복됨.

### 1-4. 정량 목표·기대효과 (현재값 → 목표값)

| 지표 | 현재값(As-Is) | 목표값(To-Be) |
|------|---------------|----------------|
| 장애 인지 ~ MVNO 통지 소요시간(평균) | 약 40분 | 5분 이내 |
| 장애 발생 시 고객센터 문의량(월) | 약 5,000건 | 3,500건(30% 감소) |
| 영향범위(가입자·지역) 파악 방식 | 수작업·사후 추정 | 실시간 조회(3초 이내) |

- 키 메시지와 기대효과가 수미상관을 이룸: "실시간 가시성 확보(WHY) → 통지 40분→5분·문의 30% 감소(효과)".

## 2. 요구사항

> 요구 분류: 업무(REQ-BIZ)·기술(REQ-TEC)·운용(REQ-OPS). 각 요구는 요구ID로 부록 A(RTM)와 연결됨.  
> 우선순위: 필수(M, Mandatory) / 권장(R, Recommended) / 선택(O, Optional).  
> 모든 요구에 측정 가능한 확인 기준(PoC 판정 기준)을 부여함.

### 2-1. 업무 요구 (무엇을)

| 요구ID | 우선순위 | 요구 내용 | 확인 기준(측정 가능) |
|--------|:---:|-----------|----------------------|
| REQ-BIZ-01 | M | 장애·보안사고 발생 즉시 **심각도별 자동 통지** — MNO 이벤트를 심각도(긴급/높음/보통)로 구분하여 MVNO 담당자에게 자동 통지 | 사고 인지 후 **5분 이내** MVNO 담당자 통지 도달(운영 프로세스 포함 종단 기준) |
| REQ-BIZ-02 | M | **영향 가입자 규모·지역·서비스 범위 실시간 조회** — 특정 사고의 영향 대상을 조회 | 임의 조회 1건 요청 후 **3초 이내(P99)** 응답, 표본 100건 대사 시 **100% 일치** |
| REQ-BIZ-03 | M | **복구 진행률·예상 복구시각(ETR) 갱신 제공** — 사고 진행 상황을 주기적 갱신 | **15분 주기** 갱신, 복구 완료 시점까지 이력 **연속 조회** 가능 |
| REQ-BIZ-04 | M | **사고 종료 후 장애 이력·재발방지대책 리포트 자동 생성** | 사고 종료 후 **3영업일 이내** 리포트 제공(통지 타임라인·영향범위·MTTR 포함) |
| REQ-BIZ-05 | R | **AI 기반 영향범위 예측 및 고객센터 응대 지원** — ① 사고 이벤트 입력 시 과거 유사 사고 패턴 기반으로 영향 가입자 규모·확산 추이를 예측, ② 예상 문의 FAQ·응대 스크립트 초안을 자동 생성하여 고객센터에 제공 | ① 예측 영향 가입자 수가 실제 확정치 대비 **오차율 ±15% 이내**(과거 사고 5건 백테스트), ② 사고 통지 후 **10분 이내** 응대 스크립트 초안 생성, 상담사 검수 후 사용 |

- 엔드유저 상충 식별: 사내 운영자(고객센터·상황실)는 "정확·신속 통지와 응대 부담 경감"을, 경영진은 "가입자
  이탈 방지·신뢰 유지"를 원함. 상충 시 **가입자 보호(통지 신속성)**를 최우선 순위로 함.
- AI 요구(REQ-BIZ-05)는 기대효과(문의량 30% 감소)에 직접 기여하도록 응대 지원에 초점을 둠 —
  예측·생성 결과는 **참고용**이며 최종 통지·안내 문구는 담당자 검수(HITL)를 거침.

### 2-2. 기술 요구 (어떻게 — NFR 정량화)

> 기술 요구는 4대 분류(IT 인프라·시스템 능력·장해 대책·보안)로 망라함.  
> 모든 NFR은 [지표 + 목표값 + 측정조건] 3요소로 기술함.

| 요구ID | 우선순위 | 요구 내용 | 지표 + 목표값 + 측정조건 |
|--------|:---:|-----------|--------------------------|
| REQ-TEC-01 | M | **Webhook/API 기반 실시간 이벤트 푸시** — MNO 장애·사고 이벤트를 MVNO로 실시간 전달 | 지표: 발생~통지 지연 / 목표: **1분 이내**, 전송 성공률 **99.9% 이상** / 측정: 월 단위, 정상·피크 포함 |
| REQ-TEC-02 | M | **대규모 동시 조회 처리** — 장애 피크 시 영향범위 조회 폭주 대응 | 지표: 처리량·응답시간 / 목표: **300 TPS 이상**, 평균 응답 **2초 이내** / 측정: 장애 피크 시간대 부하시험 |
| REQ-TEC-03 | M | **다중 채널 통지 이중화** — API + 이메일 + SMS 다중화, 단일 채널 장애 시 자동 전환 | 지표: 채널 자동 전환시간 / 목표: **30초 이내** / 측정: 임의 채널 강제 차단 후 전환 시험 |
| REQ-TEC-04 | M | **접근 인증·전송 보안·위변조 방지** — 인증·암호화·무결성 로그 | 지표: 인증/암호화 표준 / 목표: **OAuth 2.0** 인증, 전송구간 **TLS 1.2 이상**, 통지 이력 **위변조 방지 로그**(무결성 검증) / 측정: 보안 점검·로그 무결성 검증 |
| REQ-TEC-05 | R | **AI 예측·생성 모델 서비스화** — REQ-BIZ-05의 예측·생성 기능을 API로 제공, 신규 사고 데이터로 주기적 재학습 | 지표: 추론 응답시간·재학습 주기 / 목표: 예측 추론 **3초 이내** 응답, **월 1회 이상** 재학습·성능 리포트 / 측정: 추론 부하시험, 재학습 로그 |

**시스템 능력(Capacity) 수치표**

| 구분 | 지표 | 목표값 | 측정조건 |
|------|------|--------|----------|
| 성능 | 이벤트 통지 지연 | 1분 이내 | 발생~통지 종단, 월 측정 |
| 성능 | 영향범위 조회 응답 | 평균 2초 / 최대(P99) 3초 | 피크 300 TPS 부하 시 |
| 용량 | 동시 조회 처리량 | 300 TPS 이상 | 장애 피크 시간대 |
| 용량 | 통지 대상 담당자 계정 | 500 계정 이상 | 개통 1년차 기준 |
| 용량 | 통지 이력 보관 | 3년 이상 | 위변조 방지 로그 포함 |
| 확장성 | 무중단 Scale-out | 지원 | 노드 증설 시 서비스 무중단 |

**장해 대책(가용성 → 이중화 → 배상 3단 연계)**

- 가용성 목표: 통지 시스템 자체 가용성 **99.95% 이상/월**(연 다운타임 약 4시간 22분 이내). → 상세 SLA는
  2-3(운용)·6(계약)과 수치 일치.  
- 이중화: 통지 채널(API/이메일/SMS) 다중화(REQ-TEC-03), 시스템 무중단 Scale-out(Capacity표).
  구체적 이중화 아키텍처(Active-Active/Standby, DR)는 **벤더 제안에 위임**하되 위 가용성 목표 달성을 전제로 함.  
- 배상: 가용성·통지 SLA 미달 시 서비스 크레딧(→ 6. 계약조건).

**보안·규제 준수**

- 접근권한 3단계(관리자 / 운영자 / 조회자) 분리(REQ-TEC-04 연계).  
- 개인정보보호법 안전성 확보조치 기준 준수(접근권한 최소화·전송 암호화·접근기록 보관). 영향범위 조회 시
  **가입자 식별정보는 집계·범위 단위로 제공**하고 개인 식별 최소화.  
- 규제 항목(개인정보 암호화·접근통제·망 연동 보안)은 **상세 지정**, 세부 아키텍처·튜닝 수단은 벤더 위임으로 배분.

### 2-3. 운용 요구 (얼마나 잘 — SLA)

> 운용 요구 5분류(운용·보수·교육·이행·아웃소스)를 검토하여 본 사업 범위에 해당하는 항목을 명시함.

| 요구ID | 우선순위 | 요구 내용 | 확인 기준(SLA 정량) |
|--------|:---:|-----------|---------------------|
| REQ-OPS-01 | M | **통지 시스템 가용성 보장** | 시스템 가용성 **99.95% 이상/월**(연 다운타임 약 4시간 22분 이내), 월간 가용성 리포트 제공 |
| REQ-OPS-02 | M | **장애 통지 SLA 준수율 관리** — 미준수 시 페널티 | 통지 SLA(발생~통지 1분·인지~담당자 5분) **준수율 월 98% 이상**, 미준수 시 서비스 크레딧 적용(→6장) |
| REQ-OPS-03 | M | **24x365 공동 대응 체계·합동 훈련** | 24x365 공동 대응 **핫라인 운영**, **분기 1회** 합동 장애대응 모의훈련 실시(결과 리포트 제출) |
| REQ-OPS-04 | M | **월간 장애대응 리포트 제공** | 매월 통지 소요시간·MTTR·재발률 포함 리포트 제공, 정기 리뷰 미팅 |

**SLA 지표표 (운용·계약 공통 기준선)**

| SLA 지표 | 목표값 | 측정 방식 | 미달 시 |
|----------|--------|-----------|---------|
| 통지 시스템 가용성 | 99.95%/월 | Availability = MTBF/(MTBF+MTTR), 월 집계 | 서비스 크레딧(6장) |
| 발생~통지 지연 | 1분 이내 | 이벤트 타임스탬프 자동 측정 | 준수율에 반영 |
| 통지 SLA 준수율 | 98% 이상/월 | (준수 건 / 전체 통지 건) | 서비스 크레딧(6장) |
| 장애등급별 MTTR | 긴급 2h / 높음 4h / 보통 8h 이내 | 접수~복구 자동 측정 | 준수율에 반영 |
| 채널 자동 전환시간 | 30초 이내 | 전환 이벤트 로그 | 준수율에 반영 |

- 서비스 시간: 통지·조회는 24x365 기준(장애·사고는 시간대 무관 발생). 정기점검(PM)은 사전 통지 후
  **트래픽 저부하 시간대(심야)**에 수행하여 오버스펙을 방지함.  
- 교육·연수: 고객센터 상담사·상황실 운영자·관리자 대상 시스템 사용 교육 포함(계상처는 예산 내, 3장).  
- 이행(전환): 운영 런북 작성, 3단계 지식 이전(Shadow→분담→독립), 안정화(hypercare) 1개월 SLA 적용,
  전환 실패 대비 롤백 계획 포함.

## 3. 사업비·대가 산정 (얼마에)

> 본 예산은 교육용 가상 예제 수치이며, 제안 시 항목별 상세 견적 제출을 요구함.  
> 공개 전략: **총액 상한(band) 제시 + 항목별 명세 제출** 방식.

- **비용 구분(초기 NRC / 계속 MRC 분리 제출 필수)**

| 구분 | 항목 | 가상 예산 상한(band) |
|------|------|----------------------|
| 초기(NRC) | 설계·개발·MNO 연동·AI 모델 구축·데이터 이행·테스트 | **8억원 이내** |
| 계속(MRC) | 회선·인프라 이용료, 유지관리, 헬프데스크, AI 모델 재학습·운영 | **월 3천만원 이내** |
| TCO | 5년 총소유비용(NRC + MRC×60개월 + DR·이중화) | **25억원 이내** |

- **별도 예산 항목 계상처 명시**: 데이터 이행·병행운영(예산 내), 채널 이중화·DR 사이트(예산 내),
  AI 모델 재학습·성능관리(MRC 내), 하자보수 종료 후 유상 유지관리(별도), 변경요청(CR) 맨먼스 단가(별도 단가표).  
- **유지관리 요율**: 요율 상한 또는 산정 근거(SW사업 대가산정 가이드 준거)를 제출 요구.  
- **실비정산 vs 정액 구분**: 출장·교통비 등은 실비정산, 유지관리·운영은 정액으로 구분 요구.  
- TCO 관점: HW·SW 획득비만이 아니라 운영·유지보수·다운타임 비용을 포함한 5년 생애주기 총비용으로 비교함.

## 6. 계약 조건 (SLA·NDA·저작권)

### 6-1. SLA·서비스 크레딧

- SLA 지표는 2-3(운용) SLA 지표표를 계약 기준선으로 준용함(수치 일치).  
- **서비스 크레딧 산정식(예시·인센티브 균형)**: 월 통지 SLA 준수율 기준 —

| 월 SLA 준수율 | 서비스 크레딧(MRC 대비) |
|---------------|-------------------------|
| 98% 이상 | 없음(목표 달성) |
| 97% 이상 ~ 98% 미만 | MRC의 5% |
| 95% 이상 ~ 97% 미만 | MRC의 10% |
| 95% 미만 | MRC의 15% (상한 20%) |

- 크레딧은 징벌이 아닌 개선 유인으로 설계하며, 기대목표치/최소목표치를 이원화함.

### 6-2. 저작권

- **분리 귀속**: 신규 개발 산출물은 발주사 귀속, 벤더 기존 보유자산(선행 저작물·AI 사전학습 모델 등)은 벤더
  귀속 + 발주사에 사용권 부여.

### 6-3. 하자보수 vs 유지관리

- **하자보수(무상)**: 사업 완료일로부터 **1년**, "오류·결함 수정"에 한정.  
- **유지관리(유상)**: 추가개발·기능개선·AI 재학습·추가 교육·환경변경 등은 유상 유지관리로 구분.

### 6-4. 지체상금

- 요율: SW사업(물품+용역 일괄) 기준 **1000분의 1.5**. 산식 = 계약금액 × 요율 × 지체일수.  
- 면책: 발주사 귀책(요구 확정·검수 지연)·불가항력.

### 6-5. NDA·규제 준수

- **NDA 2회**: RFP 배포 시점(제안 참여 벤더 전원) + 계약 체결 시점. 비밀정보 범위·목적 외 사용금지·제3자
  공개금지·반환/파기·위반 시 손해배상.  
- **규제 준수 서약**: 개인정보보호법 안전성 확보조치, 망 연동 보안 준수 의무를 NDA와 함께 명시.
