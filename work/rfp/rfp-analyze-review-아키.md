## 1. 총평

- 초안의 대응가능여부 판정은 **전반적으로 기술적으로 타당**함. 가능 13·부분가능 3·불가 0 분포,  
  AI 3건 부분가능, 5분⊃1분 부분집합 인식, MNO 데이터 종속을 최대 외생 리스크로 지목한 점은 적절함.  
- 다만 Capacity표의 **DR RTO 1시간/RPO 15분이 어느 요구에서도 명시적으로 판정되지 않아** "누락 0건"  
  주장과 상충함(REQ-TEC-06 근거는 네트워크·DB·DNS 이중화만 언급).  
- 또한 SLA·크레딧이 걸린 REQ-TEC-01의 "가능" 근거가 페일오버 **감지 방식**을 규정하지 않아 1분 예산  
  성립 조건이 불완전하며, 조회 성능 판정 근거(사전집계·캐싱)가 실시간 정확성·라이브 사고 중 읽기/쓰기  
  경합을 다루지 못해 보강이 필요함.

## 3. 기술 리스크 보완

### R1. DR RTO/RPO 미판정·미설계 (→ 수정필수)

- Capacity표는 **DR RTO 1시간·RPO 15분**을 명시하나, REQ-TEC-06 근거는 "네트워크·DB·DNS 이중화"만  
  언급하고 A4 기술리스크에도 DR 항목이 없음. 가용성 99.95%(REQ-OPS-01)와 DR RTO/RPO는 **별개 지표**임  
  (전자는 상시 이중화, 후자는 사이트 장애 시 복구목표).  
- 보완: REQ-TEC-06 근거에 **RPO 15분 = 비동기 복제 허용**, **RTO 1시간 = 웜 스탠바이+복구 자동화**로  
  달성 방식과 대응가능여부 판정을 명시. DR 사이트·복제 비용은 NRC/MRC(3장 DR·이중화 항목)에 반영 확인.

### R2. 페일오버 감지 방식과 1분 예산 (→ 수정필수)

- REQ-TEC-01 "가능"은 SLA·서비스 크레딧이 걸린 필수 지표임. 전환 30초만으로는 부족하며, **장애 감지  
  타임아웃**이 수동(전송 실패 대기)이면 감지(수십초)+전환 30초+재전송 합이 **1분을 초과**함.  
- 보완: "가능" 판정의 조건을 **능동 헬스체크 기반 사전 페일오버**(수동 타임아웃 방식 배제)와 **채널 병렬/비동기  
  전송**으로 명문화. SRS 인수기준은 "페일오버 발생 케이스 포함 수신~통지 1분·성공률 99.9%"로 확정.

### R3. 조회 성능 vs 실시간 정확성·버스트 (→ 권장)

- REQ-BIZ-02/REQ-TEC-02 근거 "사전집계·캐싱"은 두 지점에서 상충 위험: ① 라이브 사고 중 영향범위는  
  실시간 변동(15분 ETR 갱신)하므로 **캐시 신선도**가 "표본 100건 100% 일치" 대사와 충돌 가능,  
  ② 사고는 **순간 폭증(버스트)**이라 반응형 오토스케일링 웜업 지연 시 초기 P99 스파이크 발생.  
- 보완: **CQRS/읽기전용 복제본 분리 + 이벤트기반 집계 갱신**으로 읽기/쓰기 경합 해소, 캐시 TTL·일관성  
  스냅샷 기준 정의, 피크 대비 **상시 헤드룸(사전 프로비저닝)** 확보를 근거에 추가.

### R4. Webhook 수신 신뢰성(멱등·순서) (→ 권장)

- REQ-TEC-01 성공률 99.9%·1분은 안정적 이벤트 수신이 전제임. RFP가 SLA 기점을 "수신 시각"으로 정의해  
  **MNO 전달 지연은 벤더 책임 밖**인 점은 유리하나, 수신 후 **이벤트 유실·중복·순서역전** 처리는 벤더 책임.  
- 보완: 수신 파이프라인의 **멱등성 키·중복 제거·순서 보장(또는 순서 무관 설계)**를 REQ-TEC-01 근거에 반영.

### R5. AI 데이터 희소성·인수기준 성격 (→ 권장)

- REQ-BIZ-05a 합격기준은 "가용 과거 사고 전수 백테스트 + **합격건수 리포트**"인 **프로세스형**이므로,  
  "부분가능"이 산출물 미달로 오해되지 않게 SRS 인수기준을 프로세스형으로 명시해야 함.  
- 대형 망사고는 연 수건으로 **표본이 본질적으로 희소**함 → REQ-TEC-05 "10건 누적 재학습" 트리거는 장기  
  미발동 가능 → **룰 폴백이 사실상 주경로**가 될 수 있음을 근거·SRS에 반영(과대약속 방지).

### R6. 개인정보 집계·범위 단위 제공 제약 (→ 권장)

- RFP 보안 섹션은 영향범위 조회 시 **가입자 식별정보를 집계·범위 단위로 제공**(개인 식별 최소화)을 요구함.  
  이는 REQ-BIZ-02 조회 설계의 제약이자 REQ-TEC-04 보안과 교차하는 항목이나 초안 근거에 미반영됨.  
- 보완: REQ-BIZ-02 근거에 "집계/범위 단위 응답(개인 미식별)" 제약을 명시(판정은 오히려 용이해짐).

## 4. 결론

- **수정 필수 2건**  
  1. **REQ-TEC-06 — DR RTO 1시간/RPO 15분 대응가능여부 판정 누락**. Capacity표 명시 NFR을 이중화와  
     구분하여 판정·근거(웜 스탠바이·비동기 복제) 추가. "누락 0건" 주장과의 상충 해소.  
  2. **REQ-TEC-01 — 페일오버 성립조건 미명문화**. SLA·크레딧 지표이므로 "가능"의 조건을 능동 헬스체크·  
     병렬전송으로 확정하고 SRS 인수기준에 페일오버 케이스 포함 명시.  
- **권장 4건**  
  1. REQ-BIZ-02/REQ-TEC-02 — 조회 성능 근거를 CQRS·캐시 신선도·버스트 헤드룸으로 보강(§3-R3).  
  2. REQ-TEC-01 — Webhook 수신 멱등·중복제거·순서 보장 리스크 명시(§3-R4).  
  3. REQ-BIZ-05a/REQ-TEC-05 — 인수기준 프로세스형 명확화·룰 폴백 주경로 가능성 반영(§3-R5).  
  4. REQ-BIZ-02/REQ-TEC-04 — 개인정보 집계·범위 단위 제공 제약 교차 반영(§3-R6).  
- **판정 타당 항목**: 위 지적 외 REQ-BIZ-01·03·04, REQ-TEC-03·04·05, REQ-OPS-01~04, AI 부분가능
