## A0. 분석 개요·방법·범위

- **목적**: RFP 요구 16건(REQ-BIZ 6·REQ-TEC 6·REQ-OPS 4)을 표면 요구와 실제 니즈로 구분하고,  
  우선순위·대응가능여부를 판정하여 SRS(05단계) 작성의 검증된 입력자료를 제공함.  
- **방법**: 각 요구를 RTM 5필드(요구ID·표면요구·실제니즈·우선순위+근거·대응가능여부+근거)로 판정함.  
  실제 니즈는 RFP 1장 취지(WHY: "MNO 사고에도 가입자를 가장 먼저 안심시키는 사업자")와 1-3 As-Is  
  문제·1-4 정량 목표(현재값→목표값)에서 역산함.  
- **범위**: RFP 2장(2-1 업무·2-2 기술·2-3 운용) 전 요구 + 취지 정합성. 요구ID가 없는 확인기준(2-2  
  Capacity표·DR·2-3 이행 요건)은 상위 REQ-ID에 귀속하여 분석하고 신규 ID를 만들지 않음. 예산(3장)·  
  일정(4장)·계약(6장)은 우선순위·리스크 판정 근거로 참조함.  
- **판정 척도**  
  - 우선순위: **필수**(취지 정합 높음·배점 큼·미이행 시 감점 확정) / **권장**(정합 있으나 대체수단  
    존재) / **선택**(제안 기대형).  
  - 대응가능여부: **가능**(현 역량 충족) / **부분가능**(조건부·추가 협의 필요, 조건 명시) /  
    **불가**(기술·정책상 수용 불가, 사유 명시).

## A4. 핵심 발견사항·리스크 요약

### 표면 요구 ↔ 실제 니즈 괴리 (핵심 발견사항)

1. **"알림 시스템" ↔ "가입자 선제 안심 + 응대 부담 경감 + 신뢰·해지율 방어"**: 표면은 장애 알림이나,  
   실제 니즈는 통지 정확성(조회)·응대지원(AI)·사후 신뢰(리포트)가 한 세트로 작동해야 충족됨. 단순  
   알림봇으로는 미충족 — SRS는 기능 간 인과 고리를 유지해야 함.  
2. **"5분" ⊃ "1분" 부분집합 관계**: 인지~담당자 5분(운영 포함 종단)과 시스템 전송 1분(REQ-TEC-01)은  
   상·하위 관계임. 혼동 시 SLA 정산 분쟁 소지 → SRS 인수 기준에서 두 지표를 명확히 분리 정의해야 함.  
3. **MNO 정보 종속성(최대 외생 리스크)**: 통지·ETR·영향범위의 원천 데이터가 MNO 제공에 의존함.  
   시스템 성능이 좋아도 MNO 데이터 적시성이 낮으면 목표 미달 → MNO 연동 규격·SLA 선행 협의 필수.  
4. **AI 요구는 참고용·HITL 필수·룰 폴백 병행**: REQ-BIZ-05a/05b·REQ-TEC-05는 과대약속 금지 대상.  
   데이터 확보량이 성능을 좌우하므로 부분가능 판정 — SRS에서 확률형·측정/리포트형 인수 기준 필요.  
5. **벤더 SLA 경계**: MNO 망 복구 MTTR은 관찰 지표로 벤더 페널티에서 제외됨. 벤더 책임은 통지·가용성·  
   ETR 갱신 성실성으로 한정 — SRS·계약에서 경계를 명문화해야 분쟁을 예방함.

### 우선순위 분포

- **필수 13건**: REQ-BIZ-01·02·03·04, REQ-TEC-01·02·03·04·06, REQ-OPS-01·02·03·04.  
- **권장 3건**: REQ-BIZ-05a·05b, REQ-TEC-05 (모두 AI 요구, 룰 폴백 대체수단 존재). **선택 0건**.

### 대응가능여부 분포

- **가능 13건 / 부분가능 3건(REQ-BIZ-05a·05b, REQ-TEC-05) / 불가 0건**.  
- REQ-TEC-01·06은 "가능(조건)"으로, 조건(페일오버 감지 방식·MNO 연동 리드타임)을 명시함.

### 기술 리스크 (아키)

- REQ-TEC-01: 1분 예산 대비 채널 전환 30초 → 능동 헬스체크·병렬 전송 설계 필수(수동 타임아웃 배제).  
- REQ-TEC-02: 300 TPS·P99 3초는 부하시험 확증 대상, 캐시 신선도와 표본 100% 대사의 양립 설계 필요.  
- AI 3건(BIZ-05a·05b, TEC-05): 과거 사고 데이터 확보량 의존 → 부분가능, 룰 폴백·HITL 전제.

### 일정 리스크 (아키)

- REQ-TEC-06: MNO 연동 보안검토·승인 리드타임 → 컷오버(2027-03-02) 역산 버퍼 필요, 연동 협의 선행.

### 계약 리스크 (딜)

- **가용성 크레딧 산정 경로 부재**: 6-1 크레딧 표 트리거가 "통지 SLA 준수율" 단일 지표뿐 → REQ-OPS-01  
  가용성 미달 시 크레딧 계산 불가. SRS·계약에서 가용성 크레딧 산정식을 별도 정의해야 분쟁 예방.  
- **AI 재학습 귀속(±15%) 판정 절차 공백**: drift vs 구현결함 판정 주체·입증책임·절차가 미정의(최대  
  분쟁 지점). SRS 인수기준에 판정 주체·증빙·기한을 명문화 필요.  
- **ETR 갱신 성실성 전가 리스크**: 원천 복구정보가 MNO 종속임에도 미갱신이 통지 준수율 크레딧에 전가될  
  소지 → MNO 미제공 기인 면책 조항 필요.  
- **정상 확인(이상 없음)**: 서비스 크레딧 연 누적 상한 15%, 지체상금 cap NRC 30%, MNO MTTR 벤더 SLA  
  배제 — 3항목은 계약 리스크 통제가 적절하게 설계됨.

### 가격(예산) 리스크 (딜)

- **MRC 월 2,500만원 상한 압박**: 24x365 운영 인력 + DR·이중화 운영 + AI 모델 재학습·운영을 동시 계상  
  해야 하므로 MRC 상한 내 수용성 검증 필요(초과 시 범위·SLA 조정 협의).  
- **AI 재학습 정액/실비 경계**: 재학습 트리거(신규 사고 누적 10건)가 사고 빈도에 종속 → 정액(운영) vs  
  실비(대량 재학습) 구분 기준을 SRS·견적에 명시.  
- **하자보수(무상 1년) ↔ 유지관리(MRC 유상) 중첩**: 오류·결함 수정(무상)과 기능개선·환경변경(유상)의  
  경계를 인수기준에 명문화하여 이중 청구·무상 요구 분쟁 예방.

### RFP 커버리지 대사

- RFP 원문 요구 16건(REQ-BIZ 6·REQ-TEC 6·REQ-OPS 4) = 본 분석 16건. **누락 0건**.  
- 요구ID 미부여 확인기준(2-2 Capacity·DR RTO/RPO·무중단 Scale-out·용량, 2-3 이행 요건)은 상위  
  REQ-ID(REQ-TEC-06·REQ-OPS)에 귀속 분석함 — 신규 ID 미생성, SRS S4·S7 입력으로 연계.
