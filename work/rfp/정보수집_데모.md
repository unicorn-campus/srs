## 섹션 3. 업무 요구 작성

### 3-1. 업무 요구의 성격과 추출 방법

[이론] 업무 요구는 취지에 따라가면서 추출함. 고객 만족도 향상 등 목표에 대해 "필요한 것, 현재 안 되어  
있는 것, 있지만 불완전한 것"을 알아냄. 처음에는 랜덤이어도 자꾸자꾸 밝혀낸 뒤 중복·정리하여 종합함  
(RFP작성이론 3-1, 원문 p.15).  
[이론] 업무 요구는 정보시스템 부서만으로 작성하기 곤란하므로 실제 업무 종사자인 엔드유저의 참가가  
불가결함(RFP작성이론 3-5, 원문 p.18).

[데모 지식] PoC 관점에서 업무 요구 추출은 "무엇을 데모로 먼저 보여줘야 이해관계자가 OK/NG를 판정할 수  
있는가"를 기준으로 우선순위를 매기는 작업임. 말로 합의된 요구는 반드시 "말했다/안 했다"로 흔들리므로,  
추출 즉시 "이 요구가 충족되었음을 어떻게 눈으로 확인할지(확인 방법)"를 한 줄이라도 붙여두는 습관이  
후속 요건정의·검수 단계의 분쟁을 크게 줄임.  
[데모 지식] 랜덤 추출 단계에서는 발산(brainstorming)을, 정리 단계에서는 수렴(중복 제거·유사 병합)을  
분리 진행함. 발산 중 검증가능성을 따지면 아이디어가 죽으므로, 검증가능성 검토는 수렴 이후로 미룸.

[외부자료] Elicitation Techniques in Business Analysis (PMWorld 360) - https://www.pmworld360.com/elicitation-techniques-in-business-analysis/  
[외부자료] Requirements Elicitation: A Survey of Techniques, Approaches, and Tools -  
https://eecs481.org/readings/requirements.pdf

### 3-2. 업무 요구 6계층 — 검증 단위로의 분해

[이론] 업무 요구는 6계층으로 구체화됨(RFP작성이론 3-2, 원문 p.15~16).  
① 업무 기본 요구: 근본적 업무 니즈(데이터 일원화 등). 기본 요구만으로도 업무 요구로 충분한 효용  
② 업무 기능 요구: 기본 요구를 실현하는 개별 기능(정보 입력·화면/장표 취출, 기본정보 사전 설정 등)  
③ 업무 플로우 요구: 업무 처리 순서·흐름(내점→검색→정보체크→접객 등)  
④ 데이터 요구: 데이터 항목수·형식·테이블 구조·문자수 제약 등  
⑤ 화면 요구(인터페이스 요구): 입력·조회 화면의 항목 배치·표시 형식  
⑥ 장표 요구: 출력 장표의 항목·용지 규격(A4 가로 등)

[데모 지식] 6계층은 PoC 관점에서 "검증 리드타임이 짧은 순 → 긴 순"으로 재배열 가능함. 기본·기능 요구는  
목업(mockup)·클릭더미로 하루 내 검증 가능하나, 플로우·데이터 요구는 실제 연동·이관이 필요해 리드타임이  
길어짐. RFP 단계에서 계층별로 "PoC 검증 난이도(상/중/하)"를 표시해 두면 벤더가 데모 계획을 현실적으로  
세울 수 있어 오버스펙 제안을 억제함.  
[데모 지식] 6계층을 아래처럼 "검증 산출물"에 매핑하면 요구의 testability가 한눈에 드러남.  
- 기본 요구 → 개념 검증(성립 가능성 O/X)  
- 기능 요구 → 기능 데모(단위 시나리오 통과 여부)  
- 플로우 요구 → 시나리오 데모(엔드투엔드 통과 여부)  
- 데이터 요구 → 데이터 이관 리허설(건수·정합률·오류율)  
- 화면 요구 → 화면 프로토타입(항목 누락·오배치 여부)  
- 장표 요구 → 샘플 장표 출력(항목·양식 일치 여부)

[이론] 상세도가 높을수록 개발 착수 후 요건정의 단계 작업을 줄일 수 있으나, "이 레벨까지가 RFP"라는 절대  
기준은 없고 기업·안건 사정에 따라 깊이가 다름(RFP작성이론 3-3, 원문 p.16~17).

### 3-3. 요구 표현 형식 — 비즈니스/기능/비기능 요구의 구분

[외부자료] 비즈니스 요구는 프로젝트의 "왜(why)·무엇을(what)"을, 기능 요구는 "어떻게(how)"를 설명하며,  
비기능 요구는 성능·보안·사용성·신뢰성 등 "얼마나 잘(how well)" 수행하는지를 규정함  
(Requiment / altexsoft).  
[외부자료] Functional Requirements vs Business Requirements Explained (Requiment) -  
https://www.requiment.com/functional-requirements-vs-business-requirements-explained/  
[외부자료] Functional and Non-Functional Requirements (altexsoft) -  
https://www.altexsoft.com/blog/functional-and-non-functional-requirements-specification-and-types/

[데모 지식] 6계층 중 ①기본 요구는 비즈니스 요구, ②~⑥은 기능 요구에 가깝고, 성능·용량·보안 등  
비기능 요구는 섹션 4(기술 요구)로 넘어감. 업무 요구 작성 시 "왜(취지 연결)"를 잃지 않도록 각 기능  
요구 앞에 원 비즈니스 요구를 명시하면, 벤더가 기능만 구현하고 목적을 놓치는 사고를 방지함.

[데모 지식] 개별 요구를 문장으로 적을 때 두 가지 표준 형식이 유용함.  
- 유저 스토리: "As a [역할], I want [행위], so that [목적]" — 경량·빠른 슬라이싱에 적합  
- 유스케이스: 액터·사전조건·주 흐름(번호 단계)·대안 흐름·사후조건 — 규제·대형 안건의 전체 흐름 문서화에  
  적합  
[외부자료] Difference Between User Stories & Use Cases (Avion) - https://www.avion.io/blog/use-case-vs-user-story/

### 3-4. 요구의 검증가능성(testability) 확보

[외부자료] 요구가 검증 가능(verifiable)하다는 것은 데모·분석·검사·테스트 실행 중 하나로 최종 산출물에  
구현되었음을 증명할 수 있음을 의미함(Prolifics Testing).  
[외부자료] 모든 인수 기준(acceptance criteria)은 객관적으로 테스트 가능해야 하며, "쉽다·빠르다·많다"  
같은 정성 표현은 측정·수량화하여 재작성해야 함. Given-When-Then 형식으로 "초기 상황·행위·기대 결과"를  
명확히 함(eitt.academy / boost.co.nz).  
[외부자료] Ten Attributes of a Testable Requirement (Prolifics) -  
https://www.prolifics-testing.com/news/ten-attributes-of-a-testable-requirement  
[외부자료] Testable requirements – how to write good acceptance criteria (EITT) -  
https://eitt.academy/management/testable-requirements-how-to-write-good-acceptance-criteria/  
[외부자료] INVEST criteria for Agile user stories (Boost) - https://www.boost.co.nz/blog/2021/10/invest-criteria/

[데모 지식] RFP의 업무 요구를 그대로 두면 검수 시 "충족했다/안 했다" 분쟁이 발생함. 각 요구에 측정  
가능한 확인 기준을 붙이면 PoC·검수 시 즉시 판정 가능함. 나쁜 예 → 좋은 예 변환 규칙:  
- 나쁨: "화면 조회가 빨라야 함" → 좋음: "고객 검색 결과가 조회 요청 후 3초 이내 표시됨"  
- 나쁨: "데이터가 정확히 이관되어야 함" → 좋음: "기존 회선 100만 건 이관 시 정합률 99.99% 이상,  
  불일치 건은 오류 리포트로 출력됨"  
- 나쁨: "청구서를 쉽게 확인함" → 좋음: "고객이 로그인 후 3클릭 이내 당월 청구 상세에 도달함"  
[데모 지식] INVEST 기준(Independent·Negotiable·Valuable·Estimatable·Small·Testable) 중 특히 Small과  
Testable이 PoC 검증가능성과 직결됨. 요구가 크면 데모로 쪼갤 수 없으므로, RFP 단계에서 큰 요구는 검증  
가능한 소단위로 분할해 두는 것이 유리함.

### 3-5. 통신사 B2B 업무 요구 예시 (도메인)

[외부자료] 통신사 시스템은 OSS(운영지원)와 BSS(비즈니스지원)로 구분됨. BSS는 상품 관리·주문 관리·  
수익(청구) 관리·고객 관리 4개 프로세스를, OSS는 망 성능·인벤토리·장애 관리·서비스 프로비저닝(개통)을  
담당함. 고객이 신규 요금제를 주문하면 주문 관리가 OSS의 서비스 프로비저닝·망 구성 작업을 트리거함  
(SRM Tech / Wikipedia OSS/BSS).  
[외부자료] The Role of OSS and BSS in Telecom Management (SRM Tech) -  
https://www.srmtech.com/knowledge-base/blogs/the-role-of-oss-and-bss-in-telecom-management/  
[외부자료] OSS/BSS (Wikipedia) - https://en.wikipedia.org/wiki/OSS/BSS

[데모 지식] 통신사 B2B RFP에서 자주 나타나는 업무 요구를 6계층 + 확인 기준으로 예시함.

- 고객관리(CRM)  
  - 기본 요구: 분산된 고객·계약 정보를 단일 뷰로 일원화  
  - 기능 요구: 고객·법인·회선 계층 검색, 계약 이력 조회, 담당자 권한별 마스킹  
  - 확인 기준(PoC): 법인 고객 1건 검색 시 소속 회선·계약·청구가 1화면에 표시, 조회 3초 이내

- 회선관리(Inventory)  
  - 기본 요구: 회선 자원 현황과 상태(가용/사용/장애)를 실시간 파악  
  - 기능 요구: 회선 신규·변경·해지 이력 관리, 대량 회선 일괄 조회/엑셀 취출  
  - 확인 기준(PoC): 회선 10만 건 목록 조회·필터 후 엑셀 다운로드가 오류 없이 완료

- 청구(Billing)  
  - 기본 요구: 사용량 기반 정확한 요금 산정·청구  
  - 기능 요구: 요금제·할인·부가서비스 반영 청구서 생성, 재청구·정정 처리  
  - 확인 기준(PoC): 표본 계약 100건 청구 산정 결과가 검증표와 100% 일치, 정정 건은 사유·이력 기록

- 개통·프로비저닝(Provisioning)  
  - 기본 요구: 주문 접수 후 개통까지 자동 처리, 실패 시 원인 가시화  
  - 기능 요구: 주문→자원 할당→망 구성→개통 완료 상태 추적, 실패 재처리(rollback/retry)  
  - 확인 기준(PoC): 신규 주문 1건이 자동 플로우로 개통 완료 상태까지 진행, 중간 실패 시 오류 단계·사유  
    표시 후 재처리 성공

- 주문관리(Order Management)  
  - 기본 요구: 다채널(포털·콜센터) 주문의 단일 접수·상태 관리  
  - 기능 요구: 주문 유형별 워크플로우, SLA 기반 처리 기한 알림  
  - 확인 기준(PoC): 포털 접수 주문이 상태 변경 시 담당자·고객에게 알림 발송, 처리 지연 시 에스컬레이션

[데모 지식] 위 예시는 "구체성보다 망라성 우선"(RFP작성이론 3-4) 원칙에 따라, 우선 업무 영역을 빠짐없이  
나열하고 각 영역의 확인 기준으로 검증가능성을 확보하는 방식임. 확인 기준을 다 채우기 어려운 항목은  
"벤더 제안에 위임"으로 남기되(3-6 참조), 최소한 항목 레벨과 기대 결과의 방향성은 제시함.

### 3-6. 구체화 vs 벤더 위임 트레이드오프

[이론] RFP 단계에서 모든 업무 요구를 구체화할 수는 없음. 시스템 요원이 부족한 기업은 무리하게 구체화  
하기보다 추상적 작성을 남기고 불명확한 부분은 벤더 제안에 맡기는 것도 방법임. 단, 최소한 요구는 항목  
레벨로 제시함. 업무 요구의 추상도는 불규칙해도 무방하며 "구체성보다 망라성이 중요"함(RFP작성이론 3-4,  
원문 p.13, p.18).

[데모 지식] PoC 관점에서 "구체화할지 위임할지"의 판단 기준은 "검증 리드타임 대비 학습 가치"임.  
- 구체화 권장: 검수 분쟁 소지가 크거나(정합률·성능 수치), 법·규제로 반드시 충족해야 하는 요구  
- 벤더 위임 권장: 구현 방식 선택지가 많고 벤더 노하우가 결과 품질을 좌우하는 요구(개통 자동화 방식,  
  대량 데이터 이관 전략 등). 위임 시에도 "기대 결과(확인 기준)"만은 제시하여 제안 비교축을 확보함.  
[데모 지식] 위임 요구라도 PoC 시나리오는 RFP에 제시해 두면 벤더 제안의 검증가능성이 확보됨 — "이런  
시나리오를 데모로 보여달라"는 요구는 구체적 기술을 지정하지 않으면서도 결과를 객관 비교하게 함  
(Fast fail 관점: 제안 단계에서 데모로 미리 실패를 걸러냄).

### 3-7. 엔드유저 협력 절차 — 숨은 요구 발굴

[이론] 컨설턴트가 엔드유저 협력을 얻는 표준 순서(RFP작성이론 3-5, 원문 p.18~19).  
1. 간단한 앙케이트(설문)로 예비 정보 입수  
2. 설문조사표를 기본으로 엔드유저 인터뷰 실시  
3. 핵심 인력에게 RFP 검토 미팅 참가 의뢰(매니저 경유가 유리), 업무 견학 셋업 의뢰  
4. 벤더에게 RFP 제시 전 반드시 여러 번 리뷰를 받음  
[이론] 업무 견학은 엔드유저의 참가 의식을 높이는 효과도 있음(원문 p.19).

[외부자료] 인터뷰는 대화형으로 개인·그룹에서 요구를 끌어내고, 설문은 다수 이해관계자에게 동시 배포해  
효율적으로 수집하며, 프로토타입은 초기 수집 이후 요구를 정련(refine)하는 기법임. 관찰(observation)은  
사용자가 말로 표현하지 못하는 실제 업무 행태를 포착함(PMWorld 360 / SDH).  
[외부자료] Requirements Elicitation Techniques (SDH) -  
https://sdh.global/blog/business/requirements-elicitation-techniques/  
[외부자료] Elicitation Techniques to Gather Business Requirements (BA Toolkit) -  
https://businessanalyststoolkit.com/ba-essentials-elicitation-techniques/

[데모 지식] "숨은 요구(unstated needs)"는 인터뷰만으로는 잘 드러나지 않음. PoC 프로세스에서 검증된 발굴  
순서는 다음과 같음(Learn and Pivot).  
1. 설문으로 정량 예비 데이터 확보(사용 빈도·불만 항목 순위) → 인터뷰 질문을 집중시킴  
2. 인터뷰에서 나온 요구를 "즉석 목업/화면 스케치"로 되비추어(reflect back) 확인 → 오해를 조기 발견  
3. 업무 견학(관찰)으로 "말로 안 한 우회 작업(workaround)"을 포착 → 진짜 요구는 종종 여기서 나옴  
4. 초기 프로토타입을 보여주고 반응을 관찰 → 정련. 완벽한 프로토타입 대신 빠른 버전으로 반복  
[데모 지식] 통신사 B2B에서 엔드유저는 사내 운영자(콜센터·개통·청구 담당)와 고객사 관리자로 나뉨. 양측을  
분리 인터뷰해야 함 — 사내 운영자는 "처리 효율·오류 감소"를, 고객사 관리자는 "가시성·셀프서비스"를 원해  
요구가 상충할 수 있으므로, 상충 지점을 RFP 단계에서 미리 식별하여 우선순위를 명시함.
