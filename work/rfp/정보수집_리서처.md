## 섹션 1. RFP의 3대 핵심과 문서 골격

### 1-1. RFP의 정의와 존재 이유

[이론] RFP(Request for Proposal, 제안요청서)는 HW·SW·서비스 구입 요건을 정리하여 발주사(구매자)가  
벤더(판매자)에게 제안을 의뢰하는 문서임. 작성 측과 읽는 측의 이해가 반드시 일치하지 않으므로, RFP는  
양자가 WIN-WIN 관계로 나아가도록 필요한 정보를 정확히 전달하는 수단임 (RFP작성이론.md 1절, p.6).  
[이론] IT 업계에 표준 서식은 없으나 "만들고 싶은 시스템의 개요를 벤더에게 정확히 전달"한다는 목적에서  
기본 작성법은 정해짐 (p.7).

[리서처 지식] 통신사 B2B 맥락에서 RFP의 성격은 일반 IT 대비 다음이 강조됨.  
- 발주 주체가 공공기관·대기업인 경우가 많아 RFP가 곧 평가·계약의 법적 기준 문서로 작동함  
- 네트워크·회선·클라우드·보안 등 상시 운영형 서비스가 대상이라 "구축 후 운영·SLA"까지 RFP 범위에 포함됨  
- 다수 벤더 경쟁입찰 구조에서 평가위원 배점표와 1:1 매핑되므로, 요구사항 누락은 곧 감점으로 직결됨

[외부자료] 통신사 RFP는 흔히 400개 안팎의 질의 항목, 네트워크 사양용 중첩 Excel 매트릭스, 여러 PDF  
첨부에 흩어진 SLA 요구, 복수 규제 프레임워크를 참조하는 컴플라이언스 섹션으로 구성되는 경향이 있음.  
- 10 RFP Automation Tools Telecom Companies Are Using in 2026 -  
  https://www.getanchor.ai/articles/rfp-automation-tools-telecom-2026

### 1-2. RFP에 필요한 3개의 핵심 정보 (3대 요구)

[이론] RFP에서 필요한 정보는 3가지로 집약됨 (p.7, 그림1-2).  
- 무엇을 하고 싶은가 → 업무 요구, 기술 요구, 운용 요구  
- 얼마에 하고 싶은가 → 예산  
- 언제까지 하고 싶은가 → 납기  

[이론] 거대 시스템이 수백 페이지에 달해도 "무엇을·얼마에·언제까지"의 3개 요구로 집약된다는 점은 불변임.  
가장 심플하게는 3행으로도 성립함(예: 분산 고객정보 일원화·2000만엔 이내·금년 12월말까지). RFP를 받는  
벤더는 이 3개 요구가 반드시 포함되어 있는지 체크할 필요가 있음 (p.7~8).

[리서처 지식] 통신사 B2B에서 3대 요구의 구체 형태.  
- 무엇을(WHAT): 회선/전용선/MPLS/SD-WAN 등 서비스 종류, 커버리지(전국 국사·해외 PoP), 대역·이중화 구성,  
  보안(망분리·전용회선 암호화), 관리 포털·API 연동 요구  
- 얼마에(HOW MUCH): 초기 구축비(회선 개통·장비)와 월 정액 이용료(러닝)를 분리 제시, 3~5년 TCO 기준  
- 언제까지(WHEN): 회선 개통 리드타임(국사 실사→포설→개통)이 길어 컷오버 역산이 일반 IT보다 보수적이어야 함

### 1-3. RFP 문서 전체 구성 (표준 골격)

[이론] RFP 본체는 다음 요소로 구성됨 (샘플 기준, p.11~12).  
- 0. 시작(인사문)  
- 1. 요지(취지)  
- 2. 시스템에 요구하는 기능 = 2-1 업무 요구 / 2-2 기술 요구 / 2-3 운용 요구  
- 3. 예산  
- 4. 프로젝트 기간  
- 5. 특기사항(제출요령·사무국)  

[이론] 3개 요소 중 가장 중요한 것은 "무엇을 하고 싶다"이며, 이는 다시 업무 요구·기술 요구·운용 요구로  
나뉨 (p.13).

[외부자료] 국내 공공 SW사업 RFP는 소프트웨어진흥법 제44조(과업범위)와 관리감독 지침 제11조(제안요청서  
작성)를 근거로 하며, 요구사항을 기능·성능·시스템장비구성·인터페이스·데이터·테스트·품질·제약·프로젝트관리·  
프로젝트지원 등 유형으로 상세화하는 표준 골격을 제시함.  
- 공공SW사업 제안요청서 작성을 위한 요구사항 실무 가이드(FESI) -  
  https://www.fesi.or.kr/post/2021-%EA%B3%B5%EA%B3%B5sw%EC%82%AC%EC%97%85-%EC%A0%9C%EC%95%88%EC%9A%94%EC%B2%AD%EC%84%9C-%EC%9E%91%EC%84%B1%EC%9D%84-%EC%9C%84%ED%95%9C-%EC%9A%94%EA%B5%AC%EC%82%AC%ED%95%AD-%EC%8B%A4%EB%AC%B4-%EA%B0%80%EC%9D%B4%EB%93%9C  
- SWIT 공공 SW사업 제안요청서 작성을 위한 요구사항 가이드 -  
  https://www.swit.or.kr/IS/web/isCbmRefView.jsp?ref_sq=1104&schCode=

[외부자료] 국제표준 ISO/IEC/IEEE 29148:2018은 요구사항 문서를 3계층(이해관계자 요구 StRS → 시스템  
요구 SyRS → 소프트웨어 요구 SRS)으로 정의하며, 각 요구가 갖춰야 할 속성(명확성·검증가능성·추적가능성)과  
필수 콘텐츠 요소를 규정함. IEEE 830-1998을 대체하는 최신 표준임.  
- ISO/IEC/IEEE 29148:2018 Requirements engineering -  
  https://www.iso.org/standard/72089.html  
- ISO/IEC/IEEE 29148 Requirements Specification Templates (ReqView) -  
  https://www.reqview.com/doc/iso-iec-ieee-29148-templates/

[리서처 지식] 이론의 5개 섹션(취지·기능·예산·기간·특기)과 공공/국제표준을 매핑하면 통신사 B2B RFP는  
다음 확장 골격을 권장함.  
- 0. 인사문 / 1. 사업 개요·추진 배경(취지) / 2. 요구사항(기능=업무, 기술=인프라·성능·보안, 운용=운영·SLA·이관)  
- 3. 사업비·대가 산정 / 4. 사업 기간·일정 / 5. 제안 안내(평가기준·배점·제출요령) / 6. 계약 조건(SLA·NDA·저작권)  
- 통신사 특성상 "운용 요구"에 SLA 지표(가용률·지연·손실·복구시간)를 정량 명시하는 것이 핵심 차별점임

### 1-4. 조달 4단계 내 RFP의 위치

[이론] 시스템 구축 전체는 「정보화 기획 → IT 리소스 조달 → 시스템 개발·도입 → 운용·보수」로 진행되며,  
RFP 작성은 "조달 단계"에 포함됨. 조달 단계는 다시 ①사전조사 ②RFP 작성 ③제안서 평가·결정 ④계약의  
4개 작업으로 나뉨. RFP 공표는 "조달 과정을 좌우하는 정보를 발신"하는 중요 마일스톤임 (p.8~9).

[리서처 지식] 통신사 B2B에서 ①사전조사 단계에 반드시 포함할 항목: 고객사 기존 회선·장비 현황 실사,  
망 구성도 확보, 국사/PoP 커버리지 확인, 경쟁사 서비스 벤치마킹. 이 실사 데이터가 RFP의 Capacity·이관  
요구 정확도를 좌우함.

### 1-5. RFP 작성 메리트와 미작성 리스크

[이론] 메리트(p.10): 요구를 명확·구체화, 우선순위 명확화, 벤더에 공평한 정보 제공, 복수 벤더 동시 의뢰로  
시간 단축, 평가축 확보, 투자 대 효과 검토 계기. 단기(3개월~반년) 안건에서도 RFP는 필수임.  
[이론] 미작성 리스크(p.10): 요구 애매로 과잉·일반론 제안, "말했다/안 했다" 전달 미스, 가격만으로 판단.  
3자(정보시스템부·현업·벤더)의 공통 인식 부재로 예외처리 폭증·커스터마이즈 비용 급증 사례 존재.

[외부자료] 우수한 통신 RFP는 상세 요구사항 명세, 명확한 평가기준, 표준화된 가격 포맷, 일정, 그리고  
동일 조건 비교를 가능하게 하는 채점 방법론을 갖춰야 함. 이 요소들이 곧 미작성/부실작성 리스크의 해소책임.

### 1-6. [리서처 지식] 요구사항 추적표(RTM)와 누락 방지 체크리스트

[리서처 지식] RFP 문서 골격을 실제 감점 없이 완성하려면 요구사항 추적표(RTM)로 항목 망라성을 관리해야 함.  
RTM은 각 요구사항이 제안서·평가·계약·검수까지 끊김 없이 추적되도록 연결하는 표임.  
- 권장 컬럼: 요구ID / 요구분류(업무·기술·운용) / 요구내용 / 우선순위(필수·권장·선택) / 출처(RFP 원문 조항) /  
  평가배점 매핑 / 제안서 대응 페이지 / 검증방법(시험·문서·시연) / 계약·SLA 반영 여부

[외부자료] RTM 베스트 프랙티스: 명확·단순 구조 유지, Requirement ID·상태 등 핵심 컬럼 중심, 요구사항을  
명확히 기술, 마일스톤이 아닌 전 생애주기에 걸쳐 지속 갱신, 순방향(요구→시험)·역방향(구현→요구) 양방향  
추적 검증. 이는 범위 이탈 조기 감지와 실패 리스크 감소에 기여함.  
- How to Make a Requirements Traceability Matrix (ProjectManager) -  
  https://www.projectmanager.com/blog/requirements-traceability-matrix  
- How to Create and Use a Requirements Traceability Matrix (Jama Software) -  
  https://www.jamasoftware.com/requirements-management-guide/requirements-traceability/how-to-create-and-use-a-requirements-traceability-matrix-rtm/

[리서처 지식] 통신사 B2B RFP 누락 방지 체크리스트(감점 리스크 사전 검출용).  
- 3대 요구 존재 확인: 무엇을·얼마에·언제까지가 모두 명기되었는가  
- Capacity 수치 명시: 전체 이용자수·동시 사용자수·대역·트래픽 피크 — 누락 시 벤더가 반드시 질문(이론 p.24)  
- SLA 정량 지표: 가용률(예 99.9%), 지연(RTT), 패킷손실률, 장애 복구시간(MTTR), 위약 조항 명시 여부  
- 운용·교육·이관 요구 누락 여부: 프로그램 개발에 관심 쏠려 경시되기 쉬움(이론 p.25)  
- 별도 예산 항목 명시: 보수비·연수비를 예산 내/별도 중 어디로 계상하는지(이론 p.33~34)  
- 계약 조항: 저작권 귀속·NDA·개인정보/망분리 컴플라이언스(이론 p.112, 통신 규제 반영)  
- 제출요령·평가배점·사무국 창구: 평가배점과 요구사항의 1:1 매핑 확인

## 섹션 9. RFP 샘플·워크시트 활용

### 9-1. 완성 샘플(상세판) 구조

[이론] 부록 상세판 샘플(p.108~112)은 본문 2페이지 압축 샘플(p.11~12)을 확장한 완성형으로 다음 구조를 가짐.  
- 0. 시작(인사문)  
- 1. 제안 요청의 취지 (정량 목표 포함, p.108)  
- 2. 시스템에 요구하는 기능  
  - 2-1. 업무 요구: (1) 업무 기본 요구 (2) 업무 기능 요구 (3) 업무 플로우 요구 (4) 계획 요구 (5) 장표 요구  
  - 2-2. 기술 요구: (1) IT 인프라 요구 (2) 시스템 능력 요구 (3) 장해 대책 요구 (4) 시큐리티 요구  
  - 2-3. 운용 요구(광의): (1) 운용 (2) 보수 (3) 교육·연수 (4) 이행 (5) 아웃소스 요구  
- 3. 예산 (초과 시 범위 명시, 보수 월간·연간 별도 견적, 실비정산 등)  
- 4. 프로젝트 기간 (시험운용 면밀 입안, 협의 제약 등)  
- 5. 특기사항 (제안서 구성·견적 명세, 저작권 귀속, NDA, 마감·사무국)  

[이론] 상세판은 본문 샘플 대비 업무 기능·플로우·장표 요구, 보수의 松竹梅(선택지) 제시, 이행·아웃소스  
요구, 액세스권 3단계 등이 구체화됨 (p.109~111).

[리서처 지식] 통신사 B2B 각색 — 상세판 샘플을 통신 서비스 조달용으로 치환한 예.  
- 1. 취지: 정량 목표를 통신 KPI로(예: 지점 간 통신 장애 연 X건 이하, 대역 부족 민원 Y% 감소)  
- 2-2 기술 요구: IT 인프라→회선/전용선/장비·PoP, 시스템 능력→대역·동시세션·확장성, 장해 대책→이중화·  
  우회경로·자동 절체, 시큐리티→망분리·회선 암호화·접근권한(관리자·운영자·조회자 3단계)  
- 2-3 운용 요구: 운용→NOC 24/365 모니터링, 보수→온사이트 SLA 등급(松竹梅=Gold/Silver/Bronze),  
  이행→기존 회선 무중단 전환, 아웃소스→관제 위탁 범위  
- 5. 특기사항: 개인정보보호법·망분리 규제 준수 서약, 저작권·NDA, 정보보안 서약서

### 9-2. 워크시트 빈칸 항목

[이론] 부록 워크시트(p.113~116)는 항목만 제시한 빈 템플릿으로 각 섹션 필수 기입 항목을 나열함.  
- 0. 시작: 인사문 (p.113)  
- 1. 취지: 경영전략·경영자 메시지 / 자사 경영환경·문제점 / 시스템 기대 효과 (p.113)  
- 2-1. 업무 요구: 업무 기본·기능·플로우 요구 / 데이터 요구 / 화면 요구 / 장표 요구 / 기타 (p.114)  
- 2-2. 기술 요구: IT 인프라 / 시스템 능력 / 장해 대책 / 시큐리티 요구 / 기타 (p.114)  
- 2-3. 운용 요구(광의): 운용(협의) / 보수 / 교육·연수 / 이행 / 아웃소스 요구 / 기타 (p.115)  
- 3. 예산: 예산액 제시 유무 / 보수비용 요구 / 실비정산 요구 / 기타 (p.115)  
- 4. 프로젝트 기간: 시스템 가동예정일 / 테스트·이행 기간 예정 / 업무상 스케줄 제약 / 기타 (p.116)  
- 5. 특기사항: 제안서 체재·포맷 지정 / 마감일시·제출방법 / 프레젠테이션 안내 / 사무국 연락처 / 기타 (p.116)

### 9-3. 예제 RFP 조립 방법

[이론] 워크시트 빈칸 항목을 각 섹션 순서(취지 → 업무 요구 → 기술 요구 → 운용 요구 → 예산 → 프로젝트  
기간 → 특기사항)대로 채워 나가면 상세판 샘플과 동일한 구조의 RFP가 조립됨 (p.113~116).  
[이론] 채우는 원칙: 3대 요구(무엇을·얼마에·언제까지)를 빠짐없이 포함(망라성 우선), 구체화가 어려운 항목은  
벤더 제안에 위임하되 최소 항목 레벨로는 제시, 별도 예산 항목·계약 조항(저작권·NDA)·제출요령 명시 (p.7,  
p.18, p.33, p.44, p.112).

[리서처 지식] 워크시트 조립 시 리서처 관점의 품질 게이트(제출 전 셀프 리뷰).  
- 망라성 우선 원칙 적용: 구체성이 부족해도 항목 자체를 빠뜨리지 않는 것이 최우선(이론 p.13, p.18)  
- 각 빈칸을 RTM 요구ID로 부여하여 이후 제안서·평가·계약까지 추적 가능하게 함  
- 통신 B2B 필수 추가 워크시트 칸: SLA 지표표, Capacity 수치표, 커버리지(국사/PoP) 목록, 규제 준수 체크란

[외부자료] 통신 RFP 응답에는 엔지니어링·상품·법무·컴플라이언스·재무·영업 등 다부서 기여가 필요하므로,  
워크시트/RFP는 섹션별 담당(오너십) 지정과 진행 가시성을 갖추는 것이 효과적임(리서처는 요구사항 오너로  
배점 매핑을 담당). SLA는 서비스 설명·측정 지표·양자 책임·위약·지표 추가/삭제 프로토콜을 포함해야 함.
