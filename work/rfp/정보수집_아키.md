## 섹션 4. 기술 요구 작성

### 4-0. 섹션 개요와 위치

[이론] 기술 요구와 운용 요구를 합쳐 "시스템 요구"라 부름. 업무 요구는 정보시스템 부서·현업 부서가  
협력해 밝혀내지만, 시스템 요구 정리는 정보시스템 부서(=아키텍트 관점)가 주체가 되는 작업임.  
[이론] 기술 요구는 OS·프로그래밍 언어·DB·미들웨어에 대해 "어떤 기술/제품을 선택할지, 그 중 어떤  
기능을 이용할지"를 정보시스템 부서 관점에서 정리하는 것임.  
[아키 지식] 통신사 B2B 제안에서 기술 요구 섹션은 벤더 아키텍처 제안의 채점 기준이 되므로, 측정 가능한  
수치(정량)로 명시할수록 제안 비교가 명확해지고 감점·분쟁 리스크가 줄어듦.

### 4-1. 기술 요구 4대 분류 (작성 골격)

[이론] 기술 요구는 다음 4개 요소로 구성됨 (RFP작성이론 4-2, 상세판 샘플 p.111 근거).  
- IT 인프라: OS·미들웨어, 하드웨어, 네트워크  
- 시스템 능력: 퍼포먼스(성능), Capacity(용량), 확장성  
- 장해 대책: HW·SW 장해, 네트워크 장해, 장해 시 대체안  
- 보안(시큐리티) 요구: 정보 누출 방지 대응책, 액세스권 설정(예: 관리자·매니저·일반사원 3단계)  

[이론] 성능은 시스템 전체 처리능력 외에 HW·네트워크 등 각 요소까지 브레이크다운하여 기술하는 것이  
바람직함. 장해 대책도 분야별로 브레이크다운해 두는 것이 바람직함.  
[이론] 이들 항목은 추상적이라도 좋으므로 프로젝트 규모·유저 기업 사정에 맞춰 케이스 바이 케이스로  
망라해 둠 (구체성보다 망라성 우선).  
[아키 지식] 작성 순서 권장: (1) IT 인프라 → (2) 시스템 능력(정량 지표) → (3) 장해 대책(가용성 목표)  
→ (4) 보안. 뒤 항목이 앞 항목의 스펙을 결정하므로(예: Capacity 수치가 HW 스펙을 결정), 능력·장해  
목표를 먼저 확정한 뒤 인프라 스펙을 역산하는 것이 실무상 효율적임.

### 4-2. IT 인프라 요구 (OS/미들웨어/HW/네트워크)

[이론] IT 인프라 요구는 OS·미들웨어, 하드웨어, 네트워크로 구성됨.  
[이론] 제품 지정 시에도 "능력에 맞는 기술 요구"가 중요함. 범용적·시장 수요 높은 제품 선택으로 유지보수  
리스크를 회피함. 마이너 언어·서포트 미확립 해외 제품은 3년 후 유지보수 리스크를 수반함.  
[이론] 기존 인프라·설비 활용 요구도 실무 RFP에 자주 나타남 — 이유 예: 초기 투자 억제, 신규 기기 설치  
스페이스 부재, 리스 기간 잔존, 사업소 이전 플랜 등.  

[아키 지식] IT 인프라 요구 명시 체크리스트 (아키텍트 관점).  
- OS/미들웨어: 지원 OS·버전, WAS·DBMS·연계 미들웨어 종류, 라이선스 방식(코어/사용자/구독),  
  기술지원(EOS/EOL) 잔여기간 — EOS 임박 제품 배제 조건 명시 권장  
- HW: CPU 코어·메모리·스토리지(IOPS 포함)·랙 규격, 물리/가상/컨테이너 방식, 국산 vs 외산 제약  
- 네트워크: 회선 대역폭, 이중화 회선 유무, 사업자망 연동 규격(전용회선·VPN·인터넷), IP 대역/방화벽  
  정책, DMZ 구성  
- 배포 형태: 온프레미스 / 프라이빗 클라우드 / 퍼블릭 클라우드 / 하이브리드 중 선택 기준 명시  

[아키 지식] 클라우드 vs 온프렘 선택 기준 (통신사 B2B 실무).  
- 클라우드 우선: 트래픽 변동 큼(탄력적 확장 필요), 초기 CAPEX 최소화, 글로벌·다지역 서비스, 빠른 출시  
- 온프렘/프라이빗 우선: 규제상 데이터 국내·자사 보관 의무, 망분리 대상 업무, 초저지연 요구, 기존 설비  
  활용·리스 잔존, 장기 운영 시 TCO 유리  
- 공공·금융 고객 대상 클라우드 제안 시 CSAP 인증 보유 여부를 인프라 요구에 필수 조건으로 명시  

### 4-3. 시스템 능력 요구 (성능/용량/확장성) — NFR 정량화

[이론] 시스템 능력은 퍼포먼스(성능)·Capacity(용량)·확장성으로 구성됨.  
[이론] Capacity 요구(전체 유저수·동시 사용 유저수)는 HW 스펙 검토에 필요한 숫자이므로 누락 시 벤더가  
반드시 질문함 — 즉 사전에 정량 제시하는 것이 필수임.  

[아키 지식] 비기능 요구(NFR)를 측정 가능한 수치로 명시하는 법 (아키텍트 핵심 역량).  
NFR은 "빠를 것"·"안정적일 것" 같은 형용사가 아니라 [지표 + 목표값 + 측정조건]의 3요소로 기술해야  
제안·검수·SLA 연계가 가능함.  
- 성능: 평균 응답시간(예: 화면 조회 2초 이내), 95퍼센타일 응답시간, TPS(초당 트랜잭션),  
  동시접속자 수, 배치 처리시간(예: 일 마감배치 2시간 이내)  
- 용량: 전체 사용자 수, 동시 사용자 수, 연간 데이터 증가량(GB/년), 피크 트래픽 배수(평상 대비 N배)  
- 확장성: Scale-out 지원 여부, 무중단 증설 가능 여부, 향후 3년 사용자·트래픽 증가 목표 수용  

[아키 지식] NFR 작성 예시 표 (RFP 삽입용 양식 권장).  

| 구분 | 지표 | 목표값 | 측정조건 |
|------|------|--------|----------|
| 성능 | 화면 평균 응답시간 | 2초 이내 | 동시접속 100명, 정상부하 시 |
| 성능 | 온라인 TPS | 500 TPS 이상 | 피크 시간대 |
| 용량 | 동시 사용자 | 1,000명 | 개통 1년차 기준 |
| 용량 | 데이터 증가량 | 500GB/년 | 3년 보관 전제 |
| 확장성 | 무중단 Scale-out | 지원 | 서비스 중단 없이 노드 증설 |

[외부자료] TPS 계산은 Little's Law 기반 — TPS = 동시사용자(CU) / Request Interval(응답시간 + Think  
Time). 동시사용자는 트랜잭션 발생 사용자와 미발생 사용자를 모두 포함함.  
- RFP 비기능 요구 성능 지표 명시(응답시간·TPS·동시접속자) / 지식덤프 성능테스트  
  http://www.jidum.com/jidums/view.do?jidumId=562  
- Little's Law 활용 성능 계산 / SKCC Engineering  
  https://engineering-skcc.github.io/performancetest/LittlesLaw/  
[외부자료] 실제 RFP 관행상 "평균 응답 속도 2초 이내", "동시 접속 처리 100명 이상" 등으로 정량 명시함.  
목표 응답수준을 계약업체가 현 시스템 체크 후 수행계획서에 명시하도록 요구하는 사례도 있음.  
- RFP 작성법(비기능 요구사항 정량 명시) / 이랜서 블로그  
  https://www.elancer.co.kr/blog/detail/790  

[외부자료] ISO/IEC 25010(SQuaRE) 제품 품질 모델 — NFR 분류의 국제 표준 근거. 2023 개정판은 9개  
품질특성 정의: 기능적합성(Functional Suitability), 성능효율성(Performance Efficiency), 호환성  
(Compatibility), 상호작용성(Interaction Capability), 신뢰성(Reliability), 보안(Security),  
유지보수성(Maintainability), 유연성(Flexibility), 안전성(Safety). 신뢰성은 가용성·결함내성·복구성을  
포함하고, 보안은 기밀성·무결성·인증성·책임추적성·부인방지를 포함함. RFP의 시스템 능력·장해·보안 요구를  
이 표준 특성에 매핑하면 누락 없이 망라 가능함.  
- ISO/IEC 25010:2011 System and software quality models / ISO  
  https://www.iso.org/standard/35733.html  
- ISO/IEC 25010 Quality Model / arc42  
  https://quality.arc42.org/standards/iso-25010  

### 4-4. 장해 대책 요구 (가용성/이중화) — SLA 연계

[이론] 장해 대책은 HW·SW 장해, 네트워크 장해, 장해 시 대체안으로 구성됨. 분야별 브레이크다운 권장.  
[이론] 장해 대책은 가능하면 기준(현상 동일 수준 또는 현상 플러스)을 제시하면 벤더로부터 양호한 제안을  
받기 쉬워짐.  

[아키 지식] 통신사 B2B 특유의 이중화·SLA 연계 기술요구 (아키텍트 관점).  
장해 대책은 "가용성 목표(정량) → 이중화 구성(수단) → SLA 배상(계약)"의 3단 연계로 설계함. 가용성  
목표만 RFP에 못박고 이중화 방식은 벤더 제안에 위임하면(제안 위임형) 더 나은 아키텍처를 유도 가능함.  
- 가용성 목표: 서비스별 목표 가용률(예: 99.9% / 99.95% / 99.99%)을 정량 명시  
- 이중화 범위: 서버·스토리지·네트워크·DB·전원·회선까지 어느 계층을 이중화할지 지정  
- 재해복구(DR): RTO(복구 목표시간)·RPO(복구 목표시점) 정량 명시, DR 센터 거리·동기/비동기 복제  
- 장해 대체안: Active-Active / Active-Standby, 자동 페일오버 시간(예: 30초 이내)  

[외부자료] 가용률별 연간 다운타임 — 가용성 = Uptime / (Uptime + Downtime). 99.9%는 연간 약 8시간  
46분, 99.99%는 약 52분의 다운타임을 허용함. 99.9%↔99.99% 격차를 줄이려면 네트워크·스토리지·DB·DNS  
까지 전체 아키텍처를 이중화해야 함. 서버 이중화만으로는 SLA 99.99% 달성 불가.  
- 서버 이중화와 가용성 99.9% vs 99.99% 차이 / 스피디 블로그  
  https://www.speedykorea.com/blog/server-redundancy-availability-sla-difference  
[외부자료] 데이터센터 Tier 등급은 전기·통신·소방 등 설비 백업 구성 수준을 등급화한 것으로, 보증 연간  
가동시간이 SLA 계약으로 보장됨. RFP에 요구 Tier 등급(예: Tier 3 이상)을 명시하면 인프라 가용성 수준을  
정량 지정 가능함.  
- 데이터센터 Tier 등급 가이드(Tier 1~4) / 토요컨설턴시  
  https://www.toyocs.net/blog/2025/10/01/  
[아키 지식] SLA 연계 주의점: RFP의 가용성 목표는 운용·계약 섹션의 SLA 배상 조항과 수치가 일치해야 함  
(딜·케어와 정합성 확인 필요). "영업시간=가동시간=보수시간" 기본 사고를 적용해 오버스펙(불필요한  
24×365 온사이트) 방지도 함께 검토함 (RFP작성이론 5-2 연계).

### 4-5. 보안(시큐리티) 요구 — 망분리·개인정보·클라우드 인증

[이론] 보안 요구는 정보 누출 방지 대응책, 액세스권 설정(예: 관리자·매니저·일반사원 3단계)로 구성됨.  

[아키 지식] 통신사 B2B 보안 요구 명시 항목 (법무 리스크 사전 검토 관점).  
- 접근통제: 역할 기반 접근제어(RBAC) 등급, 최소권한 원칙, 계정·권한 변경/말소 절차  
- 암호화: 전송구간(TLS)·저장데이터 암호화, 개인정보 항목 암호화 대상 명시  
- 로그·감사: 접속기록 보관기간·위변조 방지, 이상행위 탐지  
- 망분리: 대상 업무·방식(물리적/논리적) 지정 여부  
- 규제 준수: 개인정보보호법, (공공·금융 대상 시) CSAP·ISMS-P 인증 요구  

[외부자료] 망분리 요구 — 내부망/외부망을 분리하는 네트워크 보안기법. 물리적 분리(PC 2대)와 논리적  
분리(가상화로 PC 1대에 업무·인터넷 영역 분리, VDI/CBC 방식)로 구분됨. 금융분야는 2013년 대규모  
전산사고 후 물리적 망분리를 도입했으며, 최근 로드맵은 대상업무 축소·물리/논리 선택가능성 부여로 개선  
중임. RFP에 망분리 대상·방식을 명확히 지정해야 벤더 아키텍처가 규제에 부합함.  
- 금융분야 클라우드 및 망분리 규제 개선방안 / 금융위원회  
  https://www.fsc.go.kr/no010101/77672

### 4-6. 상세 지정형 vs 벤더 제안 위임형 (작성 전략)

[이론] 기술 요구 작성법은 두 타입임.  
- 상세 지정형: 사용 기술을 확실히 제안에 포함시킴(메리트). 단, 더 좋은 기술이 있어도 제안 안 될 수  
  있음(디메리트). 예: "특정 OS 버전을 사용할 것".  
- 벤더 제안 위임형: 애매하게 작성. 예: "OS는 최신·시장 share 높은 것을 사용".  
[이론] 문언 차이가 커도 결과적으로 동일 기술·제품이 제안되는 경우가 많아, 제안 위임형이 반드시  
뒤떨어지는 것은 아님.  
[이론] 금융업 등 시스템 밀접·사회적 영향 큰 기업은 기술 요구가 엄격(OS 상세 버전 지정, 3·4중  
장해대책). 시스템 요원 확보가 어려운 중소기업은 벤더 제안을 받는 편이 더 나은 기술 검토가 되기도 함.  

[아키 지식] 항목별 지정 전략 권장 (아키텍트 판단).  
- 상세 지정 권장: 규제 준수 항목(망분리·CSAP·개인정보 암호화), 기존 인프라 연동 규격, 필수 가용성  
  목표·SLA 수치 — 협상·법적 리스크 직결 항목은 못박음  
- 벤더 위임 권장: 이중화 아키텍처 방식, 성능 튜닝 수단, 클라우드 세부 서비스 구성 — 목표값만 제시하고  
  달성 수단은 위임하여 최적 제안 유도  
- 공통 원칙: 최소한 요구는 항목 레벨로 제시(RFP작성이론 3-4·4-3), 구체성보다 망라성 우선

### 4-7. 기존 인프라 활용 요구 (필수 명시 항목)

[이론] 기존 인프라·설비 활용 요구는 실무 RFP에 자주 나타남. 이유: 초기 투자 억제, 신규 기기 설치  
스페이스 부재, 리스 기간 잔존, 사업소 이전 플랜 등.  
[아키 지식] 기존 인프라 활용 요구 명시 항목: 재사용 대상 자산 목록(서버·회선·라이선스), 리스 잔여기간,  
연동 필수 레거시 시스템·인터페이스 규격, 데이터 이행 범위. 활용 자산을 명시하지 않으면 벤더가 전면  
신규 구성으로 과잉 견적하거나 연동 누락으로 재작업이 발생함.

## 요약 (섹션 4 작성 핵심)

1. 기술 요구는 IT 인프라·시스템 능력·장해 대책·보안 4대 분류로 망라함 (구체성보다 망라성 우선).  
2. NFR(성능·용량·확장성·가용성)은 [지표+목표값+측정조건] 3요소로 정량 명시 — ISO/IEC 25010 품질특성  
   매핑으로 누락 방지.  
3. 장해 대책은 가용성 목표(99.9~99.99%)→이중화→SLA 배상의 3단 연계로 설계, 딜·케어와 수치 정합.  
4. 보안 요구는 개인정보보호법 안전성 확보조치·망분리·CSAP(공공)·ISMS-P를 준수 조건으로 명시.  
5. 규제·연동·SLA 항목은 상세 지정, 아키텍처·튜닝 수단은 벤더 위임으로 전략 배분.  
6. 기존 인프라 활용 자산·연동 규격을 명시해 과잉 견적·연동 누락을 방지함.
