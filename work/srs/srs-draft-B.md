## 4. 입력 데이터 요구사항

> 시스템이 처리·판정에 사용하는 입력 데이터를 도메인별(4.1 ~ 4.6)로 묶어 필드 단위로 상세 나열함.  
> 필드 누락은 개발 단계 인터페이스 재협의를 유발하므로 망라성을 우선함.

- ⚠ 가입자 정보·통지 대상 연락처는 개인정보를 포함할 수 있으므로 저장·조회 시 필드 단위 마스킹 정책을  
  적용함(별첨 마스킹 규칙 준거). [REQ-TEC-04]
- ⚠ 입력 데이터는 처리·판정에 필요한 최소 항목만 수집함(최소수집 원칙). 가입자 컨텍스트(4.4)는 집계  
  단위로만 수신하고 개인 식별자(성명·주민번호·단말 IMEI 등)를 포함하지 않음. [REQ-TEC-04]
- ⚠ 원천 데이터의 원본 무결성 확인을 위해 수신 시 서명·해시 검증 대상 필드를 함께 수집함(상세는 5.1). [REQ-TEC-04]

### 4.1 사고 이벤트 정보(MNO 수신)

> MNO 망에서 Webhook/API로 실시간 수신하는 원천 사고 이벤트. 심각도 판정·통지 트리거의 기준 데이터임.

- 이벤트ID(원천 고유식별자) [REQ-TEC-01]
- 수신시각(Webhook 수신 타임스탬프, SLA 측정 기준) [REQ-TEC-01]
- 사고 발생시각(MNO 원천 발생 타임스탬프) [REQ-TEC-01]
- 사고유형(장애/보안 구분 및 세부 유형 코드) [REQ-BIZ-01]
- 심각도 원천값(MNO 제공 원천 등급) [REQ-BIZ-01]
- 영향 국사/셀 식별자(장애 발생 위치) [REQ-BIZ-02]
- 영향 서비스 범위(음성/데이터/SMS 등 서비스 구분) [REQ-BIZ-02]
- 원천 서명·서명 알고리즘 식별자(위변조 검증용) [REQ-TEC-04]
- 재전송 여부·재전송 시퀀스(중복/재전송 병합용) [REQ-TEC-01]

### 4.2 영향범위 산출 입력

> 자사 가입자 분포와 MNO 영향정보를 조인하여 영향 규모·지역·서비스를 산출하기 위한 입력. 조회 성능(P99 3초)의 기반 데이터임.

- 자사 가입자 분포(지역·서비스·요금제 집계 차원) [REQ-BIZ-02]
- 조인 키 — 지역 식별자(행정구역/기지국 매핑) [REQ-BIZ-02]
- 조인 키 — 망 식별자(MNO 망/셀 매핑) [REQ-BIZ-02]
- 조인 키 — 서비스 구분(음성/데이터/SMS 코드) [REQ-BIZ-02]
- 사전집계 뷰(지역×서비스×요금제 사전 집계 결과) [REQ-TEC-02]
- 사전집계 뷰 갱신시각(캐시 신선도 판정 기준 타임스탬프) [REQ-TEC-02]

### 4.3 통지 대상 정보

> 심각도·시간대에 따라 통지 대상·채널·발송 순서를 결정하기 위한 담당자 및 채널 데이터임.

- 담당자 사번(통지 대상 식별자) [REQ-BIZ-01]
- 소속 팀ID(에스컬레이션·집계 단위) [REQ-BIZ-01]
- 역할 등급(담당자/슈퍼바이저/상황실 구분) [REQ-BIZ-01]
- 통지 채널 우선순위(API/SMS/이메일 순서) [REQ-TEC-03]
- 채널 헬스 상태(단일 채널 장애 시 전환 판정용) [REQ-TEC-03]
- 수신 가능 시간대(야간/근무 구분) [REQ-BIZ-01]
- 연락처(휴대전화·이메일, ⚠ 마스킹 대상) [REQ-TEC-04]

### 4.4 가입자 컨텍스트(집계)

> 통지 우선순위·정책 판단을 보조하는 가입자 집계 정보. ⚠ 개인 식별자를 포함하지 않는 집계 단위 데이터임.

- 영향 가입자 규모(지역·서비스별 집계 건수) [REQ-BIZ-02]
- 등급 분포(VIP/일반 비율 집계) [REQ-BIZ-02]
- 최근 클레임 이력(집계 건수·추이) [REQ-BIZ-02]
- 해지의향 지표(집계 스코어) [REQ-BIZ-02]
- ⚠ 개인 식별자(성명·연락처·단말 식별자 등) 미포함 — 집계·범위 단위만 수신함. [REQ-TEC-04]

### 4.5 정책 판단 보조 정보

> 강화 정책·지연 판정·AI 예측 보조에 사용하는 컨텍스트 데이터. 판정의 일관성·설명가능성 근거임.

- 과거 유사사고 이력(사고유형·영향규모·처리결과) [REQ-BIZ-05a]
- MNO 원천 데이터 적시성(수신 지연 여부·지연 시간) [REQ-BIZ-03]
- 팀 통지 준수 이력(준수율 집계) [REQ-OPS-02]
- 시간대 구분(야간·대량 클레임 구간 플래그) [REQ-OPS-02]
- AI 예측 신뢰도(예측 결과 신뢰도 점수) [REQ-BIZ-05a]

### 4.6 MNO 복구정보(ETR 원천)

> MNO가 제공하는 복구 진행 원천 데이터. 15분 주기 ETR 갱신·복구 상태 통지의 입력이며 원천은 MNO 제공에 의존함.

- 복구 진행률(MNO 제공 복구 진척도 %) [REQ-BIZ-03]
- 예상 복구시각(ETR, MNO 제공 예상 완료 시각) [REQ-BIZ-03]
- 복구정보 수신시각(ETR 원천 수신 타임스탬프, 갱신 주기 판정 기준) [REQ-BIZ-03]
- 복구 상태(진행 중/완료/미제공 구분) [REQ-BIZ-03]
- 원천 제공 여부(MNO 미제공 시 면책 판정 근거) [REQ-BIZ-03]

## 5. 데이터 가공·검증(전처리) 요구사항

> 입력 데이터가 판정 로직에 들어가기 전 거치는 검증·정규화·AI 입력 전처리 규칙을 명세함.  
> 원천 데이터 품질 편차를 흡수하여 후속 판정·SLA 측정의 일관성을 확보함.

### 5.1 원천 검증·보정

> 원천 이벤트·복구정보의 유효성·무결성·중복을 검증하고 결측을 보정하는 전처리 대상임.

- 페이로드 스키마 검증(필수 필드·타입·값 범위 검증) [REQ-TEC-01]
- 서명 검증(원천 서명·해시 대사로 위변조 여부 판정) [REQ-TEC-04]
- 중복 이벤트 제거(이벤트ID·시퀀스 기준 중복 폐기) [REQ-TEC-01]
- 재전송 병합(재전송 시퀀스 병합·최신값 우선 반영) [REQ-TEC-01]
- 결측 필드 보정(기본값·직전값 보정 및 보정 사실 기록) [REQ-TEC-01]
- 심각도 원천값 이상치 탐지(허용 등급 외 값 예외 처리) [REQ-BIZ-01]

### 5.2 정규화

> 코드·식별자·시각·집계 차원을 표준화하여 조인·판정·SLA 자동 측정의 정합성을 확보하는 전처리 대상임.

- 사고유형 코드 표준화(MNO 원천 코드 → 내부 표준 코드 매핑) [REQ-TEC-01]
- 지역/망 식별자 매핑(행정구역·기지국·MNO 망 식별자 정규화) [REQ-BIZ-02]
- 서비스 코드 표준화(음성/데이터/SMS 등 서비스 구분 통일) [REQ-BIZ-02]
- 시각 표준화 — 수신시각 기준 정렬 및 NTP 동기로 시각 정합성 확보(수신 ~ 통지 1분 SLA 자동 측정 전제) [REQ-TEC-01]
- 영향범위 집계 차원 정규화(지역×서비스×요금제 집계 차원 통일) [REQ-BIZ-02]

### 5.3 AI 입력 전처리

> 과거 사고 데이터·백테스트 데이터셋을 AI 예측·응대지원 입력으로 가공하는 전처리 대상. 결과는 참고용·HITL 전제임.

- 과거 사고 데이터 품질 검증(결측·이상치·라벨 정합성 점검, 표본 부족 시 룰 폴백 표시) [REQ-BIZ-05a]
- 가명·집계 처리(학습·추론 입력의 개인 식별자 제거·가명화·집계화) [REQ-TEC-04]
- 백테스트 데이터셋 정규화(사고유형·영향규모 스케일 정규화·기간 정렬) [REQ-TEC-05]
- 응대지원 입력 정규화(FAQ·응대 이력 텍스트 정제·프롬프트 입력 표준화) [REQ-BIZ-05b]

### 5.4 캐시 신선도·대사 기준

> 조회 성능(P99 3초)과 표본 대사 정확성(100건 100% 일치)을 양립시키기 위한 스냅샷·신선도 판정 규칙임.

- 사전집계 뷰·캐시에 신선도 상한(TTL)을 설정하고, TTL 초과 데이터는 재집계 후 조회에 사용함. [REQ-TEC-02]
- 캐시 신선도(갱신시각)와 조회 응답의 정합성을 유지하여 라이브 사고 중 읽기/쓰기 경합에 대비함. [REQ-TEC-02]
- 표본 100건 100% 대사는 조회 응답과 원천 집계를 동일 스냅샷(동일 갱신시각) 기준으로 대조하여 판정함. [REQ-TEC-02]

## 12. 결과 필수 포함 정보(필드)

> 처리 결과(통지 결과·조회 응답)에 반드시 포함할 필드를 전량 나열함. 사후 추적·설명가능성·SLA 판정·감사에  
> 필요한 최소 집합을 망라함(설명 가능성 원칙 연계). AI 판정 포함 시 신뢰도 점수·사유 코드를 함께 기록함.

- 판정 요청ID(처리 단위 고유식별자) [REQ-BIZ-01]
- 이벤트ID(원천 사고 이벤트 식별자) [REQ-BIZ-01]
- 사고유형(장애/보안 및 세부 유형) [REQ-BIZ-01]
- 심각도(긴급/높음/보통 판정 결과) [REQ-BIZ-01]
- 처리 일시(판정·발송 처리 타임스탬프) [REQ-BIZ-01]
- 결과 상태(11장 처리 결과 상태값 중 하나) [REQ-OPS-01]
- 통지 대상(담당자 사번·팀 단위) [REQ-BIZ-01]
- 통지 채널(API/SMS/이메일 등 발송 채널) [REQ-TEC-03]
- 발송 시각(채널별 발송 타임스탬프) [REQ-TEC-01]
- 수신 확인(채널별 수신·확인 여부·시각) [REQ-TEC-03]
- SLA 판정 — 수신 ~ 통지 지연(시스템 전송 1분 기준) [REQ-TEC-01]
- SLA 판정 — 인지 ~ 담당자 도달(종단 5분 기준) [REQ-OPS-02]
- 적용 정책ID(판정에 적용된 정책 식별자) [REQ-OPS-02]
- 적용 룰(허용/억제 룰 식별자·유형) [REQ-TEC-04]
- 영향범위 요약(지역·서비스·규모 집계) [REQ-BIZ-02]
- ETR(예상 복구시각) [REQ-BIZ-03]
- 복구 상태(진행 중/완료/미제공 구분) [REQ-BIZ-03]
- AI 예측 신뢰도 점수(예측 판정 신뢰도, 참고용) [REQ-BIZ-05a]
- 사유 코드(통지·지연·면책 판정 근거 코드) [REQ-TEC-05]
- 담당자 사번(처리·확인 담당자 식별자) [REQ-BIZ-01]
- 팀ID(담당 팀·에스컬레이션 단위) [REQ-BIZ-01]
