# SRS 초안 — SEC4. 기술 요건(NFR)·데이터 요건

> 문서 상태: **초안(DRAFT)** — 05단계 SRS 본문 SEC4. 스펙 아젠다 확정(자동 승인) 후 아키 작성.  
> 근거: RFP분석서 `work/rfp/rfp-analyze.md`(A2 REQ-TEC 대응가능·조건, A4 기술·일정·계약 리스크),  
> SRS 가이드 `work/srs/srs-guide.md`(S4 NFR·S5 데이터 요건의 작성원칙·정량표·체크리스트).  
> 원칙: "측정할 수 없으면 요구가 아님" — 모든 NFR을 **[지표 + 목표값 + 측정조건]** 3요소로 정량 기술.  
> NFR-ID는 가이드 S4 체계를 그대로 승계함(신규 ID·임의 수치 미생성). 각 항목에 근거 REQ-ID를 부기함.  
> 추적: 각 NFR·데이터 항목은 인수기준(SEC5)·RTM(SEC6)으로 연결됨(측정 가능성·추적성을 스펙 리뷰 대상으로 의식).  
> 작성: 아키(솔루션 아키텍트) / 리뷰(측정 가능성·추적성): 스펙 / 조율: 클로니.

## 4.0 작성 범위·원칙

- **대상 정형화**: RFP 2-2 기술 요구(REQ-TEC-01~06)·2-2 Capacity표·REQ-OPS-01(가용성)을 NFR로,  
  RFP 데이터 관련 요구(사고·영향범위·통지·복구·리포트·AI)를 데이터 요건으로 정형화함.  
- **NFR 12건 구성**: 성능 3(NFR-P01~03) · 용량 3(NFR-C01~03) · 가용성/이중화/확장/복구 4(NFR-A01~A04)  
  · 보안 2(NFR-S01~S02). ISO/IEC 25010 매핑 = 성능효율성(P·C)·신뢰성(A)·보안성(S)로 누락 방지함.  
- **작성 순서**: 시스템 능력(성능·용량) → 장해 대책(가용성·DR) → 보안 순으로 확정함(뒤 항목이 앞 스펙  
  결정). **가용성 목표(정량)는 못 박고, 이중화 방식(Active-Active/Standby)·배포형태는 벤더 제안에 위임함**.  
- **규제 상세·수단 위임 배분**: 인증·전송 암호화·개인정보 취급·연동 보안은 상세 지정하고, 세부 이중화  
  구성·튜닝 수단은 위임함.  
- **수치 출처**: 목표값·측정조건은 RFP/RFP분석서 근거만 사용함. 신규 NFR-ID·임의 수치를 생성하지 않음.

## 4.1 기술 요건(NFR)

### 4.1.1 NFR 정량 명세표

각 NFR은 근거 REQ-ID로 인수기준(SEC5)·SLA·RTM(SEC6)에 추적 연결됨. Capacity표 항목(요구ID 미부여  
확인기준)은 RFP분석서 귀속에 따라 **REQ-TEC-06(2-2 Capacity)** 로 부기함(신규 ID 미생성).

| 구분 | 지표(NFR-ID) | 목표값 | 측정조건 | 근거 REQ-ID |
|------|--------------|--------|----------|-------------|
| 성능 | 수신~통지 지연(NFR-P01) | 1분 이내 | 수신~통지 종단, 월 측정, 정상·피크 포함 | REQ-TEC-01 |
| 성능 | 통지 전송 성공률(NFR-P02) | 99.9% 이상 | 월 단위 집계, 정상·피크 포함 | REQ-TEC-01 |
| 성능 | 영향범위 조회 응답(NFR-P03) | 평균 2초 & P99 3초(동시 충족) | 피크 300 TPS 부하 시 | REQ-TEC-02 |
| 용량 | 동시 조회 처리량(NFR-C01) | 300 TPS 이상 | 장애 피크 시간대 부하시험 | REQ-TEC-02 |
| 용량 | 통지 담당자 계정(NFR-C02) | 500 계정 이상 | 개통 1년차 기준 | REQ-TEC-06(2-2 Capacity) |
| 용량 | 통지 이력 보관(NFR-C03) | 3년 이상 | 위변조 방지 로그 포함 | REQ-TEC-06(2-2 Capacity)·REQ-TEC-04 |
| 가용성 | 서비스 가용성(NFR-A01) | 99.95%/월(월 약 21.6분) | 이중화 전제, 서비스 중단 기준 월 집계 | REQ-OPS-01 |
| 이중화 | 채널 자동 전환(NFR-A02) | 30초 이내 | 임의 채널 강제 차단 후 전환 시험 | REQ-TEC-03 |
| 확장성 | 무중단 Scale-out(NFR-A03) | 지원(무중단) | 노드 증설 시 서비스 무중단 확인 | REQ-TEC-06(2-2 Capacity) |
| 복구 | DR RTO/RPO(NFR-A04) | RTO 1시간 / RPO 15분 | DR 전환 훈련 시 | REQ-TEC-06(2-2 Capacity) |
| 보안 | 인증·전송 암호화(NFR-S01) | OAuth 2.0, TLS 1.2 이상 | 보안 점검 | REQ-TEC-04 |
| 보안 | 위변조 방지 로그(NFR-S02) | 무결성 검증 100% 통과 | 통지 이력 로그 해시 대사 | REQ-TEC-04 |

### 4.1.2 성능(NFR-P01~P03) — 근거·설계 전제

- **NFR-P01·P02(통지, REQ-TEC-01)**: 수신~통지 1분은 인지~담당자 5분(운영 포함 종단)의 **시스템 전송  
  구간 부분집합**임(A4 발견 2). 능동 헬스체크 기반 장애 감지 + 병렬/우선순위 채널 전송으로 채널 전환  
  30초(NFR-A02)가 1분 예산을 침범하지 않게 설계함. **수동 타임아웃형 페일오버는 배제함**(A4 기술 리스크).  
- **NFR-P03(조회, REQ-TEC-02)**: **CQRS 읽기 최적화·사전집계·캐시 신선도 관리**로 피크 300 TPS에서  
  평균 2초 & P99 3초를 동시 충족함. 캐시 신선도는 표본 100건 100% 대사(FR-조회-01)와 양립하도록 관리하고,  
  순간 버스트 헤드룸을 확보하며 피크 부하시험으로 확증함(A4 기술 리스크).

### 4.1.3 용량·확장(NFR-C01~C03·NFR-A03)

- **NFR-C01(REQ-TEC-02)**: 동시 조회 300 TPS 이상. NFR-P03과 동일 부하시험에서 처리량·응답을 함께 검증함.  
- **NFR-C02(REQ-TEC-06)**: 통지 담당자 500 계정 이상(개통 1년차 기준). 계정 증가에 따른 통지 팬아웃 성능을  
  NFR-P01 예산 내로 유지함.  
- **NFR-C03(REQ-TEC-06·REQ-TEC-04)**: 통지 이력 3년 이상 보관. 보관 데이터는 위변조 방지 로그(NFR-S02)로  
  무결성을 유지하며 사후 SLA·감사 증거로 사용함.  
- **NFR-A03(REQ-TEC-06)**: 무중단 Scale-out 지원. 노드 증설 시 서비스 중단 없이 용량 확장이 가능해야 함.

### 4.1.4 가용성·이중화·DR(NFR-A01·NFR-A02·NFR-A04)

- **NFR-A01(REQ-OPS-01)**: 서비스 가용성 99.95%/월(월 허용 다운타임 약 21.6분). 이중화를 전제로 단일  
  컴포넌트 장애가 서비스 중단으로 전이되지 않음을 가정함. **벤더 SLA 대상 가용성 지표는 본 1종으로 한정함**.  
- **NFR-A02(REQ-TEC-03)**: 단일 채널 장애 시 대체 채널 자동 전환 30초 이내. 채널 헬스체크 + 페일오버  
  오케스트레이션으로 단일 장애점을 제거함.  
- **NFR-A04(DR, REQ-TEC-06)**: **DR RTO 1시간 / RPO 15분**. 웜스탠바이·비동기 복제 구성으로 충족하며 DR  
  전환 훈련으로 검증함. **04단계 아키 리뷰의 DR 판정 누락 지적을 반영하여 본 NFR을 명시함**(누락 방지).  
  MNO 연동 구간은 전용선/VPN·네트워크·DB·DNS 이중화를 전제하되, MNO 연동 보안검토·승인 리드타임이 일정  
  리스크이므로 연동 협의를 선행함(A4 일정 리스크).

### 4.1.5 보안(NFR-S01·NFR-S02)

- **NFR-S01(REQ-TEC-04)**: API 접근은 OAuth 2.0(서버 간 연동은 client credentials), 전송구간은 TLS 1.2  
  이상(신규 구축 TLS 1.3 권장)으로 암호화함. 접근권한은 관리자/운영자/조회자 3단계로 분리하고 최소권한을  
  적용함.  
- **NFR-S02(REQ-TEC-04)**: 통지 이력은 위변조 방지 로그(해시체인·전자서명·WORM 등)로 보관하고, 로그  
  재계산 해시 대사로 무결성 검증에 100% 통과함(NFR-C03·4.2.4 무결성과 연계).

### 4.1.6 SLA 경계·특이사항

- **MNO 복구 MTTR 배제**: MNO 망 복구시간(MTTR)은 복구 주체가 MNO이므로 **벤더 SLA에서 제외**하고 월간  
  리포트 관찰 지표로만 제공함. 벤더 책임은 통지·가용성·ETR 15분 주기 갱신 성실성으로 한정함(A4 발견 5).  
- **가용성 크레딧 산정 갭**: NFR-A01은 REQ-OPS-01 근거이나 RFP 6-1 크레딧 트리거가 "통지 SLA 준수율"  
  단일 지표뿐임 → 가용성 미달 시 크레딧 산정식은 인수기준(SEC5)에서 별도 정의함(A4 계약 리스크, 딜 지적).  
- **AI 성능(REQ-TEC-05) 처리**: AI 추론 3초·재학습(신규 사고 누적 10건)은 **신규 NFR-ID를 생성하지 않음**.  
  AI 성능은 FR-AI(SEC3)의 확률형 판정과 데이터 요건(4.2 AI 학습 데이터)에서 명세하며, 룰 폴백·HITL을  
  전제로 함(A4 발견 4).

## 4.2 데이터 요건

### 4.2.1 데이터 도메인·보관·개인정보표

| 데이터 도메인 | 주요 항목 | 보관/갱신 | 개인정보 취급 | 근거 REQ-ID |
|---------------|-----------|-----------|---------------|-------------|
| 사고 이벤트(MNO 수신) | 이벤트ID·수신시각·심각도·사고유형 | 이력 보관 | 원천 최소 수집 | REQ-TEC-01 |
| 영향범위(집계) | 가입자 규모·지역·서비스 범위 | 조회 시점 산출 | 집계·범위 단위, 개인식별자 미포함 | REQ-BIZ-02 |
| 통지 이력 | 통지ID·채널·발송시각·수신확인·SLA 판정 | 3년 이상, 위변조 방지 | 담당자 계정 정보 최소 | REQ-TEC-04·REQ-OPS-02 |
| 복구정보(ETR) | 진행률·예상 복구시각 | 15분 주기 갱신·연속 이력 | 해당 없음 | REQ-BIZ-03 |
| 사고 리포트 | 통지 타임라인·영향범위·MTTR | 종료 후 3영업일 | 집계 기반 | REQ-BIZ-04 |
| AI 학습 데이터 | 과거 사고 전수 | 재학습·백테스트용 | 가명·집계 처리 | REQ-BIZ-05a·REQ-TEC-05 |

### 4.2.2 개인정보 취급

- **집계·범위 단위 제공**: 영향범위 등 대외 제공 데이터는 개인 식별자(성명·전화번호·가입자ID)를 배제하고  
  집계·범위 단위로만 제공함(예: "서울 강남구 LTE 가입자 약 12,000명 영향"). 표본 검사 시 개인 특정 불가.  
- **안전성 확보조치**: 개인정보 전송 시 암호화(TLS 1.2 이상, NFR-S01), 접근권한 3단계 최소화, 접근기록  
  보관·위변조 방지를 적용함. 보유기간(3년) 경과 데이터는 파기 정책에 따라 지체없이 파기함.  
- **근거**: 개인정보 보호법·안전성 확보조치 기준, REQ-TEC-04·REQ-BIZ-02.

### 4.2.3 연동 규격

- **Webhook(수신, REQ-TEC-01)**: MNO→시스템 이벤트 푸시. 규격 = 인증(OAuth 2.0)·페이로드 스키마(이벤트ID·  
  수신시각·심각도·사고유형·영향 원천)·재전송·중복 제거·서명 검증.  
- **API(조회, REQ-TEC-02·REQ-TEC-04)**: 영향범위·복구정보·통지이력 조회. 규격 = 인증(OAuth 2.0)·전송  
  (TLS 1.2 이상)·요청/응답 스키마·오류 코드·응답시간 목표(조회 P99 3초, NFR-P03).  
- **시각 정합성(REQ-TEC-01·REQ-OPS-02)**: "발생/수신" 시각 기준은 MNO 이벤트(Webhook) 수신 시각으로  
  정렬하고, 통지 SLA(수신~통지 1분) 자동 측정을 위해 시각 동기(NTP 등)를 데이터 요건에 포함함.

### 4.2.4 통지이력 무결성

- 통지 이력은 **해시체인·전자서명·WORM** 등으로 사후 변경을 차단하고, 로그 재계산 해시 대사로 무결성을  
  검증함(REQ-TEC-04). 본 요건은 NFR-S02(무결성 검증 100% 통과)·NFR-C03(3년 보관)과 연계되어 사후 SLA·  
  감사 증거의 신뢰성을 보장함.

## 4.3 추적성 연결(인수기준 SEC5 · RTM SEC6)

- **NFR → 인수기준(SEC5)**: NFR-P01/P02→통지 SLA 준수율(REQ-OPS-02), NFR-P03/C01→조회 부하시험,  
  NFR-A01→가용성 크레딧(별도 산정식), NFR-A02→채널 전환 시험, NFR-A04→DR 전환 훈련,  
  NFR-S01/S02→보안·무결성 점검으로 각각 측정 가능한 AC에 연결됨.  
- **데이터 요건 → 인수기준(SEC5)**: 개인식별자 배제·집계 단위(표본 개인 특정 불가), 통지이력 3년·무결성  
  100% 대사, ETR 15분 주기 갱신을 AC로 판정함.  
- **RTM(SEC6)**: 각 NFR·데이터 항목의 근거 REQ-ID로 RFP↔SRS(NFR/데이터)↔AC↔검증방법(PoC·부하시험·DR  
  훈련·보안 점검)↔SLA/계약을 1:1 추적함. 신규 ID·임의 수치를 추가하지 않아 커버리지 대사(미매핑 0건)를 유지함.
