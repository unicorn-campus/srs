## Level1 섹션 구조 및 담당 배분

각 섹션은 `rfp/RFP작성가이드.md`와 동일하게 **(a)작성원칙 · (b)통신사 B2B 예시·문구 ·  
(c)작성 체크리스트** 3단 구조로 작성하며, 각 섹션에 **측정 가능한 인수 기준 서술 방식 예시를  
최소 1개** 포함함.

| 섹션 | 제목 | 다루는 SRS 항목 | 담당(정보수집·초안) |
|------|------|----------------|---------------------|
| S0 | 이 가이드 사용법 | 3단 구조·조립 순서·추적성 원칙 | 스펙(통합자 직접 작성) |
| S1 | SRS 개요·목적·범위·독자 | 문서 메타(목적·범위·제외범위·독자·용어) | 스토리텔러 |
| S2 | 고객사 문제 정의 | 고객사 문제 정의(As-Is·WHY·정량 목표) | 리서처 |
| S3 | 기능 요구사항(FR)·솔루션 범위 | 제안 솔루션의 기능·범위 | 데모 |
| S4 | 기술 요건(비기능 요구 NFR) | 기술 요건(성능·가용성·보안 정량화) | 아키 |
| S5 | 데이터 요건 | 데이터 요건(모델·보관·개인정보·연동) | 아키 |
| S6 | 인수 기준(Acceptance Criteria) | 인수 기준(측정 가능 완료 기준) | 딜 |
| S7 | 추적성·운영이행·용어 | 추적성(RFP↔SRS↔AC)·운영이행·용어집 | 케어 |

### 섹션별 핵심 내용 (아젠다 상세)

- **S0 이 가이드 사용법**: 3단 구조 설명, SRS 조립 순서, 추적성 원칙("측정할 수 없으면 요구사항이  
  아님"), RFP↔SRS↔인수기준 연결 규칙.  
- **S1 SRS 개요·목적·범위·독자**: SRS 목적 진술, 대상 시스템 범위·제외범위 명시, 독자(개발·영업·법무)  
  정의, 용어·약어 정의 위치. ISO/IEC/IEEE 29148 문서 개요 요소 반영.  
- **S2 고객사 문제 정의**: RFP 1장 취지(WHY)에서 As-Is 문제·정량 목표(현재값→목표값)를 SRS 문제  
  정의로 재구성. 문제-요구-효과 인과 고리 유지.  
- **S3 기능 요구사항(FR)·솔루션 범위**: REQ-BIZ 요구를 FR로 정형화, 기능ID 부여, 범위·제외범위·  
  가정·제약 구분, 각 FR에 검증(PoC) 연결.  
- **S4 기술 요건(NFR)**: REQ-TEC 요구를 [지표+목표값+측정조건] 3요소 NFR로 정량화(성능·용량·  
  가용성·보안·이중화·DR).  
- **S5 데이터 요건**: 데이터 모델·수집·보관 기간·개인정보 취급(집계/범위 단위·암호화)·연동 데이터  
  규격(Webhook/API)·무결성.  
- **S6 인수 기준(Acceptance Criteria)**: 측정 가능한 완료 기준 서술 규칙(기간·임계값·판정방법),  
  Given-When-Then 형식, SLA·서비스 크레딧·하자/유지 경계와의 정산 연계.  
- **S7 추적성·운영이행·용어**: RFP 요구ID↔SRS 기능ID↔인수기준 추적 매트릭스, 운영 이행(교육·전환·  
  hypercare) 요건, 용어·약어집.
