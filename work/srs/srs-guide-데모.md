## S3. 기능 요구사항(FR)·솔루션 범위

> 담당(정보수집·초안): PoC 엔지니어(데모)  
> 다루는 SRS 항목: RFP의 업무 요구(REQ-BIZ)를 검증 가능한 기능 요구사항(FR)으로 정형화, 기능ID 부여,  
> 범위·제외범위·가정·제약 구분, 각 FR을 검증(PoC 판정 기준)에 연결.  
> 정보수집 근거: `work/srs/srs정보수집-데모.md`(출처 URL 포함).

### (a) 작성 원칙

- **REQ-BIZ → FR 정형화**: RFP의 업무 요구(REQ-BIZ, "무엇을 하고 싶은가")를 SRS의 기능 요구사항(FR,  
  "시스템이 무엇을 하는가")으로 좁혀 재기술함. 요구 3층위(비즈니스=왜·무엇 / 기능=시스템 동작 / 비기능=얼마나  
  잘) 중 FR은 기능 층위이며, 성능·가용성·보안 정량치는 NFR(S4)로 이관하고 FR의 판정 기준에서 상호 참조함.[^s3-iso]  
- **1:N 분해 허용**: 하나의 REQ-BIZ가 복수 FR로 분해될 수 있음(예: "심각도별 자동 통지" = 심각도 판정 FR +  
  통지 발송 FR). 누락 없이 모든 REQ-BIZ가 하나 이상의 FR로 매핑되어야 함(망라성 우선).  
- **검증 가능한 문장 구조(ISO/IEC/IEEE 29148)**: 각 FR을 [조건(condition) + 주체(subject=시스템) +  
  동작(action) + 대상(object) + 동작 제약(constraint)] 5요소로 서술함. 개별 요구는 29148의 9특성(필요·적절·  
  명확·완전·단일·실현가능·검증가능·정확·준수)을 만족해야 하며, "검증가능성은 측정 가능할 때 강화됨"을 지킴.[^s3-iso]  
- **기능ID 부여 규칙**: `FR-{기능그룹}-{2자리 일련번호}` 형식(예: `FR-통지-01`). 기능그룹은 업무 단위로 고정하고,  
  각 FR은 도출 근거인 RFP 요구ID(REQ-BIZ-nn)를 반드시 명시하여 추적성을 확보함.  
- **범위 4구분 명문화(정산 분쟁 예방)**: 솔루션 경계를 네 갈래로 분리 기재함.[^s3-ac]  
  - 범위(In-scope): 벤더가 구현·책임지는 기능 / 제외범위(Out-of-scope): 명시적으로 제외하는 항목  
  - 가정(Assumptions): 참으로 전제하는 선행 조건 / 제약(Constraints): 반드시 지켜야 하는 규제·기술·정책 한계  
- **각 FR → PoC 판정 기준 연결**: "측정할 수 없으면 요구사항이 아님" 원칙에 따라 모든 FR에 합격/불합격을 가르는  
  측정 가능한 PoC 판정 기준을 부여함. 기준은 SMART(구체·측정가능·사업연관·현실적·시한)로 설계하며, RFP의  
  확인 기준·RTM과 값을 일치시킴. PoC의 약 62%가 성공 기준 부실로 실패하므로 기준을 사전에 못 박는 것이 핵심임.[^s3-poc]  
- **AI 기능의 확률적 판정**: AI 예측·생성 FR은 결정론적 기능과 달리 통계적 임계값(오차율)·표본·폴백·사람 검수  
  (HITL)로 판정 기준을 설계함. 결과는 참고용이며 최종 통지·안내 문구는 담당자 검수 게이트를 통과해야 함.

### (b) 통신사 B2B 예시·문구

- **본 RFP 6개 FR 그룹**: REQ-BIZ 5개 요구를 아래 기능그룹으로 정형화함(그룹 코드 고정).  
  통지(REQ-BIZ-01) · 조회(REQ-BIZ-02) · ETR(REQ-BIZ-03) · 리포트(REQ-BIZ-04) · AI예측(REQ-BIZ-05a) ·  
  AI응대(REQ-BIZ-05b).

- **FR 문장 예시(29148 5요소 구조 적용)**  
  - `FR-통지-01`: MNO 장애·보안 이벤트를 수신하면(조건), 시스템은(주체) 심각도(긴급/높음/보통)를 자동 판정하여  
    해당 MVNO 담당자에게(대상) 통지를 발송함(동작) — 수신~담당자 도달 5분 이내(제약). [REQ-BIZ-01]  
  - `FR-조회-01`: 운영자가 영향범위 조회를 요청하면(조건), 시스템은(주체) 영향 가입자 규모·지역·서비스 범위를  
    집계·범위 단위로(대상) 실시간 반환함(동작) — 개인 식별정보 원문 미포함(제약). [REQ-BIZ-02]  
  - `FR-ETR-01`: 사고가 진행 중인 동안(조건), 시스템은(주체) 복구 진행률·예상 복구시각(ETR)을(대상) 15분 주기로  
    갱신·연속 조회 제공함(동작). [REQ-BIZ-03]

- **기능ID ↔ RFP 요구ID ↔ PoC 판정 기준 매핑표(본 RFP 각색)**

| 기능ID | 기능 요약 | RFP 요구ID | PoC 판정 기준(측정 가능) |
|--------|-----------|-----------|--------------------------|
| FR-통지-01 | 심각도별 자동 통지 발송 | REQ-BIZ-01 | 수신(인지) 후 담당자 통지 도달 5분 이내(종단) |
| FR-통지-02 | 심각도(긴급/높음/보통) 자동 분류 | REQ-BIZ-01 | 심각도 분류 규칙 대사 100% 정확 매핑 |
| FR-조회-01 | 영향범위(가입자·지역·서비스) 실시간 조회 | REQ-BIZ-02 | 평균 2초 & P99 3초, 표본 100건 대사 100% 일치 |
| FR-조회-02 | 집계·범위 단위 제공(개인식별 최소화) | REQ-BIZ-02 | 개인 식별정보 원문 미출력 확인 |
| FR-ETR-01 | 복구 진행률·ETR 15분 주기 갱신 | REQ-BIZ-03 | 15분 주기 갱신·완료 시점까지 연속 조회 |
| FR-리포트-01 | 사고 이력·재발방지 리포트 자동 생성 | REQ-BIZ-04 | 종료 후 3영업일 이내(타임라인·영향범위·MTTR 포함) |
| FR-AI예측-01 | AI 영향범위·확산 추이 예측 | REQ-BIZ-05a | 과거 전수 백테스트 오차율 ±15% 이내, 표본 부족 시 룰 폴백 |
| FR-AI응대-01 | AI 응대 스크립트 초안 생성 | REQ-BIZ-05b | 통지 후 10분 이내 초안·상담사(HITL) 검수 후 사용 |

- **측정 가능한 인수 기준 서술 예시(필수)**  
  - 값 명시형: `FR-조회-01: 영향범위 조회 응답 평균 2초 & P99 3초 이내(동시 충족), 표본 100건 대사 100% 일치 —  
    REQ-BIZ-02·REQ-TEC-02 근거`  
  - Given-When-Then형: `Given 장애 피크(300 TPS) 부하에서 / When 임의 영향범위 조회 1건을 요청하면 /  
    Then 평균 2초 & P99 3초 이내 응답하고, 표본 100건 대사 시 100% 일치함 — REQ-BIZ-02 근거`  
  - AI 확률형: `FR-AI예측-01: Given 과거 사고 전수를 백테스트하면 / When 각 사고의 영향 가입자 규모를 예측하면 /  
    Then 건별 오차율 ±15% 이내 건을 합격으로 집계·리포트하고, 표본 부족 시 룰 기반 폴백을 병행함 — REQ-BIZ-05a 근거`

- **범위·제외범위·가정·제약 기재 예시(본 RFP 근거)**  
  - 범위(In): 통지·조회·ETR·리포트·AI예측·AI응대 6개 기능군, MNO 이벤트 수신 연동, 다중 채널 통지.  
  - 제외범위(Out): MNO 망 자체의 물리적 복구(벤더 SLA 제외, 관찰 지표만), 가입자 개인 식별정보 원문 제공,  
    담당자 무검수 자동 통지·안내(HITL 필수).  
  - 가정(Assume): MNO가 이벤트를 Webhook/API 규격으로 실시간 제공(REQ-TEC-01), AI 백테스트용 과거 사고  
    데이터 확보 가능.  
  - 제약(Constraint): 개인정보보호법 안전성 확보조치 준수, 접근권한 3단계(관리자/운영자/조회자) 분리,  
    AI 결과는 참고용·HITL 게이트 통과 필수.

### (c) 작성 체크리스트

- [ ] 모든 REQ-BIZ가 하나 이상의 FR로 정형화되었는가(누락 없이 망라)  
- [ ] 각 FR이 [조건+주체+동작+대상+제약] 5요소 구조로 검증 가능하게 서술되었는가(ISO/IEC/IEEE 29148)  
- [ ] 기능ID(`FR-{그룹}-{번호}`)를 부여하고 각 FR에 도출 근거 RFP 요구ID를 매핑했는가(추적성)  
- [ ] 범위·제외범위·가정·제약을 4구분으로 명시하여 정산·범위 분쟁을 예방했는가  
- [ ] 각 FR에 측정 가능한 PoC 판정 기준(합격/불합격 임계값·표본·판정방법)이 붙어 있고 RFP 확인 기준·RTM과  
  값이 일치하는가  
- [ ] AI 기능(FR-AI*)에 통계적 임계값(오차율)·폴백·HITL 검수 게이트를 명시했는가  
- [ ] FR ↔ RFP 요구ID ↔ PoC 판정 ↔ 인수 기준(S6)·추적성(S7)의 연결 고리가 닫혔는가
