## S4. 기술 요건(비기능 요구 NFR)

### (a) 작성 원칙

- 기술 요건(NFR)은 "얼마나 잘"을 규정함. 모든 NFR은 **[지표 + 목표값 + 측정조건]** 3요소로 기술해야  
  제안·검수·SLA 연계가 성립함. 3요소가 없으면 요구가 아니라 희망사항임("측정할 수 없으면 요구가 아님").  
- 4대 분류(IT 인프라·시스템 능력·장해 대책·보안)를 망라하고, ISO/IEC 25010 품질특성(성능효율성·신뢰성·  
  보안성 등)에 매핑하여 누락을 방지함.  
- 작성 순서 권장: 시스템 능력(성능·용량 정량) → 장해 대책(가용성·이중화·DR) → 보안 → IT 인프라. 뒤 항목이  
  앞 스펙을 결정하므로 능력·가용성 목표를 먼저 확정하고 인프라를 역산함.  
- **가용성 → 이중화 → SLA 배상 3단 연계**: 가용성 목표(정량)만 못 박고 이중화 방식(Active-Active/Standby)은  
  벤더 제안에 위임함. 가용성 수치는 기술(S4)·인수기준(S6)·운용/계약 SLA가 반드시 일치해야 함.  
- **규제 항목은 상세 지정, 아키텍처·튜닝 수단은 위임**으로 배분함(개인정보 암호화·접근통제·연동 보안은  
  상세 지정, 세부 이중화 구성은 위임).  
- **측정 가능한 인수 기준 서술 방식(S4 예시)**  
  - "MNO 이벤트 수신~MVNO 통지 지연 **1분 이내**, 전송 성공률 **99.9% 이상**(측정: 월 단위, 정상·피크  
    포함) — **REQ-TEC-01** 근거"  
  - "피크 **300 TPS** 부하에서 영향범위 조회 응답 **평균 2초 & P99 3초 이내**(동시 충족) — **REQ-TEC-02** 근거"

### (b) 통신사 B2B 예시·문구 (본 RFP REQ-TEC 각색)

- **NFR 정량 명세표(SRS 삽입용)** — 각 NFR은 요구ID로 인수기준(S6)·SLA와 추적 연결함

| 구분 | 지표(NFR-ID) | 목표값 | 측정조건 | 근거 |
|------|--------------|--------|----------|------|
| 성능 | 수신~통지 지연(NFR-P01) | 1분 이내 | 수신~통지 종단, 월 측정, 정상·피크 | REQ-TEC-01 |
| 성능 | 통지 전송 성공률(NFR-P02) | 99.9% 이상 | 월 단위 집계 | REQ-TEC-01 |
| 성능 | 영향범위 조회 응답(NFR-P03) | 평균 2초 & P99 3초(동시) | 피크 300 TPS 부하 시 | REQ-TEC-02 |
| 용량 | 동시 조회 처리량(NFR-C01) | 300 TPS 이상 | 장애 피크 시간대 | REQ-TEC-02 |
| 용량 | 통지 대상 담당자 계정(NFR-C02) | 500 계정 이상 | 개통 1년차 기준 | Capacity표 |
| 용량 | 통지 이력 보관(NFR-C03) | 3년 이상 | 위변조 방지 로그 포함 | Capacity표 |
| 가용성 | 서비스 가용성(NFR-A01) | 99.95%/월(월 약 21.6분) | 이중화 전제, 서비스 중단 기준 월 집계 | 장해대책·REQ-OPS-01 |
| 이중화 | 채널 자동 전환(NFR-A02) | 30초 이내 | 임의 채널 강제 차단 후 전환 시험 | REQ-TEC-03 |
| 확장성 | 무중단 Scale-out(NFR-A03) | 지원 | 노드 증설 시 서비스 무중단 | Capacity표 |
| 복구 | DR RTO/RPO(NFR-A04) | RTO 1시간 / RPO 15분 | DR 전환 훈련 시 | Capacity표 |
| 보안 | 인증·전송 암호화(NFR-S01) | OAuth 2.0, TLS 1.2 이상 | 보안 점검 | REQ-TEC-04 |
| 보안 | 위변조 방지 로그(NFR-S02) | 무결성 검증 통과 | 통지 이력 로그 무결성 검증 | REQ-TEC-04 |

- **가용성·이중화·DR 서술 예시**  
  - 가용성 목표는 **서비스 중단 시간 기준 99.95%/월**로 하며, 이중화를 전제로 단일 컴포넌트 장애가 서비스  
    중단으로 전이되지 않음을 가정함. 벤더 SLA 대상 지표는 이 가용성 1종으로 한정함.  
  - MNO 망 복구시간(MTTR)은 복구 주체가 MNO이므로 **벤더 SLA에서 제외**하고 월간 리포트의 관찰 지표로만  
    제공함. 벤더 책임은 복구정보(ETR) 15분 주기 갱신 성실성으로 한정함.  
- **보안 서술 예시**  
  - API 접근은 **OAuth 2.0** 인증(서버 간 연동은 client credentials), 전송구간은 **TLS 1.2 이상**(신규  
    구축은 TLS 1.3 권장)으로 암호화함.  
  - 접근권한은 관리자/운영자/조회자 **3단계**로 분리하고 최소권한을 적용함. 통지 이력은 위변조 방지 로그  
    (해시체인·전자서명·WORM 등)로 보관하여 무결성 검증이 가능하게 함.  
- **IT 인프라·연동 서술 예시**  
  - 배포 형태(클라우드/온프레미스)는 벤더 제안에 위임하되, MNO 연동 구간은 **전용선 또는 VPN**(인터넷  
    구간 암호화)으로 하고 네트워크·DB·DNS 이중화를 전제함(REQ-TEC-06).

### (c) 작성 체크리스트

- [ ] 모든 NFR을 [지표 + 목표값 + 측정조건] 3요소로 정량 기술했는가  
- [ ] 4대 분류(인프라·능력·장해·보안)를 망라하고 ISO/IEC 25010 특성에 매핑했는가  
- [ ] 성능·용량 수치(통지 1분·성공률 99.9%·300 TPS·평균2초&P99 3초)를 측정조건과 함께 명시했는가  
- [ ] 가용성(99.95%/월 약 21.6분)이 S4·S6·운용/계약 SLA와 수치 일치하는가  
- [ ] 채널 자동 전환 30초·DR RTO 1h/RPO 15m가 시험 방법과 함께 기술되었는가  
- [ ] 보안(OAuth 2.0·TLS 1.2+·접근권한 3단계·위변조 방지 로그)을 상세 지정했는가  
- [ ] MNO 복구 MTTR을 벤더 SLA 제외·관찰 지표로 명확히 구분했는가  
- [ ] 각 NFR에 요구ID(REQ-TEC-xx)를 부여해 인수기준(S6)·SLA까지 추적 연결했는가

## S5. 데이터 요건

### (a) 작성 원칙

- 데이터 요건은 **데이터 모델·수집·보관 기간·개인정보 취급·연동 규격·무결성**을 규정함. 데이터는 통지  
  SLA 판정·영향범위 산출·사고 리포트의 원천이므로 항목·형식·제약을 명시해야 분쟁을 예방함.  
- **개인정보 최소 취급 원칙**: 영향범위 등 대외 제공 데이터는 개인 식별자를 배제하고 **집계·범위 단위**로만  
  제공함. 이는 개인정보 보호법·안전성 확보조치 기준의 직접 근거를 가짐.  
- **무결성 원칙**: 사후 SLA·감사 증거가 되는 통지 이력은 위·변조가 불가능하도록 보관하고 무결성 검증  
  수단을 명시함.  
- **연동 규격 명문화**: Webhook/API의 인증·전송·스키마·시각 정합성을 규격으로 못 박아 MNO 연동 구간의  
  해석 차이를 제거함.  
- **측정 가능한 인수 기준 서술 방식(S5 예시)**  
  - "영향범위 조회 결과에 **개인 식별자(성명·전화번호·가입자ID 등) 미포함**, **집계·범위 단위로만 제공**  
    (표본 검사 시 개인 특정 불가) — **개인정보 보호법·안전성 확보조치 기준** 근거"  
  - "통지 이력을 **3년 이상 보관**하며 위변조 방지 로그의 **무결성 검증(해시 대사)에 100% 통과** —  
    **REQ-TEC-04·Capacity표** 근거"

### (b) 통신사 B2B 예시·문구 (본 RFP 데이터 요건 각색)

- **주요 데이터 도메인·보관·개인정보 등급표(SRS 삽입용)**

| 데이터 도메인 | 주요 항목 | 보관/갱신 | 개인정보 취급 | 근거 |
|---------------|-----------|-----------|---------------|------|
| 사고 이벤트(MNO 수신) | 이벤트ID·수신시각·심각도·사고유형 | 이력 보관 | 원천 최소 수집 | REQ-TEC-01 |
| 영향범위(집계) | 가입자 규모·지역·서비스 범위 | 조회 시점 산출 | 집계·범위 단위, 개인 식별자 미포함 | REQ-BIZ-02, 2-2 |
| 통지 이력 | 통지ID·채널·발송시각·수신확인·SLA 판정 | 3년 이상, 위변조 방지 | 담당자 계정 정보 최소 | REQ-TEC-04, Capacity |
| 복구정보(ETR) | 진행률·예상 복구시각 | 15분 주기 갱신·연속 이력 | 해당 없음 | REQ-BIZ-03 |
| 사고 리포트 | 통지 타임라인·영향범위·MTTR | 종료 후 3영업일 | 집계 기반 | REQ-BIZ-04 |
| AI 학습 데이터 | 과거 사고 전수 | 재학습·백테스트용 | 가명·집계 처리 | REQ-BIZ-05a |

- **개인정보 취급 서술 예시**  
  - 영향범위 조회는 "서울 강남구 LTE 가입자 약 12,000명 영향"과 같이 **집계·범위 단위**로만 응답하고,  
    개인을 특정할 수 있는 식별자를 포함하지 않음.  
  - 개인정보 전송 시 암호화(TLS 1.2 이상), 접근권한 최소화, 접근기록 보관·위변조 방지를 적용함(안전성  
    확보조치 기준). 보유기간(3년) 경과 데이터는 파기 정책에 따라 지체없이 파기함.  
- **연동 데이터 규격 서술 예시**  
  - **Webhook(수신)**: MNO→시스템 이벤트 푸시. 규격 = 인증(OAuth 2.0)·페이로드 스키마(이벤트ID·수신시각·  
    심각도·사고유형·영향 원천)·재전송·중복 제거·서명 검증.  
  - **API(조회)**: 영향범위·복구정보·통지이력 조회. 규격 = 인증(OAuth 2.0)·전송(TLS 1.2+)·요청/응답  
    스키마·오류 코드·응답시간 목표(조회 P99 3초).  
  - **시각 정합성**: "발생/수신" 시각 기준은 MNO 이벤트(Webhook) 수신 시각으로 정렬하고, 통지 SLA(수신~  
    통지 1분) 자동 측정을 위해 시각 동기(NTP 등)를 데이터 요건에 포함함.  
- **무결성 서술 예시**  
  - 통지 이력은 해시체인·전자서명·WORM 등으로 사후 변경을 차단하고, 로그 재계산 해시 대사로 무결성을  
    검증함(REQ-TEC-04).

### (c) 작성 체크리스트

- [ ] 주요 데이터 도메인(사고 이벤트·영향범위·통지 이력·복구정보·리포트·AI 학습)을 망라했는가  
- [ ] 각 데이터의 보관 기간·갱신 주기(통지 이력 3년·ETR 15분)를 명시했는가  
- [ ] 영향범위 등 대외 제공 데이터를 집계·범위 단위로 한정하고 개인 식별자 배제를 명문화했는가  
- [ ] 개인정보 암호화(TLS)·접근통제·접근기록·파기 정책을 안전성 확보조치 기준으로 지정했는가  
- [ ] 통지 이력 무결성(위변조 방지 로그·무결성 검증 방법)을 기술했는가  
- [ ] Webhook/API 규격(인증 OAuth 2.0·전송 TLS·스키마·시각 정합성)을 명문화했는가  
- [ ] 각 데이터 요건을 근거 요구ID(REQ-BIZ/REQ-TEC)와 인수기준(S6)에 추적 연결했는가
